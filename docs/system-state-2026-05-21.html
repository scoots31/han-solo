4. Letta ↔ Han Solo MCP connection

Letta v0.16.7 supports native MCP server integration. We registered Han Solo's MCP endpoint directly with Letta — no custom code required. Letta calls out to Han Solo as a Streamable HTTP MCP server using Ren's bearer token.

This is the key architectural change. Previously Ren had zero access to Han Solo's PostgreSQL data. Now Letta makes authenticated tool calls to the Han Solo MCP, and the results come back into Ren's context exactly like any other tool response.

Registration payload (PUT /v1/tools/mcp/servers)

{
  "server_name": "han-solo",
  "type": "streamable_http",
  "server_url": "https://han-solo-mcp.onrender.com/mcp",
  "auth_header": "Authorization",
  "auth_token": "Bearer <USER_TOKEN_REN>"
}

How it works step by step

  1. Ren decides she needs project data — e.g. user asks about the garden planner
  2. Ren calls search_t4(project_slug="garden-planner", query="phase")
  3. Letta makes an HTTP POST to han-solo-mcp.onrender.com/mcp with JSON-RPC, authenticated as Ren
  4. Han Solo MCP executes the tool, queries PostgreSQL, returns results as JSON
  5. Letta injects the results into Ren's context as a tool response
  6. Ren reads the data and responds to the user

Important schema note

The PUT /v1/tools/mcp/servers endpoint uses StreamableHTTPServerConfig schema — the body is the config directly, not wrapped in a config object. The discriminator field is type, not mcp_server_type. This differs from the CreateMCPServerRequest schema which uses mcp_server_type as the discriminator.

Context cost reality

Letta sends the FULL content of every core block to Claude Haiku on every single agent step. With 12 blocks averaging ~5,400 chars each, that's ~18,000+ tokens of core memory loaded per step. T4 schema (PostgreSQL)

CREATE TABLE t4_entries (
    id           SERIAL PRIMARY KEY,
    project_slug TEXT        NOT NULL,
    entry_type   TEXT        NOT NULL,
    entry_id     TEXT        NOT NULL,
    parent_id    TEXT,
    content      TEXT        NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(project_slug, entry_type, entry_id)
);
Why T4 ≠ Letta archival: T4 is intentionally in PostgreSQL rather than pgvector because project artifacts (slices, phases, PRDs) are structured records — they need update-in-place semantics (upsert behavior) and exact key-based retrieval. Letta's archival store is append-only vector search, which is right for unstructured discussion and signals but wrong for project state.

9. Cost context — burn & paused jobs

Jobs are currently PAUSED. Both synthesize.py and curator.py check the jobs_paused flag in the han_solo_config table at startup and exit immediately if true. The flag was set to true via POST /api/jobs-paused {"paused": true} during this session.

Root cause of $25/day burn

The Dockerfile.synth runs synthesize.py then curator.py as a Render cron job. Curator was scheduled hourly and on every run:

  1. Fetches all 392+ archival passages from Letta (no state, no checkpointing)
  2. Processes in batches of 20 with 2 Claude Sonnet calls per batch (connection finding + quality scan)
  3. That's ~40 Sonnet API calls per run × 24 runs/day = ~960 Sonnet calls/day
  4. At ~$0.026/call estimated → ~$25/day
