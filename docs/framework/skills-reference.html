Session Modes
Controls how much the framework activates at the start of a session. Bare is the default — the framework is silent until you invoke a skill. Token cost scales with activation: guided (highest) · piloted (moderate) · bare (lowest). Switch modes verbally in Cursor or with slash commands in Claude Code.
3 modes
bare
Default
The framework is silent. No routing, no always-on skills. Every skill loads only when explicitly invoked. Maximum token efficiency — nothing fires until you ask for it.
Invoke
  • Cursor: "bare mode"
  • Claude Code: /bare
  • Also the default — active when no mode has been chosen
What activates
  • Nothing — framework waits for explicit skill invocation
  • Always-on skills do not load
  • No phase routing or chain
Best for
  • Quick one-off tasks, individual skill use, or experienced builders who know exactly what they need. Lowest token cost — nothing loads until invoked.
piloted
Flexible
Always-on skills activate once and persist for the session. Phase skills are invoked manually — you choose what to run and when. The framework is present but you're driving.
Invoke
  • Cursor: "piloted mode"
  • Claude Code: /piloted
What activates
  • All four always-on skills load once: process-mapper, product-continuity, framework-health, retrospective
  • Phase skills wait for explicit invocation
  • No automatic phase routing
Best for
  • Builders who know the framework and want continuity capture without the full phase chain. Moderate token cost — always-on loads once, then skills on demand.
guided
Full chain
Full framework engagement. The start skill reads your opening message and routes automatically. Always-on skills load and persist. The phase chain drives the session — gates, anchors, QA chain, all of it.
Invoke
  • Cursor: "guided mode"
  • Claude Code: /guided
  • Named resume — Cursor: "guided on [project name]"
  • Named resume — Claude Code: /guided [project name]
What activates
  • All four always-on skills load once
  • start skill reads opening message and routes to Brainstorm or Discover
  • Full phase chain — gates, four anchors, QA chain, all active
Best for
  • New projects, unfamiliar territory, or when you want the full discipline of the framework driving the session. Highest token cost — full framework loads.
Session Hygiene

Mode is a performance decision, not just a philosophy one. Match mode and model to the work — and end sessions at gates, not mid-phase.

  • Mode footprint: guided (full chain — highest cost) · piloted (always-on only — moderate) · bare (nothing — zero). Switch mid-project as the work changes.
  • Session length: Phase gates are natural session reset points. The gate output is what the next phase reads — not conversation history. One phase = one session as a rule of thumb.
  • Model: Heavy thinking (design, architecture, discovery) → strongest model. Execution (build, bug-fix, QA) → faster, lighter model. Same guidance applies in Cursor and Claude Code.
  • Closing: Say "let's close out" — continuity tracking captures state and updates the handoff for next session.
  • Resuming by name: Register projects in projects.md (playbook root). Then "guided on [name]" picks up exactly where you left off — no saved prompt needed.
  • The reminder: Once per session, at a natural pause or gate, framework-health will surface a clean close point. That's your signal — wrap up if you're done for the day.

Entry & Routing
The front door. Reads context, routes to the right starting point.
2 skills
start
Framework entry
The framework's front door. Reads the opening message of a new project conversation and routes automatically to Brainstorm (exploratory ideas) or Discover (clear solution in mind). Checks for existing context before routing — never restarts a project already in progress. Detects deployed projects and opens a new cycle rather than continuing closed work. On named project resume, reads only the two active handoff sections ("Open right now" and "Next session picks up at") on first load — full handoff loads on demand when the session actually needs it.
Key Elements
  • Reads opening tone — exploratory vs. articulated idea
  • Checks for existing context (blueprint, discovery brief, active sprint) before routing
  • Detects deployed projects — routes to new cycle protocol, opens new phase in existing backlog, never touches completed phases
  • Asks one question if routing is ambiguous
  • Never announces the framework or explains phases
  • One sentence of orientation, then just starts
  • Lighter re-entry: named project resume reads only "Open right now" and "Next session picks up at" on load — earlier decisions, stack context, and phase history load on demand; falls back to full read if those sections are absent
Activated By
  • Framework activates automatically when a new project conversation opens
  • Any opening message with a project or feature idea triggers routing
Output
  • Routing decision: Brainstorm, Discover, or new cycle for deployed projects
  • Hands off immediately — no intermediate output
onboard
Framework entry
Brings an existing project into the framework without redoing the work. Reads the project directory, semantically maps content to gate artifacts, synthesizes with solo confirmation, places the project at the correct starting phase, and verifies the project will sync cleanly with the Solo Companion before exiting.
Invoke
  • Cursor: "I want to bring [project] into the framework"
  • Claude Code: /onboard
  • Also: start routes here when it detects existing-project language
Key Elements
  • Reads docs, notes, and planning files — skips code files
  • Semantic matching — content-based, not filename-based
  • Shows mapping preview before writing any framework file
  • Gaps surface as targeted discussion questions, not just file names
  • Classifies remaining gaps: undocumented work (ask) vs. work not done yet (starting phase)
  • Initializes continuity docs and exits with clear orientation
  • Companion compatibility pass — verifies projects.md registration, backlog section headers, handoff ## Open right now section, and Review URL field on all slice records before exiting
Output
  • Framework gate files synthesized from existing project content
  • docs/continuity/handoff.md initialized with ## Open right now
  • Clear statement of starting phase and first move
  • Companion compatibility confirmed — project syncs on next Solo Companion run

Discovery
Understanding the problem, the process, and the constraints before anything gets designed.
4 skills
brainstorming
Framework-routed
Product exploration for ideas that aren't fully formed. A genuine collaborative conversation — not a checklist or a spec session. Explores what the idea is and why it matters, surfaces what needs research before it can be designed, and ends with enough clarity to feed Discover. Includes a process sketch moment and a visual sketch mid-session when the idea benefits from something concrete to react to.
Key Elements
  • Process sketch moment — process-mapper activates when conversation surfaces a process, captures preliminary as-is and to-be sketch
  • Visual sketch mid-session — rough HTML screen as a thinking tool, not a deliverable. Triggers "warmer / colder" reaction.
  • One question per round — the right question, not the next one on a list
  • Routes to Discover, Research, or decomposition based on outcome
  • Explicitly not a spec or feasibility session
Activated By
  • Framework routes here when the opening message has an exploratory tone
  • Also reachable when working through an unclear idea at any stage
Output
  • docs/brainstorm-[date]-[topic].md
  • Idea summary, value moment, preliminary process sketch (as-is + to-be draft)
  • Research items with specific questions
  • Routing decision: Discover / Research / decompose / more brainstorming
discover
Framework-routed
The day-one conversation that builds the full product story — who it's for, what they do, where the value moment is, beginning to end. Produces the discovery brief and, critically, produces and agrees the as-is and to-be process maps before any design begins. The to-be map agreed here becomes the process contract for every downstream phase.
Key Elements
  • Narrative conversation — not a requirements form
  • Surfaces who/what/value with specificity, not user types
  • process-mapper produces as-is map (including manual processes) and to-be map
  • To-be map agreed and validated before design sprint begins — this is the gate
  • Routes to design on-ramp: scratch / Figma / reference
Activated By
  • Framework routes here when the opening message has a clear, articulated idea
  • brainstorming hands off here when the idea reaches directional clarity
Output
  • docs/discovery-brief.md
  • docs/process/as-is-[name].md
  • docs/process/to-be-[name].md
  • Design on-ramp decision documented
  • Gate: to-be map agreed before design begins
tech-context
Phase skill
Establishes the full technical context before design begins — stack, constraints, infrastructure requirements, deploy configuration, security classification, and Claude API authentication pattern. Every downstream skill reads this document. The deploy skill routes entirely based on what tech-context defines. Regulated projects receive an acknowledgment gate here and a hard stop at deploy requiring documented external compliance review.
Key Elements
  • Routes to team profile (predefined stack) or general questions for greenfield
  • Plain-language deployment question maps to one of seven named paths (Local Mac, Cloudflare, Railway, Render, Fly.io, AWS App Runner, No pipeline) — solo never sees the list
  • Captures automated vs. manual deployment preference — framework sets up GitHub Actions at deploy phase if selected
  • Observability recommendation auto-maps to deployment path (CloudWatch for AWS App Runner; Datadog for Railway/Render/Fly.io; none for Local Mac)
  • Identifies infrastructure slices — prerequisites before any feature slice can start
  • Stack-specific coding standards referenced by code-review-and-quality
  • Security classification — assigns one or more of seven classes: Standard (default), Personal, Financial, Authentication, Multi-tenant, Confidential, Regulated. Recorded in a Security Classification section in docs/tech-context.md. Read by code-review-and-quality to determine which Check 9 sub-checks run.
  • Claude API authentication — for cloud deployments (Railway, Render, AWS, GCP, Azure, Fly.io, Cloudflare), Workload Identity Federation is the preferred auth pattern: the workload uses its existing cloud identity to get a short-lived access token, with no static API key to store or rotate. Local Mac projects use an API key in an env var. Recorded in docs/tech-context.md; deploy handles setup at deploy time.
  • Required reading for: design-sprint, solo-build, deploy, principal-engineer
When It Runs
  • Phase 1.5 — runs after Discover
  • Required before Design Sprint can begin
Output
  • docs/tech-context.md
  • Stack, constraints, infrastructure slices
  • Deploy method, branching model, CI/CD configuration
  • Security Classification section — class(es) assigned, acknowledgment recorded for Regulated projects
  • Claude API auth pattern — Workload Identity Federation for cloud targets, API key in env var for Local Mac
  • Referenced by every downstream skill
research-spike
On-demand
Timeboxed investigation of a specific unknown. Callable from any phase — when a question surfaces that can't be designed around without an answer. Returns a finding and a recommendation, then hands back to the phase that triggered it.
Key Elements
  • One specific question — not open-ended exploration
  • Timeboxed — produces a finding, not a design
  • Distinguishes "feasible with tradeoffs" from "not feasible"
  • Routes back to the invoking phase with the answer
  • Triggered by design-review when a slice is Blocked
When It Runs
  • Callable from any phase — when a specific question blocks progress
  • brainstorming surfaces research items that feed here
  • design-review flags Blocked slices that need a spike
  • principal-engineer triggers it when an architectural unknown surfaces
Output
  • docs/spikes/[topic].md
  • Finding: feasible / feasible with tradeoffs / not feasible
  • Recommendation for the invoking phase
  • Unblocks slice in design-review backlog

Design
Producing the shared visual artifact that every downstream phase points at.
3 skills
design-sprint
Phase skill
Produces the HTML artifact — all screens, good enough to make real decisions from. The design sprint is the visual contract: the first shared artifact everyone can point at and agree on. It functions as a decision surface — the first time scope, data, process gaps, and implied infrastructure become visible. Reads both the discovery brief and the to-be process map; stops if the to-be map doesn't exist. In Claude Code, secondary screens fan out in parallel after the hero screen is approved on both structure and skin — the full set arrives faster, with a consistent design system applied across all of them. Cursor produces screens sequentially. Reads the Security Classification from tech-context.md in Step 1 — process-mapper checks for security process gaps (missing deletion paths, tenant-scoping issues, collection without disclosure) relevant to the assigned class.
Key Elements
  • On-ramp A: scratch → hero screen → warmer/colder loop → all screens
  • On-ramp B: Figma import via MCP — uses design as source of truth
  • On-ramp C: reference screenshot → aesthetic extraction → produce in that direction
  • Walk-through conversation surfaces every element as build/defer decision
  • process-mapper cross-references screens against to-be map, annotates with screen refs, flags gaps
  • Deferred decisions log captures both scope and implied infrastructure decisions
When It Runs
  • Phase 2 — runs after Discover and Tech Context are complete
  • Requires to-be process map — framework stops without it
Output
  • docs/design/sprint-[id].html
  • docs/design/deferred-decisions.md
  • docs/design/handoff/<date>/ (optional)
  • All screens — the shared visual artifact
  • to-be map annotated with screen references
  • Optional stakeholder handoff — HTML + PDF + feedback template for async review
  • Gate: deferred decisions log required before planning
frontend-design
Supporting skill
Aesthetic and structural guidance for the HTML artifact. Provides design system direction — color, typography, spacing, component patterns — to ensure visual consistency across all screens produced in the design sprint.
Key Elements
  • Color palette, typography scale, spacing system
  • Component patterns: buttons, cards, forms, navigation
  • Ensures cross-screen visual consistency
  • Applied during design sprint, not as a standalone exercise
When It Runs
  • design-sprint activates it when aesthetic direction is needed
  • Also reachable when refining visual design at any point
Output
  • Design system applied to sprint HTML screens
  • Consistent visual language across all screens
awesome-design-md
Supporting skill
On-ramp 3 for the design sprint — reference-based aesthetic extraction. Takes a screenshot or design reference ("make it feel like ESPN") and extracts the design language for use as an aesthetic constraint when producing the HTML artifact.
Key Elements
  • Extracts color palette, typography choices, spacing, component style from reference
  • Produces design constraints document — not a copy, an interpretation
  • Feeds extracted constraints into design-sprint as the aesthetic foundation
  • Works from screenshots, live URLs, or verbal description
When It Runs
  • design-sprint activates it when the reference on-ramp is selected
  • Also reachable when aesthetic extraction is needed
Output
  • Design constraints applied to sprint HTML screens
  • Extracted aesthetic language documented

Data & Design Review
Generating realistic mock data and converting design screens into a defined, Ready backlog.
2 skills
data-scaffold
Phase skill
Generates realistic mock data for all screen data entities. Creates a clean mock layer with a single swap seam — easy to replace with real API data when ready. Flags data behavior candidates (lists, external feeds) so design review can ask the right scale and behavior questions before UI slices are finalized. Updates mock data after those questions are resolved.
Key Elements
  • Derives all entities from design sprint screens
  • Generates realistic (not placeholder) values — names, amounts, dates that look real
  • Single mock/real seam — one place to swap, nothing hardcoded across the codebase
  • Flags list entities and external API sources as data behavior candidates in data-mapping.md — these need the design review data behavior pass
  • Mock indicator badge on screens during development
  • data-mapping.md documents field-to-source mapping, behavior notes (pagination, null rates), and connection status
  • Updates mock after data questions resolve — pagination edge cases, null field rates, empty/error states seeded before dependent slices finalize
When It Runs
  • Phase 2.5 — runs after design sprint
  • Runs before or alongside design review (not blocked by it)
  • Re-runs (Step 7) when data questions log has newly resolved entries
Output
  • data/mock/[entity].json
  • docs/data-mapping.md
  • Proto-API contract — field names, types, source, behavior notes per screen entity
design-review
Phase skill — iterative
The iterative loop between design artifact and build decision. Runs as many rounds as the project needs. Each round refines the design, defines slices, and updates the backlog. Continues in parallel with build for slices not yet started. Build begins when enough slices are Ready — not when all are. In Claude Code: Step 1.5 fires automatically before specialist review — opens each design sprint HTML in a real browser, takes desktop and mobile screenshots, inspects elements against design identity tokens, and checks console errors. Findings feed into the specialist synthesis. The three specialist lenses (PM, Tech Lead, Data) then run as a multi-agent orchestrator. Cursor users get a sequential pass with the same checks; Step 1.5 skips silently. Non-visual projects and rounds with no HTML artifacts skip Step 1.5 silently.
Key Elements
  • Reads docs/stakeholder-feedback/*.md as first-class input — per-screen comments become findings
  • Three lenses every round: PM (right problem?), Tech Lead (buildable?), Data (where does every field come from?)
  • Process coverage check every round — coverage map vs. to-be, every uncovered step surfaced as decision
  • Data behavior pass — required for every screen with external data: volume, list behavior, nulls, empty/error/loading states. Produces a data questions log. Unresolved question blocks dependent UI slices from Ready.
  • Process anchor required on every slice before it reaches Ready
  • Two descriptions per slice — both required: plain language description (what the solo will see) and technical description (what the builder needs to know). Neither copies from the other. Neither can be blank.
  • Quality contract required — four categories: Failure states, Edge cases, Input validation, Security. Each must be addressed or marked N/A — [reason]. A blank category is not valid. Missing quality contract blocks the slice from reaching Ready.
  • Interactive element inventory (Figma-sourced slices) — every interactive element in the Figma frame scope must be classified: Functional (logic wired in this slice), Deferred (logic in a future slice — companion slice created in backlog now), or Out of scope. No fourth option. Inventory written to Notes field. Deferred elements require a companion slice before the current slice reaches Ready.
  • Slice statuses: In Review · Blocked · Deferred · Ready (full lifecycle in Backlog Status Reference)
  • Build signal: enough Ready slices to form a coherent Phase 1 starting point
  • Blocked slices have a named spike — never left open without a specific question
When It Runs
  • Phase 2.6 — runs after design sprint and data scaffold
  • Multiple rounds — continues in parallel with build for remaining slices
  • Re-engages anytime: spike returns, new screen added, design updated
Output
  • docs/backlog.md
  • Slice records: ID, Name, Plain language description, Technical description, Design anchor, Data anchor, Process anchor, Done criteria, Quality contract (four categories), Self-verification checklist, Dependencies, Status
  • Process coverage map per round
  • Review log appended each round
  • Gate: enough Ready slices before build starts

Planning
Sequencing work by risk and process order, then converting to executable issues.
4 skills
prd-to-plan
Phase skill
Turns the backlog of Ready slices into a sequenced, multi-phase implementation plan. Sequences by two forces in tension — risk order (hardest things first) and process order (to-be map step sequence). Resolves conflicts explicitly. Stops if the to-be process map doesn't exist — it's the sequencing foundation.
Key Elements
  • Reads: backlog, to-be map, tech-context, deferred-decisions — stops without to-be map
  • Maps every Ready slice to its to-be process step — flags uncovered steps
  • Sequences by risk order AND process order — conflicts named explicitly
  • Infrastructure slices always sequenced first, regardless of process order
  • Each phase shows: slices, process steps completed, what it proves, dependencies, definition of done
  • On approval: updates backlog with process anchors for every slice
  • Groups slices into Deliverables — each deliverable carries a technical spec (AI-facing), solo description (solo-facing), and acceptance criteria (shared contract between both)
  • Deliverable type assigned: Screen (reviewed visually) or Logic (reviewed against evidence)
  • Integration deliverables — every Screen deliverable with external data gets a companion Logic deliverable for data integration. Always sequences after its Screen companion is Accepted. Data questions log must be resolved before integration slices can be scoped.
  • Architecture type classification — each slice is classified as Leaf node (self-contained) or Core architecture (touches data models, shared logic, or infrastructure) during the slice design step. Unclassified slices cannot be approved — the quality scan blocks the plan until every slice has a type.
When It Runs
  • Phase 3 — when design review has produced enough Ready slices
  • Requires to-be process map — framework stops without it
Output
  • Phased implementation plan (presented for approval)
  • backlog.md updated with process anchors on every slice
  • Open questions and stated assumptions documented
  • Hands off to to-issues on approval
to-issues
Phase skill
Converts the approved implementation plan into GitHub issues in dependency order. Each issue is self-contained — design reference, process anchor, data anchor, done criteria, and dependency links included. A builder can start any slice from its issue alone without reading another document.
Key Elements
  • Confirms prerequisites first: plan approved, gh CLI authenticated, process anchors on every Ready slice
  • Creates issues in dependency order — blockers first so issue numbers can be referenced in dependents
  • Required fields per issue: title, what it builds, design reference, process anchor, data anchor, done criteria, depends on, notes
  • Uses gh CLI with structured body — title format: [SL-XXX] [Slice Name]
  • Labels by phase and status: Phase 1/2, Ready, Infrastructure, Blocked
  • Deferred slices excluded — only Ready slices become issues
When It Runs
  • Immediately after prd-to-plan is approved — no gap between plan and issues
  • Plan is the source of truth — to-issues converts it, does not redesign it
Output
  • GitHub issues created in dependency order
  • Phase and status labels applied (
    Phase 1
    Ready
    Infrastructure
    )
  • Confirmation report: slice ID → issue number → status per slice
  • Each issue self-contained — builder can start without additional context
to-prd
On-demand — optional
Synthesizes accumulated project context into a formal PRD. Optional — used when a formal requirements document is needed for stakeholder alignment, approval processes, or documentation. Reads the discovery brief, design decisions, and deferred decisions log.
Key Elements
  • Reads discovery brief, deferred decisions log, backlog, design screens
  • Produces structured PRD — problem, solution, scope, out of scope, success criteria
  • Not required for the framework flow — invoked when stakeholder documentation is needed
When It Runs
  • When a formal PRD is needed for stakeholder alignment or approvals
  • Typically between discover and design sprint, or before planning
Output
  • docs/prd-[name].md
  • Structured PRD ready for stakeholder review
grill-me
On-demand — optional
Skeptical PM stress-test. Hard questions, challenged assumptions, no vague answers accepted. Used before committing to a plan, before starting a significant build phase, or anytime the solo wants a second opinion that doesn't pull punches.
Key Elements
  • Challenges the "why" — is this the right problem to solve?
  • Surfaces unstated assumptions and treats them as risks
  • Challenges scope — what's in that shouldn't be, what's missing?
  • Doesn't accept "it makes sense" — requires evidence or reasoning
  • Not adversarial — rigorous. The goal is a stronger plan.
When It Runs
  • Explicitly engaged anytime before committing to a plan or phase
  • Commonly: before planning, before Phase 1 build start
Output
  • List of challenged assumptions
  • Open questions that need answers before proceeding
  • Confirmed decisions that survived the challenge

Build
Slice-by-slice execution with discipline, anchors, and automatic QA handoff on completion.
4 skills
solo-build
Phase skill — per slice
Slice-by-slice build execution. Creates a feature branch at start, requires four anchors before writing code, follows journey order (first screen before second), surfaces mid-build discoveries immediately rather than silently resolving them, and commits on code-complete — automatically invoking the QA chain. Core architecture slices trigger an automatic principal engineer review before the branch opens. A pre-flight security check confirms the quality contract addresses the project's security class requirements before any branch opens — if not, the slice goes back to design review. Code review runs as an isolated sub-agent with no access to the build context, producing an unbiased review independent of implementation reasoning. In Claude Code: self-verification now includes a preview check — screenshot, console logs, failed network requests, interactive element click-through, and mobile viewport — all before committing. Console errors or failed network requests caused by the slice are ✗ items that must be fixed before code-complete. Skips silently for non-visual slices and in Cursor.
Key Elements
  • Creates feature branch: feature/SL-[ID]-[slug]
  • Four anchors required before writing code: design, data, done, process
  • Architecture type gate — before branch opens, checks the slice's Architecture type. If Core architecture: principal engineer review fires automatically. No exceptions. Outcome recorded in the Notes field before any code is written.
  • Journey order — first screen before second, dependencies block
  • Mid-build discoveries surfaced immediately — never silently resolved
  • Self-verification checklist run before code-complete — builder confirmation populated with ✓/✗ results before any handoff
  • Commit on code-complete: SL-[ID] code-complete — [name]
  • Status after code-complete: In QA (not Done — QA and phase-test complete the chain)
  • Auto-invokes code-review-and-quality on code-complete
  • Rollback protocol — when a slice that has already passed QA (In Test or Done) needs to be rebuilt, this is a rollback. Builder assesses scope (targeted fix vs. full rebuild), states the case, and waits for solo confirmation before any status changes. On confirmation: log entry written, slice returns to In Build, deliverable reverts to Pending Acceptance, phase reverts to In Progress if Completed. Full rebuild: existing code discarded, spec and design file re-read from scratch.
When It Runs
  • Phase 4 — engaged per slice as each one moves to build
  • Requires a Ready slice with all four anchors confirmed in the backlog
Output
  • Feature branch created
  • Code commit:
    SL-[ID] code-complete — [name]
  • Auto-handoff to code-review-and-quality
autopilot
Phase skill — autonomous build
Autonomous build mode. Builds the full product without solo participation after discovery (or design sprint). Solo-simulator plays every gate. First human touch point is phase test. Refinement cycles handle delta iteration. Single-session, context-managed.
Key Elements
  • Two entry points: post-discover (leaner anchor) or post-design-review (richer anchor, fewer simulator judgment calls)
  • Pre-flight check: simulator reads all discovery docs, surfaces judgment gaps, solo answers before build begins
  • Discovery quality guard — thin to-be map flagged before build starts
  • Simulator at every gate throughout build — solo not contacted during build
  • Deliverable boundary writes enforced automatically — not recommended
  • Mid-deliverable checkpoint at slice 4 fires automatically — handoff updated, build continues
  • Context discipline — reads from continuity docs, not conversation history
  • Phase test is first human touch point — solo sees the full product for the first time
  • Refinement cycles: implementation delta builds after phase test — structural issues exit to guided mode
  • Exit ramp to guided mode at any Refinement cycle completion
When It Runs
  • Explicitly invoked — never auto-routed
  • Requires discovery-brief.md, to-be map, tech-context.md, and backlog.md with slices
  • Alternative to solo-build for the Build phase — solo chooses which mode
Output
  • Locked brief:
    .claude/autopilot-brief.md
  • Simulator decision log:
    .claude/autopilot-decisions.md
  • All slices built and committed
  • Continuity docs updated throughout
tdd
Supporting skill
Test-driven build technique. Tests are written before implementation — red, green, refactor. Invokable from solo-build when a TDD approach is chosen for a slice. Ensures test coverage is built in, not bolted on.
Key Elements
  • Red → green → refactor cycle
  • Tests written from the done anchor criteria
  • Coverage built before implementation, not after
  • Test files committed alongside implementation
When It Runs
  • solo-build activates it when a TDD approach is chosen for a slice
Output
  • Test files covering done anchor criteria
  • Passing implementation
principal-engineer
On-demand
Architecture review lens for consequential decisions. Auto-triggered before any core architecture slice enters build — no manual invocation needed for those. Also invoked on demand at specific moments when the architectural stakes are high. Reads tech-context and the specific proposal, gives one recommendation (not a menu), names risks and debt explicitly, and holds its position under pushback unless new information changes the analysis.
Key Elements
  • Reads tech-context + discovery brief + the specific proposal
  • One recommendation — not "here are three options"
  • Names risks, debt, and watch-for signals explicitly
  • Hard stops for genuine dealbreakers — states them clearly
  • Holds position under preference-based pushback; updates on new information
  • Logs final decision in backlog regardless of outcome
When It Runs
  • Automatically — when a Core architecture slice moves to In Build (fires before branch opens)
  • After tech-context, when the architecture needs a hard look before design
  • Before planning, on consequential sequencing decisions
  • Mid-build, when structural discovery surfaces
  • After a research spike returns with architectural implications
  • Anytime architectural uncertainty is high — engaged deliberately
Output
  • One architectural recommendation with reasoning
  • Risks, debt, and watch-for signals
  • Decision logged in backlog (with reasoning, even if overridden)

QA Chain
Three skills that chain automatically from code-complete — no user invocation needed until browser sign-off. Plus process-change for map-level conflicts.
4 skills
code-review-and-quality
Auto — invoked by solo-build
Ten-check quality gate that runs automatically when solo-build marks a slice code-complete. Checks pattern compliance, data sourcing, expandability, cleanliness, scope discipline, documentation, stack compliance, quality contract compliance, security, and design fidelity (Figma-sourced slices only). On pass: logs confirmation in the backlog slice detail and automatically invokes solo-qa. On fail: returns specific notes, slice goes back to build. Runs as an isolated sub-agent with no access to build context — sees only the code, the slice spec, and the quality contract. This makes it more likely to surface things the build pass rationalized away.
Key Elements
  • Check 1: Pattern compliance — follows established patterns in the project
  • Check 2: Data sourcing — cross-references against mock data file, no hardcoded values
  • Check 3: Expandability — built to accommodate what's coming, not just today
  • Check 4: Cleanliness — no dead code, no debug artifacts, no commented-out experiments
  • Check 5: Scope discipline — only what was asked for, no extras
  • Check 6: Documentation — purpose comments, complex logic explained, tradeoffs named
  • Check 7: Stack compliance — follows tech-context standards
  • Check 8: Quality contract — reads the pre-written contract before reading the code; checks every line adversarially. A try/catch that swallows the error silently does not satisfy "user sees an error message." Contract line is ✅ only if the specific behavior exists and is reachable.
  • Check 9: Security — reads docs/tech-context.md to determine the project's Security Classification. Runs 5 baseline sub-checks on every slice (9a–9e: input sanitization, data scoped to authenticated user, auth checks server-side, injection prevention, no secrets in client-visible code). Class-specific checks layer on top: Personal (9f–9h), Financial (9i–9k), Authentication (9l–9n), Multi-tenant (9o–9q), Confidential (9r–9s), Regulated (9t). Projects without a Security Classification section default to Standard — baseline only. Up to 20 sub-checks total.
  • Check 10: Design fidelity (Figma-sourced slices only) — (10a) interactive element inventory present in Notes field; (10b) every inventoried element built; (10c) Functional elements have wired logic, Deferred elements are explicit non-interactive placeholders; (10d) representative visual values traceable to Figma node properties. Non-Figma slices: skip stated explicitly as N/A in the report.
When It Runs
  • Framework activates automatically when solo-build marks a slice code-complete
  • No manual trigger needed
  • Quality contract is read first — if missing from the backlog, code review cannot run
  • Reads docs/tech-context.md before Check 9 — Security Classification determines which sub-checks run
Output
  • ✅/❌ per check with specific notes — Check 8 lists each contract line individually; Check 9 lists each sub-check with class label; Check 10 lists each sub-check or states N/A
  • On pass:
    Code review passed [date]
    logged in backlog slice detail
  • On pass: auto-invokes solo-qa
  • On fail: returns to solo-build with specific items to address; slice status → In Build
solo-qa
Auto Part 1 / Solo Part 2
Two-part verification. Part 1 is AI-driven active testing — reads all four anchors, opens a preview, and cross-references rendered values against the mock data file with named evidence. Part 2 is a structured solo browser sign-off with specific done criteria listed. No sign-off permitted while an open missing requirement exists.
Key Elements
  • Gate check first: requires code review log in backlog AND builder confirmation populated before starting
  • Reads all four anchors (design, data, done, process) — not just done criteria
  • Active testing — opens preview, verifies rendered values against mock data file. Evidence required, not code inspection.
  • Design fidelity check against sprint HTML
  • Part 2: structured prompt showing builder confirmation + exact done criteria for solo browser verification
  • Unexpected findings → qa-triage (classify before fixing)
  • Slice status after solo sign-off: In Test (not Done — Done is set by phase-test)
  • Deliverable acceptance — when all slices in a deliverable are In Test, runs a deliverable-level check. Reads plain language description, technical description, screens list, and acceptance criteria from the backlog record. Three types: Screen (visual pass), Logic (evidence pass), Integration (both). Status moves to Accepted on solo sign-off.
When It Runs
  • Framework activates Part 1 automatically when code-review-and-quality passes
  • Part 2 requires solo browser verification before the slice moves to In Test
Output
  • QA verification with named evidence
  • Slice status: In Test in backlog (Done set later by phase-test)
  • Deliverable status: Accepted when all slices verified together
  • Feature branch pushed, PR created
  • Any unexpected findings → qa-triage log
qa-triage
Auto — invoked by solo-qa or phase-test
Routes unexpected QA discoveries to the correct path. Three types: bug (behavior defined, implementation wrong), missing requirement (never defined), regression (was Done, now broken). Classifies, scopes, and routes — never starts fixing before understanding what was found. "Fix the right thing, not the nearest thing."
Key Elements
  • Bug: defined behavior not implemented correctly → fix in current feature branch
  • Missing requirement: never defined → define or defer before sign-off
  • Regression: was Done, now broken → regression branch, root cause first
  • Scope assessment: which slice, which phase, how wide is the impact?
  • No solo sign-off while open missing requirement exists — must be defined or explicitly deferred
When It Runs
  • Framework routes here from solo-qa when testing surfaces something unexpected
  • Framework routes here from phase-test when a stage finds an issue
Output
  • Classification: bug / missing requirement / regression / map-level gap
  • Scope assessment and routing decision
  • qa-triage log entry in backlog
  • Routed to correct fix path — test run continues on unaffected areas
process-change
Solo-invoked or Auto — from solo-build, qa-triage, phase-test
Consistent protocol for changes to the agreed to-be process map — regardless of cause or where it surfaces. Stakeholder input, demo gaps, technical constraints, solo realizations: all receive the same seven-step protocol. The where is unpredictable. The response is not.
Key Elements
  • Cause-agnostic: four trigger sources, one protocol
  • Three-tier impact scan: scheduled work · in-progress work · completed work — every item gets an explicit verdict
  • Solo decides: Revise / Retire / Reopen / Create / Absorb — per affected slice
  • Re-prioritization: resume order + revised schedule + new work from the change, surfaced explicitly
  • Hard gate: solo confirms updated state before build resumes
When It Runs
  • Solo invokes directly when they realize the to-be map needs to change
  • Auto-detected by solo-build when a map-level conflict is found mid-build
  • Auto-detected by qa-triage when a missing requirement has no to-be map step at all
  • Auto-detected by phase-test Stage 6 when acceptance review finds a process-level mismatch
Output
  • Updated to-be map (Refinement / Addition / Restructure)
  • Backlog updated — revised, retired, reopened, or new slices as decided
  • Deliverable records updated where needed
  • Re-prioritized build sequence
  • Process change record in product-continuity

Phase Test & Deploy
Full-phase validation before anything ships. Stack-driven deploy after the gate opens.
2 skills
phase-test
Solo explicit — /phase-test
Full-phase testing orchestrator. Seven specialist stages run in sequence. Invoked explicitly by the solo when they decide the phase is built — a deliberate decision, not automatic. Failures at any stage route to qa-triage; the test run continues on unaffected scenarios. Gate OPEN appends the phase completion record to backlog.md. In Claude Code, Stage 4 (tester) and Stage 5 (regression specialist) run in parallel — same gate logic, faster results. Cursor gets sequential execution.
Key Elements
  • Stage 1: Environment readiness — pre-flight gate. Mock inactive, all slices deployed, app loads clean. Nothing runs until this passes.
  • Stage 2: Use case creator — derives test plan from discovery brief + to-be map. Not invented.
  • Stage 3: Data specialist — mock→real swap confirmed, API read/write verified, data-mapping.md validated
  • Stage 4+5: Tester + Regression specialist — every scenario with named evidence, all In Test slices re-walked
  • Stage 6: Acceptance reviewer — PM lens, does what was built match discovery brief intent?
  • Stage 7: Gate — OPEN (deploy) or HOLD (specific items). HOLD: re-test affected scenarios only.
  • On OPEN: all In Test slices move to Done, phase record moves to Completed, phase completion record appended to backlog.md
When It Runs
  • Explicitly engaged via /phase-test — deliberate, not automatic
  • Readiness table: three states — Ready (all slices In Test, deliverables Accepted, qa-triage clear) · Not Yet (anything still In Build or In QA) · Overdue (solo is adding scope when everything is In Test — invoke now)
  • Phase test isn't evidence you're done — it's the process for finding out if you are
Output
  • docs/phase-test-[phase]-[date].md
  • Full test report — all 7 stages
  • Gate: OPEN or HOLD with specific items
  • On OPEN: slices → Done · phase → Completed · completion record → backlog.md
  • Gate: OPEN required before deploy
deploy
Phase skill
Stack-driven deploy orchestrator. Does not assume a platform — reads tech-context.md to determine the deploy method and executes accordingly. Confirms gate is OPEN, confirms branches are merged, routes to the correct method, and verifies the environment is actually live — not just that the pipeline said it succeeded.
Key Elements
  • Reads tech-context for deploy target, CI/CD system, branch mapping, trigger, verification method
  • Confirms phase test gate OPEN via phase completion record
  • Confirms all phase feature branches merged
  • CI/CD Setup: one-time — generates .github/workflows/ci.yml, walks solo through token/secret setup in plain language per platform
  • Method A: CI/CD triggered by merge (GitHub Actions)
  • Method B: MCP tool triggered (Railway, Vercel, etc.)
  • Method C: Manual CLI — surfaces exact command, waits for solo confirmation
  • Method D: Not yet defined → asks 4 questions, updates tech-context, then deploys
  • Verifies environment live — not just pipeline green
When It Runs
  • Phase 6 — runs after the phase test gate returns OPEN
  • Requires phase completion record in backlog.md before proceeding
Output
  • Deployed environment — confirmed live
  • Phase completion record updated:
    Complete → Tested → Deployed
  • Deploy method, environment, and verification logged in backlog.md

    retrospective
    ⚡ Always-On
    Two-mode continuous learning mechanism. Flag mode captures observations in the moment as lightweight one-line notes — no interruption to flow. Retro mode processes those notes at phase end and after phase test: grouping patterns vs. one-offs, naming root causes, producing proposed fixes, and distinguishing project-level adjustments from framework-level playbook improvements.
    Key Elements

    • Positive observations captured too — what worked gets reinforced, not accidentally removed
    • Two levels: project-level (this project only) vs. framework-level (updates the playbook SKILL.md)
    • Framework improvements queue to framework-improvements.md, applied at next project start

    • Retro mode: after phase test — the most revealing moment
    • Retro mode: user explicit — /retrospective or /retro

    • docs/continuity/framework-improvements.md

    Support
    Post-deploy work on live systems — too focused for a full SBF phase. Invoked via slash command, never auto-loaded. Minimal footprint, deploy alone, verify in production.
    4 skills
    bug-fix
    Support skill
    Fix a production bug correctly — reproduce first, find the root cause, make the smallest fix, verify nothing adjacent broke, deploy. For issues in live systems, not for development-time errors caught before deploy.
    Key Elements
    • Reproduce before touching anything: if you can't reproduce it, you don't have a bug
    • Root cause, not symptom: fixing the symptom brings the bug back
    • Smallest fix: every line changed is a line that can introduce a new problem
    • Regression check: verify the fix AND that nothing adjacent broke
    • Deploy alone: never bundled with other changes
    Output
    • Commit: fix: [what was broken] — root cause: [one line]
    • Verified in production — not just locally

    security-patch
    Support skill
    Address a security vulnerability in a production system — confirm the codebase is actually exposed, apply the minimal fix, verify it's closed, deploy urgently. Speed and correctness are equally required. A patch that doesn't close the vulnerability is worse than the original exposure.
    Key Elements
    • Confirm exposure first: many CVEs affect packages in ways a specific project doesn't use
    • Minimal fix: don't refactor or bundle — patch alone
    • Verify it's closed: reproduce the attack vector, confirm it no longer works
    • Audit for the pattern: don't fix one instance and miss three others
    • Deploy urgently: skips the normal deploy queue
    Output
    • Commit: deps: security patch [package] → [version] — [CVE ID] or fix: close [type] — [CVE ID]
    • Verified closed in production
