Session Modes

Controls how much the framework activates at the start of a session. Bare is the default — the framework is silent until you invoke a skill. Token cost scales with activation: guided (highest) · piloted (moderate) · bare (lowest).

Default

bare

The framework is silent. No routing, no always-on skills. Every skill loads only when explicitly invoked. Maximum token efficiency.

Invoke: "bare mode" · /bare

Active

piloted

Always-on skills activate once and persist for the session. Phase skills are invoked manually — you choose what to run and when.

Invoke: "piloted mode" · /piloted

Guided

guided

Full framework engagement. The start skill reads your opening message and routes automatically. Always-on skills load and persist. The phase chain drives the session.

Invoke: "guided mode" · /guided · "guided on [project name]"

Session hygiene: End sessions at phase gates, not mid-phase. Say "let's close out" — the continuity skill captures state and updates the handoff. To resume: paste the resume prompt or say "guided on [project name]" if registered in projects.md.

Entry & Routing

The front door. Reads context, routes to the right starting point.

start

The framework's front door. Reads the opening message of a new project conversation and routes automatically to Brainstorm (exploratory ideas) or Discover (clear solution in mind). Checks for existing context before routing — never restarts a project already in progress. Detects deployed projects and opens a new cycle. On named project resume, reads only the two active handoff sections on first load — full handoff loads on demand.

AspectDetail
Routing logicReads opening tone — exploratory vs. articulated idea. Asks one question if routing is ambiguous.
Existing contextChecks for blueprint, discovery brief, active sprint before routing — never restarts in-progress projects
Deployed projectsDetects deployed projects — routes to new cycle protocol, opens new phase in existing backlog
OutputRouting decision: Brainstorm, Discover, or new cycle — hands off immediately

onboard

Brings an existing project into the framework without redoing the work. Reads the project directory, semantically maps content to gate artifacts, synthesizes with solo confirmation, places the project at the correct starting phase, and verifies companion compatibility before exiting.

AspectDetail
Reading approachReads docs, notes, and planning files — skips code files. Semantic matching — content-based, not filename-based.
Gap handlingGaps surface as targeted discussion questions. Classifies: undocumented work (ask) vs. work not done yet (starting phase)
Companion passVerifies projects.md registration, backlog section headers, handoff section, and Review URL fields before exiting
OutputFramework gate files synthesized · handoff.md initialized · clear starting phase statement

Discovery

Understanding the problem, the process, and the constraints before anything gets designed.

brainstorming

Product exploration for ideas that aren't fully formed. A genuine collaborative conversation — not a checklist or a spec session. Includes a process sketch moment and a visual sketch mid-session when the idea benefits from something concrete to react to.

AspectDetail
Process sketch momentprocess-mapper activates when conversation surfaces a process — captures preliminary as-is and to-be sketch
Visual sketchRough HTML screen as a thinking tool, not a deliverable. Triggers "warmer / colder" reaction.
ApproachOne question per round — the right question, not the next on a list. Not a spec or feasibility session.
Outputdocs/brainstorm-[date]-[topic].md · idea summary · process sketch · research items · routing decision

discover

The day-one conversation that builds the full product story. Produces the discovery brief and, critically, produces and agrees the as-is and to-be process maps before any design begins. The to-be map agreed here becomes the process contract for every downstream phase.

AspectDetail
ApproachNarrative conversation — not a requirements form. Surfaces who/what/value with specificity, not user types.
Process mapsAs-is map (including manual processes) and to-be map. To-be agreed before design begins — this is the gate.
GateTo-be map agreed before design sprint begins
Outputdocs/discovery-brief.md · docs/process/as-is-[name].md · docs/process/to-be-[name].md

tech-context

Establishes the full technical context before design begins — stack, constraints, infrastructure requirements, deploy configuration, security classification, and Claude API authentication pattern. Every downstream skill reads this document. Required before Design Sprint can begin.

AspectDetail
Deploy pathsPlain-language question maps to one of seven named paths (Local Mac, Cloudflare, Railway, Render, Fly.io, AWS App Runner, No pipeline)
Security classificationAssigns one or more of seven classes: Standard, Personal, Financial, Authentication, Multi-tenant, Confidential, Regulated. Read by code-review-and-quality for security sub-checks.
Infrastructure slicesIdentifies prerequisites that must be built before any feature slice can start
Outputdocs/tech-context.md · stack, constraints, deploy method, branching model, security classification, Claude API auth pattern

research-spike

Timeboxed investigation of a specific unknown. Callable from any phase — when a question surfaces that can't be designed around without an answer. Returns a finding and a recommendation, then hands back to the phase that triggered it.

AspectDetail
ScopeOne specific question — not open-ended exploration. Timeboxed — produces a finding, not a design.
Triggered byAny phase when a specific question blocks progress · design-review when a slice is Blocked · principal-engineer when an architectural unknown surfaces
Outputdocs/spikes/[topic].md · finding: feasible / feasible with tradeoffs / not feasible · recommendation for invoking phase

Design

Producing the shared visual artifact that every downstream phase points at.

design-sprint

Produces the HTML artifact — all screens, good enough to make real decisions from. The design sprint is the visual contract: the first shared artifact everyone can point at and agree on. Reads both the discovery brief and the to-be process map; stops if the to-be map doesn't exist.

AspectDetail
On-ramp AScratch → hero screen → warmer/colder loop → all screens
On-ramp BFigma import via MCP — uses design as source of truth
On-ramp CReference screenshot → aesthetic extraction → produce in that direction
Process threadprocess-mapper cross-references screens against to-be map, annotates with screen refs, flags gaps as decisions
GateDeferred decisions log required before planning
Outputdocs/design/sprint-[id].html · docs/design/deferred-decisions.md · optional stakeholder handoff package

frontend-design

Aesthetic and structural guidance for the HTML artifact. Provides design system direction — color, typography, spacing, component patterns — to ensure visual consistency across all screens produced in the design sprint. Activated by design-sprint, not standalone.

awesome-design-md

On-ramp 3 for the design sprint — reference-based aesthetic extraction. Takes a screenshot or design reference and extracts the design language for use as an aesthetic constraint when producing the HTML artifact. Works from screenshots, live URLs, or verbal description.

Data & Design Review

Generating realistic mock data and converting design screens into a defined, Ready backlog.

data-scaffold

Generates realistic mock data for all screen data entities. Creates a clean mock layer with a single swap seam — easy to replace with real API data when ready. Flags data behavior candidates so design review can ask the right questions before UI slices are finalized.

AspectDetail
Data qualityGenerates realistic (not placeholder) values — names, amounts, dates that look real
Single seamOne place to swap mock for real — nothing hardcoded across the codebase
Behavior candidatesFlags list entities and external API sources in data-mapping.md — these need the design review data behavior pass
UpdatesRe-runs when data questions log has newly resolved entries — pagination edge cases, null field rates, empty/error states seeded
Outputdata/mock/[entity].json · docs/data-mapping.md · proto-API contract

design-review

The iterative loop between design artifact and build decision. Runs as many rounds as the project needs. Each round refines the design, defines slices, and updates the backlog. Build begins when enough slices are Ready — not when all are. Three specialist lenses run every round.

AspectDetail
Three lensesPM (right problem?), Tech Lead (buildable?), Data (where does every field come from?) — every round
Data behavior passRequired for every screen with external data: volume, list behavior, nulls, empty/error/loading states. Produces a data questions log. Unresolved question blocks dependent UI slices from Ready.
Slice requirementsEvery slice needs: plain language description, technical description, process anchor, quality contract (four categories: Failure states, Edge cases, Input validation, Security), design anchor, data anchor, done criteria before reaching Ready
Coverage checkEvery design review round — every to-be step mapped to a slice or explicitly decided
GateEnough Ready slices for Phase 1 before build starts
Outputdocs/backlog.md with full slice records · process coverage map per round · review log

Planning

Sequencing work by risk and process order, then converting to executable issues.

prd-to-plan

Turns the backlog of Ready slices into a sequenced, multi-phase implementation plan. Sequences by two forces in tension — risk order (hardest things first) and process order (to-be map step sequence). Resolves conflicts explicitly. Stops if the to-be process map doesn't exist.

AspectDetail
SequencingRisk order AND process order — conflicts named explicitly. Infrastructure slices always first.
DeliverablesGroups slices by shared outcome. Each deliverable: technical spec (AI-facing), solo description (solo-facing), acceptance criteria (shared). Type assigned: Screen or Logic.
Integration deliverablesEvery Screen deliverable with external data gets a companion Logic deliverable for data integration. Always sequences after its Screen companion is Accepted.
Architecture classificationEach slice classified as Leaf node (self-contained) or Core architecture (touches data models, shared logic, or infrastructure). Unclassified slices block the plan.
OutputPhased implementation plan · backlog.md updated with process anchors · hands off to to-issues on approval

to-issues

Converts the approved implementation plan into GitHub issues in dependency order. Each issue is self-contained — design reference, process anchor, data anchor, done criteria, and dependency links included. A builder can start any slice from its issue alone.

AspectDetail
PrerequisitesPlan approved, gh CLI authenticated, process anchors on every Ready slice
Issue formatTitle: [SL-XXX] [Slice Name] · labels by phase and status · deferred slices excluded
OutputGitHub issues in dependency order · confirmation report: slice ID → issue number → status

to-prd

Synthesizes accumulated project context into a formal PRD. Optional — used when a formal requirements document is needed for stakeholder alignment, approval processes, or documentation. Reads the discovery brief, design decisions, and deferred decisions log.

grill-me

Skeptical PM stress-test. Hard questions, challenged assumptions, no vague answers accepted. Used before committing to a plan, before starting a significant build phase, or anytime the solo wants a second opinion that doesn't pull punches.

AspectDetail
ApproachChallenges the "why" — is this the right problem? Surfaces unstated assumptions and treats them as risks. Not adversarial — rigorous.
OutputList of challenged assumptions · open questions that need answers · confirmed decisions that survived the challenge

Build

Slice-by-slice execution with discipline, anchors, and automatic QA handoff on completion.

solo-build

Slice-by-slice build execution. Creates a feature branch at start, requires four anchors before writing code, follows journey order, surfaces mid-build discoveries immediately, and commits on code-complete — automatically invoking the QA chain. Core architecture slices trigger an automatic principal engineer review before the branch opens.

AspectDetail
Four anchorsDesign, data, done, and process anchors required before writing any code
Architecture gateCore architecture slices: principal engineer review fires automatically before branch opens. No exceptions.
Mid-build discoveriesSurfaced immediately — never silently resolved. Significant changes pause build and trigger design-review round.
Self-verificationChecklist run before code-complete — builder confirmation populated with ✓/✗ results before any handoff
Rollback protocolWhen a slice that has already passed QA needs to be rebuilt, solo confirms before any status changes. Log entry written, slice returns to In Build.
OutputFeature branch · commit: "SL-[ID] code-complete — [name]" · auto-handoff to code-review-and-quality

autopilot

Autonomous build mode. Builds the full product without solo participation after discovery (or design sprint). Solo-simulator plays every gate. First human touch point is phase test. Refinement cycles handle delta iteration.

AspectDetail
Entry pointsPost-discover (leaner anchor) or post-design-review (richer anchor, fewer simulator judgment calls)
Pre-flight checkSimulator reads all discovery docs, surfaces judgment gaps, solo answers before build begins
Refinement cyclesImplementation delta builds after phase test — structural issues exit to guided mode
Output.claude/autopilot-brief.md · .claude/autopilot-decisions.md · all slices built and committed

tdd

Test-driven build technique. Tests are written before implementation — red, green, refactor. Invokable from solo-build when a TDD approach is chosen for a slice. Ensures test coverage is built in, not bolted on.

principal-engineer

Architecture review lens for consequential decisions. Auto-triggered before any core architecture slice enters build. Also invoked on demand when architectural stakes are high. Gives one recommendation (not a menu), names risks and debt explicitly, and holds its position under pushback unless new information changes the analysis.

AspectDetail
Auto-triggersWhen a Core architecture slice moves to In Build · after tech-context for hard architecture questions · mid-build structural discoveries
ApproachOne recommendation — not "here are three options." Names risks, debt, and watch-for signals. Holds position under preference-based pushback.
OutputOne architectural recommendation with reasoning · risks and debt named · decision logged in backlog

QA Chain

Three skills that chain automatically from code-complete — no user invocation needed until browser sign-off. Plus process-change for map-level conflicts.

code-review-and-quality

Ten-check quality gate that runs automatically when solo-build marks a slice code-complete. On pass: logs confirmation in the backlog slice detail and automatically invokes solo-qa. On fail: returns specific notes, slice goes back to build. Runs as an isolated sub-agent with no access to build context.

CheckWhat it verifies
Check 1Pattern compliance — follows established patterns in the project
Check 2Data sourcing — cross-references against mock data file, no hardcoded values
Check 3Expandability — built to accommodate what's coming, not just today
Check 4Cleanliness — no dead code, no debug artifacts, no commented-out experiments
Check 5Scope discipline — only what was asked for, no extras
Check 6Documentation — purpose comments, complex logic explained, tradeoffs named
Check 7Stack compliance — follows tech-context standards
Check 8Quality contract — reads the pre-written contract adversarially. Each line: ✅ only if the specific behavior exists and is reachable.
Check 9Security — 5 baseline sub-checks on every slice (input sanitization, data scoped to authenticated user, auth server-side, injection prevention, no secrets in client code). Class-specific checks layer on top. Up to 20 sub-checks total.
Check 10Design fidelity (Figma-sourced slices only) — interactive element inventory present and correct. Non-Figma slices: skip stated explicitly as N/A.

solo-qa

Two-part verification. Part 1 is AI-driven active testing — reads all four anchors, opens a preview, and cross-references rendered values against the mock data file with named evidence. Part 2 is a structured solo browser sign-off with specific done criteria listed. No sign-off permitted while an open missing requirement exists.

AspectDetail
Gate checkRequires code review log AND builder confirmation populated before starting
Part 1Active testing — opens preview, verifies rendered values against mock data file. Evidence required, not code inspection.
Part 2Structured solo browser sign-off with exact done criteria listed
Deliverable acceptanceWhen all slices in a deliverable are In Test, runs a deliverable-level check. Types: Screen (visual), Logic (evidence), Integration (both). Status → Accepted on solo sign-off.
Slice status after sign-offIn Test — not Done. Done is set by phase-test.

qa-triage

Routes unexpected QA discoveries to the correct path. Three types: bug (behavior defined, implementation wrong), missing requirement (never defined), regression (was Done, now broken). Classifies, scopes, and routes — never starts fixing before understanding what was found.

ClassificationRouting
BugDefined behavior not implemented correctly → fix in current feature branch
Missing requirementNever defined → define or defer before sign-off. No solo sign-off while this is open.
RegressionWas Done, now broken → regression branch, root cause first
Map-level gapNo to-be step for this behavior → auto-invokes process-change

process-change

Consistent protocol for changes to the agreed to-be process map — regardless of cause or where it surfaces. Stakeholder input, demo gaps, technical constraints, solo realizations: all receive the same seven-step protocol.

AspectDetail
TriggersSolo invokes directly · auto-detected by solo-build (map-level conflict) · auto-detected by qa-triage (missing requirement with no to-be step) · phase-test Stage 6 (process-level mismatch)
Impact scanThree-tier: scheduled work · in-progress work · completed work — every item gets an explicit verdict
Solo decidesRevise / Retire / Reopen / Create / Absorb — per affected slice
OutputUpdated to-be map · backlog updated · re-prioritized build sequence · process change record in continuity

Phase Test & Deploy

Full-phase validation before anything ships. Stack-driven deploy after the gate opens.

phase-test

Full-phase testing orchestrator. Seven specialist stages run in sequence. Invoked explicitly by the solo when they decide the phase is built — a deliberate decision, not automatic. Failures at any stage route to qa-triage; the test run continues on unaffected scenarios.

StageWhat it does
Stage 1 — Environment ReadinessPre-flight gate. Mock inactive, all slices deployed, app loads clean. Nothing runs until this passes.
Stage 2 — Use Case CreatorDerives test plan from discovery brief + to-be map. Not invented.
Stage 3 — Data SpecialistMock→real swap confirmed, API read/write verified, data-mapping.md validated
Stage 4+5 — Tester + RegressionEvery scenario with named evidence, all In Test slices re-walked
Stage 6 — Acceptance ReviewerPM lens — does what was built match discovery brief intent? Process-level mismatch → process-change before gate.
Stage 7 — GateOPEN (deploy) or HOLD (specific items). On OPEN: all In Test slices → Done, phase → Completed, completion record appended to backlog.md.

deploy

Stack-driven deploy orchestrator. Does not assume a platform — reads tech-context.md to determine the deploy method and executes accordingly. Confirms gate is OPEN, confirms branches are merged, and verifies the environment is actually live — not just that the pipeline said it succeeded.

MethodDescription
Method ACI/CD triggered by merge (GitHub Actions)
Method BMCP tool triggered (Railway, Vercel, etc.)
Method CManual CLI — surfaces exact command, waits for solo confirmation
Method DNot yet defined → asks 4 questions, updates tech-context, then deploys

Always-On Skills

Activate on guided or piloted mode. One load per session — do not reload on each skill invocation. Silent in bare mode (the default).

process-mapper

Produces and maintains the as-is and to-be process maps throughout the framework. The to-be map agreed in Discover is the process contract — process-mapper holds every downstream phase accountable to it. Never invoked by the user.

PhaseWhat it does
BrainstormCaptures preliminary as-is/to-be sketch as conversation develops
DiscoverFormalizes both maps, validates to-be with solo
Design SprintCross-references screens vs. to-be, annotates map with screen refs, flags gaps
Design ReviewCoverage check every round — every step mapped or decided
BuildProcess anchor verification per slice — stops if slice has no mapping
Phase TestSource for use case derivation

product-continuity

Institutional memory across every session and phase. Captures decisions with reasoning, outstanding questions, assumptions, risks, changes, glossary, target state, and session handoffs. The handoff document is the first thing read when a new session opens — context is never lost between sessions.

DocumentWhat it tracks
sessions.mdLog of every session
decisions.mdWith reasoning and alternatives considered
questions.mdOwner and blocking status
assumptions.mdTreated as true, unverified
risks.mdNamed and tracked
changes.mdDirection changes with context
glossary.mdProject-specific terms
handoff.mdOverwritten each session, first read next session
retrospective.mdProcessed retro findings
onboarding.mdNew collaborator orientation
Long-gap re-entry: Three-check protocol on 21+ day gaps — codebase state, environment, context validity — before any work begins.

framework-health

Background health monitor. Checks signals — not full document reads. Surfaces one issue at a time with a specific recovery path. Silent when everything is healthy — output is always meaningful because it's rare. Never blocks progress.

When it firesWhat it checks
Mode activationExisting context, handoff, framework documents present
Between phasesPhase prerequisites exist
During buildBetween slices — one issue surfaced at a time; fires once when all phase slices are Done (phase test readiness nudge)
Session endHandoff is current
Long-gap detection21+ days since last session triggers re-entry check

retrospective

Two-mode continuous learning mechanism. Flag mode captures observations in the moment as lightweight one-line notes — no interruption to flow. Retro mode processes those notes at phase end: grouping patterns vs. one-offs, naming root causes, producing proposed fixes, distinguishing project-level from framework-level improvements.

ModeWhat it does
Flag modeOne-line observations — date, phase, what happened. No analysis. Captured in the moment.
Retro modeReads retro-notes.md only. Groups pattern vs. one-off vs. positive. Patterns get full treatment: what happened, root cause, impact, proposed fix.
Two levelsProject-level (this project only) vs. framework-level (updates the playbook). Framework improvements queue to framework-improvements.md.

Framework Partner

Scott's named framework partner. Full curator capability plus relationship context — knows the framework history, knows how Scott works, and holds both at once. Discusses, challenges, designs, and builds. Not a recall-only companion. Invoked with /ren or by name.

CapabilityDetail
Full curator capabilityCan propose, review, and execute framework changes — all curator rules apply
Cross-device memoryFetches current framework state from ren-memory repo at every session start

Workshop

For work that isn't a product — spikes, personal tools, exploratory builds. Invoked via slash command, never auto-loaded. The anti-framework: timebox over gates, outcome over process, disposable by default.

scope-check

Entry gate for Workshop. Asks one question — who is this for, and how long does it live? — then routes to Workshop (spike/tool) or hands to the framework (product). Holds the line between exploratory work and product work.

spike

The Workshop execution skill. Timeboxed exploratory work with a single artifact and a running journal of what was tried and what happened. Outcome-first, not process-first. No gates, no anchors, no phases.

AspectDetail
Running journalspike-journal.md — what was tried, what happened, what surprised, current best guess
Single file biasKeep work in as few files as possible — sprawl is a signal
No extrasNo tests, no abstractions, no README — the journal is the README
Graduation triggers"Can I share this?", "Should I add auth?", someone else wants to use it → run land with Promote intent

land

The Workshop exit skill. Three outcomes — Toss (capture learning, archive), Keep (personal tool with minimal hygiene), Promote (hand to the framework with a prepared brief). Every spike ends here. No exceptions.

Toss

Summarize learning → write to MemPalace → archive or delete. You leave with something worth keeping, without a codebase to maintain.

Keep

One-pass hygiene (rename, add header, remove secrets, sanity run) → freeze → log to MemPalace. No framework gates, no shipping requirements.

Promote

Stop treating as spike code. Prepare handoff brief → hand to framework start. Do not build on spike code as if it were production architecture.

Support

Post-deploy work on live systems — too focused for a full framework phase. Invoked via slash command. Minimal footprint, deploy alone, verify in production.

bug-fix

Fix a production bug correctly — reproduce first, find the root cause, make the smallest fix, verify nothing adjacent broke, deploy. Reproduce before touching anything. Root cause, not symptom. Deploy alone, never bundled.

enhancement

Add a small feature or improvement to an existing production application. Reads existing context first: backlog, design screens, process map, code patterns. If the change needs new design screens or a new process flow, route to design-review instead.

dependency-upgrade

Safely update project dependencies — audit what's outdated, check for breaking changes, update one at a time, test, deploy. Security-flagged packages always go first. Read changelogs on major versions before upgrading.

security-patch

Address a security vulnerability in a production system — confirm the codebase is actually exposed, apply the minimal fix, verify it's closed, deploy urgently. Confirm exposure first (many CVEs affect packages in ways a specific project doesn't use). Audit for the pattern — don't fix one instance and miss three others.

Meta / Framework

Skills that operate on the framework itself, not on projects. Used to expand, modify, fix, validate, or communicate about the playbook.

framework-curator

The framework's institutional collaborator. Invoked when expanding, modifying, fixing, or validating the framework itself. Knows every skill, every doc, every gate, every principle. Always proposes changes for approval before executing — never modifies the framework silently.

AspectDetail
Load-bearing checkChecks every proposed change against the principles that define the framework — flags violations explicitly
Documentation cascadeIdentifies every file a change touches (SKILL.md, README, framework doc, skills reference, process map, decks, cursor template)
Proposal-firstDrafts a full change proposal with preview of every edit before touching a single file
Owner/contributor modesOwner can approve and execute framework changes. Contributors can propose only — direct edits redirected to ideas backlog.
Scope boundaryOperates on the framework repo only — never on projects

comms-cascade

Sub-agent that updates all framework communications docs when a version ships. Invoked by the framework curator with a structured brief. Reads every relevant doc, writes updates, commits, pushes to GitHub, and deploys to Cloudflare in one pass. Never invoked directly by the solo.

AspectDetail
Always updatesCHANGELOG.md and blog.html on every version
Conditionally updatesSkills-reference (if skills changed), guide pages (if watchfor items provided), process-map (if process flow changed)
Never touchesManually maintained decks, unlinked reference pages, scripts
Full deploy passCommit → push to GitHub → wrangler pages deploy to Cloudflare, in that order
