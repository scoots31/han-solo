FAQ & Troubleshooting

Answers organized by situation — the real questions that come up mid-session, not a catalog of features. Find your situation, get the answer, keep going.

Before Your First Session
3 questions
Q Do I need to read all the skills before I start?
No. Say "let's start" and describe what you want to build. The framework routes you to the right skill automatically. You'll encounter skills in context, which is a much better way to learn them than reading ahead.
Q Which session mode should I use for my first project?
Guided. It loads full guidance automatically and makes decisions about sequencing for you. Once you've been through a full project cycle and know how the phases connect, switch to piloted — it gives you more control without the hand-holding.
Q I have an existing project with code and notes. Do I start from scratch?
No. Run onboard directly. It reads your project directory, semantically maps prior work to framework artifacts, confirms its interpretation with you, and places you at the correct starting phase. Nothing gets thrown away — it gets organized. Don't run start first; onboard is the entry point when prior work exists.

Mid-Discovery
3 questions
Q Discover keeps going — how do I know when it's done?
The gate is two artifacts: docs/discovery-brief.md and docs/process/to-be-[name].md. When both exist and the to-be map is agreed, discover is done. If either is missing or still in draft, you're not through the gate.
Q I don't know what my to-be process looks like yet. Can I skip it?
No.
The to-be map is what every downstream skill builds against. Design-sprint cross-references screens against it. Design-review cross-references slices against it. Without it, everything downstream is building against nothing. Go deeper in discover — Zone 2 (the journey) is where the to-be emerges.
Q My idea changed significantly during discovery. Do I start over?
Update the discovery brief and to-be map — don't restart. The framework works with living documents. Capture the new direction in the brief, update the to-be map to reflect it, and continue. Design-sprint reads the updated brief and builds from there.

Mid-Design
4 questions
Q The warmer/colder loop in design-sprint isn't converging. We keep going in circles.
Stop the design sprint and go back to discover.
The loop stalls when the story isn't clear yet. More design rounds won't fix a fuzzy journey — a clearer story will. Return to Zone 2 (the journey) in discover, deepen it until the sequence of steps is unambiguous, then re-enter design-sprint.
Q Design-review keeps producing new slices. When does it end?
Design-review ends when there's a coherent set of Ready slices for Phase 1 — not when all slices across the project are Ready. Build starts when Phase 1 is covered. Design-review continues in parallel for later phases as build is underway.
Q A stakeholder wants to see the design before I proceed. What do I share?
Design-sprint produces an optional stakeholder handoff package at docs/design/handoff/<date>/ — HTML artifact, PDF export, and a structured feedback template. Share whichever format works for your stakeholder. Completed feedback goes in docs/stakeholder-feedback/ and design-review reads it on the next round.
Q A slice is blocked on a spike and the spike isn't resolving.
Timebox the research-spike hard. If it won't resolve within the timebox, close it — document what was tried, what was learned, and what remains unknown — and mark the blocked slice as Deferred in the backlog with a clear trigger: "revisit when X is resolved."
Don't let one unresolved research-spike hold the entire backlog. A blocked slice is a known risk. Name it, defer it, and keep the rest of Phase 1 moving. Note: research-spike is an SBF skill — it's not the same as a Workshop spike, and land does not apply here.

Mid-Build
3 questions
Q grill-me found a blocker I can't resolve. Do I keep building?
No.
A real blocker means the plan has a hole. Either resolve it — research spike, architecture decision, external dependency confirmed — or pivot the plan. Building through a known blocker produces work you'll throw away. Stop, address it, then continue.
Q A mid-build discovery changes the design. Do I go back to design-review?
Surface it immediately — solo-build has a "mid-build discoveries surfaced immediately" rule. If it's significant enough to change a slice's behavior or add a new one, pause build and run another design-review round. If it's minor, log it in the deferred decisions log and keep going. The key is not to silently absorb it into the code.
Q I lost my session context. How do I resume?
Find the resume prompt generated by product-continuity at your last close-out and paste it into a new session. start reads it and picks up where you left off. If you didn't close out cleanly, read docs/HANDOFF.md directly — product-continuity maintains it throughout the session, not just at close-out.
This is why close-out matters. A clean close-out takes 30 seconds and makes the next session instant. Skipping it adds 10 minutes of re-orientation every time.

QA & Testing
3 questions
Q Solo-qa keeps finding issues. When do I declare a slice Done?
Done requires three things: code review logged, AI verified with evidence, and solo confirmed in the browser. All three. No shortcuts.
If solo-qa keeps surfacing issues, the done criteria may be too vague. Sharpen the slice definition before re-running. A moving target never passes QA — a clear target usually does on the first serious pass.
Q Phase test failed. Do I re-run the whole thing?
No. Fix the affected items, then re-test only the affected scenarios. Unaffected scenarios don't need to re-run — they passed. If you're unsure which scenarios are affected by your fix, run qa-triage — it classifies what changed and scopes the minimum re-test set.
Q QA found something that wasn't in the original slice definition. What now?
Route it through qa-triage. It classifies the finding as one of three things: a bug (fix now, it was always implied by the slice), a missing requirement (scope it as a new slice and decide when it ships), or a regression (identify what change caused it before touching anything else).
Don't silently expand the slice. Every new item gets a triage decision. Undefined scope grows into an undefined product.

Out of Order / Skipped a Phase
3 questions
Q I skipped discover and went straight to building. Now what?
Run onboard. It reads what you've built, maps it to framework artifacts, and identifies what's missing. You'll likely need to produce a discovery brief and to-be map retroactively — onboard surfaces the gaps and gives you a path to fill them. The work you've already done counts; it just needs framing.
Q My project already has a Figma and partial backend. Where do I enter?
Run onboard directly. It reads your project directory, maps what exists to framework artifacts, and places you at the correct starting phase. For the Figma specifically — design-sprint has On-ramp 2 for existing Figma files, which imports screens via Figma MCP rather than generating new ones. You don't recreate the design work; you bring it in and run the review loop from there.
Q The to-be process map changed mid-project. Do I restart?
Update the map. Process-mapper (in piloted or guided mode) flags drift between the updated map and existing slices — which slices no longer map to a process step, which new steps need coverage. Run a design-review round to surface the gaps. The project doesn't restart; the backlog gets realigned.
A changed to-be map is normal. Discovery is iterative. The framework is designed to handle drift — the answer is always "update and realign," never "throw away and restart."

Workshop Questions
3 questions
Q How do I know if something is a spike or a product?
Run /scope-check. It asks one question: who is this for, and how long does it live? If the answer is "for me, hours to days" — it's a spike. If the answer is "for others, indefinitely" — it's a product. The routing from there is automatic.
Don't guess. Starting a spike when you mean a product (or vice versa) sets the wrong expectations from session one. scope-check takes 60 seconds and gets you on the right track before you write a line.
Q My spike is turning into something bigger. Do I keep spiking?
No. Stop and run land with Promote intent.
Spike code is a prototype, not a foundation. Prepare a handoff brief: what was proved, what's in the code, what the graduation triggers revealed, known hacks. Hand it to SBF start. The framework takes it from there with the right framing. Do not build on spike code as if it were production architecture.
Q I landed a spike as Keep. Now I want to share it with others. What do I do?
Run land again with Promote intent. The prior Keep decision is overridden by the new intent. The scope has changed — it's no longer a personal tool, it's a product — and that change needs to be explicit. Prepare the handoff brief and hand to SBF start to begin the product build properly.
