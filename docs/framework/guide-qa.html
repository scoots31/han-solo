Solo Builder Framework — Phase Guide

QA & Acceptance

QA is not a step at the end of build. It is a chain that auto-activates the moment code-complete is declared. A slice is Done only when all three links in the chain have passed. No shortcuts — no exceptions.

What This Phase Is

The moment solo-build declares a slice code-complete, the QA chain activates automatically. It does not wait for the solo to invoke it. It does not skip. It runs.

There are three links in the chain. Each one covers something the previous cannot verify. Code quality is not sufficient on its own. AI verification is not sufficient on its own. The solo's eyes in a running browser are not optional. All three are required, in order, for a slice to reach Done.

Link 1 — code-review-and-quality: Auto-invoked by solo-build on code-complete. Runs 7 structured checks. On pass, it logs confirmation in the backlog and invokes solo-qa. On fail, the slice returns to In Build.

Link 2 — solo-qa Part 1 (AI verification): The AI reads the full slice spec — all four anchors — then actively tests every done criterion against the running output with named evidence. Not code inspection. Evidence means values cross-referenced against mock data, states observed in the preview, edge cases checked against the seeded mock. If anything fails, the slice goes back to In Build with specific notes.

Link 3 — solo-qa Part 2 (solo browser sign-off): The AI cannot do this part. The solo opens the running slice in a browser, alongside the design reference, and confirms each done criterion holds in practice. Not from memory. Not from a summary. Live, running software in the browser. This is the final gate and it cannot be substituted or skipped.

Skills in This Phase

code-review-and-quality — Auto-invoked

The quality gate that runs first. Invoked automatically by solo-build when a slice is declared code-complete. Runs 7 structured checks. On pass, logs confirmation in the backlog and chains directly to solo-qa. On fail, the slice returns to In Build — QA does not proceed.

Invocation
  • Auto-invoked by solo-build on code-complete
  • Never invoked manually mid-slice
  • Can be run directly if a solo suspects a quality issue outside of a formal build session
What it checks
  • 7 structured quality checks (see full list below)
  • Pattern conformance, data sourcing, expandability, cleanliness, scope, docs, stack
Outcomes
  • Pass: logs confirmation in backlog, invokes solo-qa automatically
  • Fail: returns slice to In Build with specific notes — QA does not proceed

solo-qa — Two-part

Active verification that the right thing was built — and confirmation in practice that it works. Part 1 is the AI actively testing against the full slice spec with evidence. Part 2 is the solo confirming in the browser. Both are required. Neither can substitute for the other.

Part 1 — AI verification
  • Gate check: code review pass logged in backlog
  • Read all 4 anchors: design, data, done criteria, process
  • Active verification of each done criterion with named evidence
  • Design fidelity against the design screen
  • Edge cases verified from mock data
Part 2 — Solo browser sign-off
  • Open the running slice in the browser — not a screenshot
  • Design reference open alongside
  • Confirm each done criterion holds in practice
  • Visual check, behavior check, feel check
  • Cannot be performed by the AI
Outcomes
  • Pass: slice marked In Test, backlog updated
  • Fail (Part 1): back to In Build with specific notes
  • Unexpected discovery: invoke qa-triage immediately
  • Open missing req: define or defer before sign-off prompt

qa-triage — On-demand

Routes unexpected discoveries during QA to the correct path. Invoked from solo-qa when something surfaces that isn't a clean pass or a simple build fix. Classifies the discovery — bug, missing requirement, or regression — before deciding what to do.

When it fires
  • During solo-qa when something unexpected surfaces
  • During build when the solo spots something unexpected
  • Post-deploy if a defect surfaces in use
  • Any time the right response isn't clearly "pass" or "back to In Build"
What it does
  • Classifies: bug, missing requirement, or regression
  • Determines scope: current slice, Done slice, unbuilt, adjacent, flow-level
  • Routes to the correct path based on type + scope
  • Logs every triage decision in the backlog
Key behavior
  • QA chain continues on unaffected scenarios while triage runs
  • Small scope discoveries resolved fast — large scope routed to design review
  • Every decision is logged — next session inherits the context
  • Never patch a regression without finding the root cause first

The 7 Checks

Every check in code-review-and-quality. All 7 must pass before QA proceeds. A single fail returns the slice to In Build.

#CheckWhat it means
1 pattern The implementation follows established patterns in the codebase. No new patterns introduced without justification. Style, structure, and naming conventions match what already exists.
2 data sourcing Data flows from the mock layer correctly. Values are read from the defined mock files — not hardcoded in the component. Fields match the established data model.
3 expandability The implementation can accommodate future slices without requiring structural rewrites. Seams exist where needed. Hardcoded structure that will need to change is flagged, not silently introduced.
4 cleanliness No dead code, commented-out blocks, debug artifacts, or unused imports. The implementation is production-clean — not just functional.
5 scope The implementation covers exactly the slice spec — nothing more, nothing less. Features not in the slice definition were not speculatively added. No slice drift.
6 docs Anything that needs to be understood by the next session is documented at the appropriate level. Non-obvious decisions have a brief note. Complex logic has a comment.
7 stack The implementation stays within the defined stack and architecture constraints from docs/tech-context.md. No new dependencies introduced without explicit decision. No architectural deviations from established patterns.

What "Done" Actually Requires

A slice moves through two status milestones before it is truly Done. Solo QA sign-off is not the finish line — it is the gate to In Test. Done requires phase-test to confirm the slice works as part of the complete phase, end-to-end with real data.

Solo confirmed in browser solo-qa Part 2 → In Test The solo looked at the running slice in a browser — not a screenshot, not a summary — with the design reference open alongside. Phase-level verification phase-test → Done Phase-test confirms all slices work together as a complete experience with real data. When the gate opens, every In Test slice in the phase moves to Done and the phase record moves to Completed.
One hard rule: no solo sign-off while an open missing requirement exists. If testing surfaces behavior that was never defined, it must be defined (added to done criteria) or explicitly deferred to a new slice before the sign-off prompt goes out.

qa-triage — When It Fires

When something unexpected appears during QA — during AI verification, during solo sign-off, or at any other moment — invoke qa-triage immediately. Do not improvise a fix without classifying what was found first. The classification determines the correct path.

Bug

The behavior is defined. The implementation doesn't match it. The spec exists — the code doesn't follow it.

  • Current slice: Return to In Build with specific notes. Re-run full QA chain when fixed.
  • Done slice: Reopen that slice. Status → In Build. Full QA chain before returning to Done.
  • Unbuilt slice: Add a note to that slice's backlog record. Continue current QA.
  • Flow-level: Flag for design review. Don't try to fix it in QA.
Missing Requirement

The behavior was never defined. This is a specification gap — not a code defect.

  • Small gap (1 behavior): Define it now. Add to done criteria. Verify before sign-off.
  • Larger gap (slice-level): Create a new slice. Current slice ships with a scope note referencing the new slice ID.
  • Multiple slices / flow: Design review trigger. Don't define slice-spanning requirements in a QA session.
  • Hard rule: No solo sign-off on a slice with an open missing requirement.
Regression

Something previously Done and working is now broken. A later slice, shared component change, or dependency update caused it.

  • Single Done slice: Reopen it. Root cause note required. Full QA chain before Done again.
  • Multiple Done slices: Identify the shared cause first. Fix the root, then re-QA affected slices together.
  • Widespread: Flag before touching anything. Diagnose root cause and blast radius before fixing anything.
The QA chain continues on unaffected scenarios while triage handles the discovery. A bug in an unbuilt slice does not stop QA on slices currently in flight. Large-scope discoveries — flow-level issues — do pause affected slices and route to design review, but the rest of the build continues.

The Gate

Gate condition: all phase slices at In Test status.

Every slice in the phase must reach In Test status before the phase gate opens. Not most of them. Not the important ones. Every one.

When all slices in a deliverable are In Test, QA runs a deliverable-level acceptance check. When the solo accepts, deliverable status moves to Accepted. Once all deliverables are Accepted, the phase is ready for Phase Test.

Phase Test does not auto-trigger — the solo invokes it explicitly. Phase Test confirms the complete phase works end-to-end with real data, and its gate-open moves all In Test slices to Done and the phase to Completed.

Handoff to Phase Test

docs/backlog.md

The primary reference. All slices at Done status with code review, AI verification, and solo sign-off recorded in each slice's detail. Phase test reads this to know what was built, what criteria were met, and that the QA chain passed for every slice.

Design screens

The fidelity reference. Phase test's design reviewer compares the running implementation against the design screens to assess whether what was shipped matches the intent of what was designed.

Discovery brief

The intent reference. Phase test's acceptance reviewer (PM lens) checks whether what was built matches what discovery established as the product intent — not just whether it matches the design.

Phase test runs 7 specialist lenses: functional testing, regression testing, data integrity, UX review, accessibility, design fidelity, and acceptance review. It produces a gate decision — OPEN (proceed to deploy) or HOLD (specific items to resolve before proceeding). The output is a phase test report appended to backlog.md. A gate OPEN decision is what authorizes the deploy skill to run.

What to Watch For

Anti-patternWhy it's a problem
Declaring Done without solo browser sign-off The solo sign-off is not a rubber stamp — it requires the solo to look at running software in a browser. "I'll check it later" is not a sign-off. A summary of what was built is not a sign-off. Wait for it. Every time.
Solo sign-off with an open missing requirement If testing surfaces behavior that was never defined, it must be classified and handled before sign-off. Giving a sign-off with undefined behavior in the slice is not a QA pass — it is undefined behavior shipping as if it were intentional.
Silently absorbing unexpected discoveries instead of routing through qa-triage A missing requirement patched as a bug produces code that solves the wrong problem. A regression fixed in the current slice without finding the root cause reappears in the next build. Classify it first. Then route it. Then fix it.
Treating code inspection as active verification Evidence means values observed in the running preview cross-referenced against mock files. "The code reads from mock data" is code inspection — not evidence. Open the preview. Look at the output. Cross-reference the values.
Starting solo-qa without a code review confirmation in the backlog solo-qa Part 1 begins with a gate check: is the code review pass logged in the backlog? If it isn't there, Part 1 stops immediately. The chain does not run in reverse order. No confirmation in the backlog means run code-review-and-quality first.
