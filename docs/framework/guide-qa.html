Solo Builder Framework — Phase Guide

QA & Acceptance

QA is not a step at the end of build. It is a chain that auto-activates the moment code-complete is declared. A slice is Done only when all three links in the chain have passed. No shortcuts — no exceptions.

What this phase is
Code-complete is not Done. This is what it takes to get there.

The moment solo-build declares a slice code-complete, the QA chain activates automatically. It does not wait for the solo to invoke it. It does not skip. It runs.

There are three links in the chain. Each one covers something the previous cannot verify. Code quality is not sufficient on its own. AI verification is not sufficient on its own. The solo's eyes in a running browser are not optional. All three are required, in order, for a slice to reach Done.

Link 1 — code-review-and-quality: Auto-invoked by solo-build on code-complete. Runs 7 structured checks against quality standards: pattern conformance, data sourcing discipline, expandability, cleanliness, scope adherence, documentation, and stack. This is the gate before verification can begin. On pass, it logs confirmation in the backlog and invokes solo-qa. On fail, the slice returns to In Build.

Link 2 — solo-qa Part 1 (AI verification): The AI reads the full slice spec — all four anchors — then actively tests every done criterion against the running output with named evidence. Not code inspection. Not assumptions from reading source. Evidence means values cross-referenced against mock data, states observed in the preview, edge cases checked against the seeded mock. Design fidelity is verified against the design screen. If anything fails, the slice goes back to In Build with specific notes.

Link 3 — solo-qa Part 2 (solo browser sign-off): The AI cannot do this part. The solo opens the running slice in a browser, alongside the design reference, and confirms each done criterion holds in practice. Not from memory. Not from a summary. Live, running software in the browser. This is the final gate and it cannot be substituted or skipped.

Skills in this phase
Three skills. One chain. Each one covering what the last cannot.
code-review-and-quality
Auto-invoked
The quality gate that runs first. Invoked automatically by solo-build when a slice is declared code-complete. Runs 7 structured checks. On pass, logs confirmation in the backlog and chains directly to solo-qa. On fail, the slice returns to In Build — QA does not proceed.
Invocation
  • Auto-invoked by solo-build on code-complete
  • Never invoked manually mid-slice
  • Can be run directly if a solo suspects a quality issue outside of a formal build session
What it checks
  • 7 structured quality checks (see full list below)
  • Pattern conformance, data sourcing, expandability, cleanliness, scope, docs, stack
Outcomes
  • Pass: logs confirmation in backlog, invokes solo-qa automatically
  • Fail: returns slice to In Build with specific notes — QA does not proceed
solo-qa
Two-part
Active verification that the right thing was built — and confirmation in practice that it works. Part 1 is the AI actively testing against the full slice spec with evidence. Part 2 is the solo confirming in the browser. Both are required. Neither can substitute for the other.
Part 1 — AI
  • Gate check: code review pass logged in backlog
  • Read all 4 anchors: design, data, done criteria, process
  • Active verification of each done criterion with named evidence
  • Design fidelity against the design screen
  • Edge cases verified from mock data
Part 2 — Solo
  • Open the running slice in the browser — not a screenshot
  • Design reference open alongside
  • Confirm each done criterion holds in practice
  • Visual check, behavior check, feel check
  • Cannot be performed by the AI
Outcomes
  • Pass: slice marked Done, backlog updated, git action per branching model
  • Fail (Part 1): back to In Build with specific notes
  • Unexpected discovery: invoke qa-triage immediately
  • Open missing req: define or defer before sign-off prompt
qa-triage
On-demand
Routes unexpected discoveries during QA to the correct path. Invoked from solo-qa when something surfaces that isn't a clean pass or a simple build fix. Also invokable directly when the solo notices something outside a formal QA session. Classifies the discovery — bug, missing requirement, or regression — before deciding what to do.
When it fires
  • During solo-qa when something unexpected surfaces
  • During build when the solo spots something unexpected
  • Post-deploy if a defect surfaces in use
  • Any time the right response isn't clearly "pass" or "back to In Build"
What it does
  • Classifies: bug, missing requirement, or regression
  • Determines scope: current slice, Done slice, unbuilt, adjacent, flow-level
  • Routes to the correct path based on type + scope
  • Logs every triage decision in the backlog
Key behavior
  • QA chain continues on unaffected scenarios while triage runs
  • Small scope discoveries resolved fast — large scope routed to design review
  • Every decision is logged — next session inherits the context
  • Never patch a regression without finding the root cause first

The 7 checks
Every check in code-review-and-quality. All 7 must pass before QA proceeds.
code-review-and-quality quality gate
All 7 required
These checks are not a suggestion list — they are all required. A single fail returns the slice to In Build. The code review does not proceed to partial confirmation. The chain does not continue to solo-qa until all 7 pass.
1
pattern
The implementation follows established patterns in the codebase. No new patterns introduced without justification. Style, structure, and naming conventions match what already exists.
2
data sourcing
Data flows from the mock layer correctly. Values are read from the defined mock files — not hardcoded in the component. Fields match the established data model. No silent assumptions about data shape.
3
expandability
The implementation can accommodate future slices without requiring structural rewrites. Seams exist where needed. Hardcoded structure that will need to change is flagged, not silently introduced.
4
cleanliness
No dead code, commented-out blocks, debug artifacts, or unused imports. The implementation is production-clean — not just functional. Code that only exists for debugging was removed before code-complete was declared.
5
scope
The implementation covers exactly the slice spec — nothing more, nothing less. Features not in the slice definition were not speculatively added. No slice drift. Adjacent concerns that surfaced were noted, not silently absorbed.
6
docs
Anything that needs to be understood by the next session is documented at the appropriate level. Non-obvious decisions have a brief note. Complex logic has a comment. Nothing requires archaeology to understand intent.
7
stack
The implementation stays within the defined stack and architecture constraints from docs/tech-context.md. No new dependencies introduced without explicit decision. No architectural deviations from the established patterns. The slice builds within the system, not around it.

What "Done" actually requires
Four checks, two statuses. Solo sign-off moves the slice to In Test. Phase-test moves it to Done.
The complete Done definition
All required
A slice moves through two status milestones before it is truly Done. Solo QA sign-off is not the finish line — it is the gate to In Test. Done requires phase-test to confirm the slice works as part of the complete phase, end-to-end with real data. Meeting all QA checks is required but not sufficient on its own.
What Who Status What it means in practice
Code review logged code-review-and-quality logged in backlog All 7 quality checks passed and the confirmation is recorded in the slice's backlog entry with the date. This is the gate that solo-qa checks before beginning Part 1.
AI verified with evidence solo-qa Part 1 named evidence Every done criterion tested actively against the running output — not inferred from code. Evidence means: a value observed in the preview cross-referenced against the mock file. A state rendered in the UI. Not "the code reads from mock data."
Solo confirmed in browser solo-qa Part 2 solo sign-off → In Test The solo looked at the running slice in a browser — not a screenshot, not a summary — with the design reference open alongside. Walked through each criterion. Gave an explicit confirmation. Slice moves to In Test. Not Done yet — phase-test is the final gate.
Phase-level verification phase-test gate open → Done Phase-test confirms all slices work together as a complete experience with real data. When the gate opens, every In Test slice in the phase moves to Done and the phase record moves to Completed.

One hard rule: no solo sign-off while an open missing requirement exists. If testing surfaces behavior that was never defined, it must be defined (added to the slice's done criteria) or explicitly deferred to a new slice before the sign-off prompt goes out. Undefined behavior that ships with a sign-off becomes assumed behavior — and assumed behavior is a debt that compounds across slices.

qa-triage — when it fires
Not everything that surfaces during QA is a clean pass or a simple build fix.
Three classifications. Each one routes differently.
When something unexpected appears during QA — during AI verification, during solo sign-off, or at any other moment — invoke qa-triage immediately. Do not improvise a fix without classifying what was found first. The classification determines the correct path. Getting it wrong produces the wrong fix, or no fix at all.
Bug
The behavior is defined. The implementation doesn't match it. The spec exists — the code doesn't follow it.
Current slice: Return to In Build with specific notes. Re-run full QA chain when fixed.
Done slice: Reopen that slice. Status → In Build. Full QA chain before returning to Done.
Unbuilt slice: Add a note to that slice's backlog record. Continue current QA — don't pause.
Flow-level: Flag for design review. Don't try to fix it in QA.
Missing Requirement
The behavior was never defined. This is a specification gap — not a code defect. The implementation may be reasonable, but it was never agreed on.
Small gap (1 behavior): Define it now. Add to done criteria. Verify before sign-off.
Larger gap (slice-level): Create a new slice. Current slice ships with an explicit scope note referencing the new slice ID.
Multiple slices / flow: Design review trigger. Don't define slice-spanning requirements in a QA session.
Hard rule: No solo sign-off on a slice with an open missing requirement.
Regression
Something previously Done and working is now broken. A later slice, shared component change, or dependency update caused it.
Single Done slice: Reopen it. Root cause note required — what changed and why. Full QA chain before Done again.
Multiple Done slices: Identify the shared cause first. Fix the root, then re-QA affected slices together.
Widespread: Flag before touching anything. Diagnose root cause and blast radius before fixing anything.

The QA chain continues on unaffected scenarios while triage handles the discovery. A bug in an unbuilt slice does not stop QA on slices currently in flight. A missing requirement with a clean scope boundary does not stall the entire build. Large-scope discoveries — flow-level issues — do pause affected slices and route to design review, but the rest of the build continues.

The gate
When this phase is complete and the next can begin.
Gate condition: all phase slices at In Test status

Every slice in the phase must reach In Test status before the phase gate opens. Not most of them. Not the important ones. Every one. A slice reaches In Test only when all QA chain links have passed: code-review-and-quality confirmed, AI verified with evidence, solo confirmed in browser.

When all slices in a deliverable are In Test, QA runs a deliverable-level acceptance check. This is a separate, higher-level verification — not a repeat of slice QA. solo-qa reads the deliverable's plain language description, technical description, screens list, and acceptance criteria from the backlog record. Three deliverable types, three review methods: Screen deliverables get a visual pass in the browser against the deliverable acceptance criteria. Logic deliverables get an evidence pass — test results, data state, or a simulated flow. Integration deliverables get both — visual output and data source evidence. When the solo accepts, deliverable status moves to Accepted.

Once all deliverables are Accepted, the phase is ready for Phase Test. Phase Test does not auto-trigger — the solo invokes it explicitly. Phase Test confirms the complete phase works end-to-end with real data, and its gate-open moves all In Test slices to Done and the phase to Completed.

Handoff to Phase Test
What phase test reads when the solo invokes it.
What phase test reads at invocation
Invoked explicitly by the solo — not auto-triggered
docs/backlog.md
The primary reference. All slices at Done status with code review, AI verification, and solo sign-off recorded in each slice's detail. Phase test reads this to know what was built, what criteria were met, and that the QA chain passed for every slice.
design screens
The fidelity reference. Phase test's design reviewer compares the running implementation against the design screens to assess whether what was shipped matches the intent of what was designed. The sprint artifact from design-sprint is the authoritative source.
discovery brief
The intent reference. Phase test's acceptance reviewer (PM lens) checks whether what was built matches what discovery established as the product intent — not just whether it matches the design. Problems surface here when implementation is technically correct but misses the point.

Phase test runs 7 specialist lenses across the completed phase: functional testing, regression testing, data integrity, UX review, accessibility, design fidelity, and acceptance review. It produces a gate decision — OPEN (proceed to deploy) or HOLD (specific items to resolve before proceeding). Failures at any stage in phase test invoke qa-triage, but the test run continues on unaffected scenarios.

The output of phase test is a phase test report appended to backlog.md. A gate OPEN decision is what authorizes the deploy skill to run.

What to watch for
The failures that happen here are quiet ones. They don't announce themselves.
✕
Declaring Done without solo browser sign-off
This is the most common failure. The AI verification passes, everything looks correct in the evidence, and there's a pull to mark it Done and move on. The solo sign-off is not a rubber stamp — it is a genuine check the AI cannot perform. It requires the solo to look at running software in a browser. "I'll check it later" is not a sign-off. A summary of what was built is not a sign-off. Wait for it. Every time.
✕
Solo sign-off with an open missing requirement
If testing surfaces behavior that was never defined, it must be classified and handled before sign-off. Define it (add to done criteria and verify) or defer it to a new slice with an explicit scope note. Giving a sign-off with undefined behavior in the slice is not a QA pass — it is undefined behavior shipping as if it were intentional. Define or defer. Then sign off.
✕
Silently absorbing unexpected discoveries instead of routing through qa-triage
When something unexpected surfaces — a behavior that seems off, a state that wasn't accounted for, something that was Done but now looks wrong — the instinct is often to fix it and move on. This produces bad fixes. A missing requirement patched as a bug produces code that solves the wrong problem. A regression fixed in the current slice without finding the root cause reappears in the next build. Classify it first. Then route it. Then fix it.
✕
Treating code inspection as active verification
Evidence in solo-qa Part 1 means values observed in the running preview cross-referenced against mock files. "The code reads slot_target from the mock layer" is not evidence — it is code inspection. It does not confirm the rendered value matches the mock. It does not confirm edge cases render without breaking. Open the preview. Look at the output. Cross-reference the values.
✕
Starting solo-qa without a code review confirmation in the backlog
solo-qa Part 1 begins with a gate check: is the code review pass logged in the backlog for this slice? If it isn't there, Part 1 stops immediately and returns a specific message. The chain does not run in reverse order. Code review before QA. No confirmation in the backlog means run code-review-and-quality first.
