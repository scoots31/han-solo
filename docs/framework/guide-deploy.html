6 ·Deploy

The finish line. Deploy is the only phase that is entirely determined by the stack — it reads docs/tech-context.md to know what to do and does not ask questions that tech-context already answered. Its job is to confirm the gate is open, confirm code is ready, execute the correct deploy method for the stack, and then verify the environment is actually live — not just that a pipeline went green.

What this phase is
Deploy is stack-determined execution — not a decision point, not a QA loop

Every other phase in the framework involves judgment: what to discover, how to design, what to build first, how to test it. Deploy involves almost none of that. By the time deploy runs, the decisions are locked in. The code is correct — phase test confirmed it. The scope is settled — the backlog says so. The only question is: how does this project ship, and does it actually ship correctly?

The answer to the first question lives in docs/tech-context.md. The deploy skill reads it, finds the deploy method — CI/CD pipeline, MCP tool, manual CLI command, or an undefined method that needs to be filled in — and routes accordingly. It does not guess. If the method is not defined, deploy stops and asks exactly the right questions to define it, then proceeds.

The answer to the second question requires active verification. Deploy does not stop at "the pipeline went green." A pipeline that completes successfully and an environment that is actually running correctly are two different things. The skill verifies the environment is live using whatever method tech-context defines — hitting an endpoint, checking a health route, confirming a version string, validating a module registered in the target platform. Until that verification passes, the deploy is not done.

There are two deploy moments in a full phase lifecycle. The first is before phase test: all Done slices need to be accessible in a test environment so phase-test Stage 1 can confirm environment readiness. Deploy confirms this happened — it does not initiate it. The second moment is the one this guide covers: after the phase test gate opens, the explicit production deploy.

💡
The gate is a prerequisite, not a step. Deploy cannot begin until the phase test gate shows OPEN. This is not a soft check — if the gate is not open, deploy stops completely and directs back to /phase-test. The gate exists precisely so deploy is never executing in parallel with open quality questions.

Deploy methods and deployment paths
Deploy routes to the correct method based on what tech-context defines. When GitHub Actions is selected but not yet configured, CI/CD Setup runs first — one time, at first deploy.
📌
Method is determined by tech-context, not by the deploy skill. If the method isn't defined in docs/tech-context.md, the deploy skill stops and asks what it needs to know before writing it in and proceeding — Method D handles that path explicitly.

CI/CD Setup — one-time, fires at first deploy when GitHub Actions is selected

When tech-context records CI/CD: GitHub Actions but no .github/workflows/ci.yml exists, the deploy skill sets it up before routing to Method A. The framework generates the workflow file entirely — the solo never writes or reviews YAML. The only solo action is a one-time token or key paste in GitHub and the platform dashboard — walked through step by step in plain language.

What the solo does (one time only):
  • Railway — copy a token from the Railway dashboard, paste it as a GitHub secret named RAILWAY_TOKEN
  • Render — copy a deploy hook URL from Render, paste as RENDER_DEPLOY_HOOK
  • Cloudflare — create an API token, paste as CLOUDFLARE_API_TOKEN
  • Fly.io — run one terminal command to get a token, paste as FLY_API_TOKEN
  • AWS App Runner — create IAM credentials + copy service ARN, paste four secrets (~15 min, most setup of any path)

After setup: every push to the base branch runs the full test suite automatically. Every merge to main deploys. Nothing else to configure.

Method A
CI/CD — triggered by merge
What triggers it
Code merged to the production branch (e.g. main or release) automatically starts a GitHub Actions, CircleCI, or equivalent pipeline.
  1. 1Confirm the production branch has the full phase merged in — all Done slice branches resolved into the base branch
  2. 2If not yet merged: merge the base branch (development or equivalent) into the production branch per the branching model in tech-context
  3. 3Confirm the CI/CD pipeline triggered — check the system named in tech-context
  4. 4Monitor until the pipeline completes — do not declare success during a run
  5. 5Verify the deployment using the verification method from tech-context
Verify success: CI dashboard shows green AND the environment responds correctly to the verification check defined in tech-context.
Method B
MCP tool — triggered directly
What triggers it
A connected MCP tool — Railway, Vercel, Netlify, or similar — can trigger the deploy without a CLI command or a git merge.
  1. 1Confirm the MCP tool is connected and accessible in the current session
  2. 2Identify the project and target environment using the configuration notes in tech-context
  3. 3Trigger the deploy via the MCP tool and capture the deploy ID or run reference
  4. 4Monitor deployment status until complete — not just submitted
  5. 5Verify the environment is live using the method defined in tech-context
Verify success: Tool reports deployment complete AND the environment URL or health endpoint returns expected response.
Method C
Manual CLI — solo runs the command
What triggers it
Deploy requires running a specific command — a platform CLI, a custom script, or a direct push. The deploy skill surfaces the exact command from tech-context; the solo executes it.
  1. 1Read the exact deploy command from tech-context, including the working directory it must run from
  2. 2Surface it to the solo clearly — the full command, the directory, any flags or environment requirements
  3. 3Wait for solo to confirm successful execution
  4. 4Log the deploy with timestamp and solo confirmation
  5. 5Verify the environment is live — confirmation from the solo alone is not enough
Verify success: Solo confirms command completed AND the environment verification check defined in tech-context passes.
Method D
Not yet defined — define then deploy
What triggers it
Tech-context exists but has no deploy configuration. This isn't an error — it means the solo hasn't set up the deploy definition yet. The skill asks exactly what it needs.
  1. 1Ask: "Where does this live when deployed — what platform or environment?"
  2. 2Ask: "Is deploy automated (CI/CD on git events) or manual (you run a command or click something)?"
  3. 3Ask: "What branch or action triggers production deploy?"
  4. 4Ask: "How do you verify the deploy succeeded?"
  5. 5Write the answers into the Deployment section of docs/tech-context.md, then proceed via the appropriate method above
Note: Once defined, the deploy section in tech-context serves all future deploys — Method D should only run once per project.

Pre-deploy checklist
What deploy confirms before anything executes — hard stops are hard stops

The deploy skill reads and confirms four conditions before triggering anything. These are not suggestions. A failure on any of them stops the deploy entirely — the deploy skill surfaces what's missing, where to find it, and what to do before continuing.

  • Phase test gate is OPEN — hard stop if not Reads docs/backlog.md and finds the phase completion record for the phase being deployed. Confirms status shows the gate as OPEN and a phase test report is referenced. If the gate is HOLD or missing, deploy stops: "Phase test must pass before deploy. Run /phase-test to complete testing." There is no bypass.
  • All phase branches merged Every slice that reached Done status in this phase must have its feature branch merged to the base branch. The deploy skill checks for any open feature branches from this phase and surfaces them by name if found: "SL-007 feature branch hasn't been merged yet. Merge it to [base branch] before proceeding." Partial phases cannot ship.
  • Environment variables set in the target environment Cross-references environment variables documented in tech-context against what's confirmed present in the deploy target. Variables that exist locally in .env but haven't been set in the target environment (Railway project, Vercel deployment, server config) will cause a runtime failure the first time they're accessed. Surface any gaps before deploying.
  • No open missing requirements in the backlog Checks the backlog for any items marked as Missing Requirement that haven't been resolved or formally deferred. A missing requirement that was discovered during QA and left unresolved is a gap in what was committed to. If one is open: confirm it was deliberately deferred (and the backlog reflects that decision) before proceeding.
🔍
Verification — environment actually live
Pipeline green and environment running are two different facts — deploy confirms both

The deploy skill does not declare success when the pipeline completes. A pipeline can succeed — every build step, every test, every artifact — and still result in an environment that isn't serving traffic correctly. Misconfigured environment variables, a failed container start, a missing database migration, a routing misconfiguration: all of these produce a passing pipeline and a broken environment.

Deploy is not done until the environment is confirmed live. What "live" means is defined in tech-context — but the deploy skill always verifies it, regardless of method.

Verification sequence
After deploy completes
1
Pipeline or tool shows complete
CI dashboard, Railway deploy log, Netlify deploy summary — whatever the system is — shows the deploy process finished without errors. This is a necessary condition, not a sufficient one.
2
Environment URL is accessible
The deployed environment responds to an HTTP request. For a web app: the root URL loads. For an API: a known endpoint returns a response. For a module in a platform (e.g. Bayer Aurora): the module registers and appears in the expected location.
3
Health check or version confirmation
If the project exposes a /health or /version endpoint, hit it. The response should match what was built — right version string, right environment identifier. If no endpoint exists: confirm a visible element on the first screen reflects the work that was deployed in this phase.
4
Failure path — environment issue, not code issue
If verification fails after a successful deploy pipeline, this is an environment-specific failure — not a reason to reopen phase test or QA. The code was confirmed correct. Route separately: diagnose the environment configuration, fix the specific failure, re-verify. Do not re-run QA on code that already passed.
⚠️
"Green pipeline" and "working in production" are not the same thing. The pipeline validates the build process. Verification validates the running environment. Both must pass. Always check both.
📋
The phase completion record
After successful deploy, the backlog is updated — the phase lifecycle is documented in one place

The phase completion record in docs/backlog.md captures the full lifecycle of a phase in one place. It was created by phase-test when the gate opened — it already shows what was delivered, what comes next, and the test report reference. Deploy adds the final layer: how it shipped, where it went, and how it was verified.

After deploy is confirmed, the status changes from Tested and ready to deploy to Deployed. The record now reads: Complete → Tested → Deployed. This is the authoritative record of the phase.

The completed phase record becomes the reference point for the next phase. When Phase 2 starts, it reads the completion record to confirm what was shipped, what was deferred, and where to pick up. Nothing is held in session memory — it's all in the backlog.

What deploy does not do
Scope boundaries — what belongs to other skills, phases, or tech-context
Deploy does not do these things
  • Define the deploy method. That's tech-context. If it isn't defined there, deploy asks questions and writes it in — it does not invent a method on its own.
  • Run tests. That's phase-test. The gate being open means testing is done. Deploy does not re-run QA, re-run phase test, or validate the code itself.
  • Decide what to ship. That's the entire framework that came before — planning, build, QA, phase test. Deploy ships what the framework already decided to ship.
  • Merge feature branches. That happened at solo-qa when slices were marked Done. Deploy confirms the merges are complete, but does not do them.
  • Proceed with the gate closed. No path forward if phase test gate is not OPEN. No manual override, no "soft launch," no "we'll run phase test after."
Deploy does do these things
  • Reads tech-context every time. Never assumes a platform. The deploy method could be anything — it routes from what tech-context says, not from what the last project used.
  • Confirms the gate hard. Checks the phase completion record in the backlog, not just a verbal confirmation. Gate must show OPEN in the document.
  • Verifies the environment. Active check — hits an endpoint, confirms a response, validates the deployed state. Not passive pipeline monitoring.
  • Writes the completion record. Updates docs/backlog.md with deploy date, method, environment URL, and verification result when successful.
  • Routes environment failures separately. If the environment fails post-pipeline, it diagnoses the environment — not the code. Does not reopen QA.
⚠️
What to watch for
Anti-patterns that lead to bad deploys — each one is specific, each one has a cost
Anti-pattern What goes wrong What to do instead
Deploying without phase test OPEN Untested code ships. If something breaks in production, there's no phase test baseline to compare against. The gate exists to make "it worked before deploy, it broke after deploy" a diagnosable statement — not a guess. Hard stop. Run /phase-test. Get the gate to OPEN. Then deploy.
Skipping branch merge verification A slice that's marked Done in the backlog but whose branch wasn't merged ships a partial phase — or ships nothing, if the pipeline builds from a branch that doesn't have the work. The backlog says it's done. Production says otherwise. Check explicitly. For every Done slice in the phase, confirm the feature branch was merged to the base branch. Surface any that weren't before triggering the deploy.
Declaring success when the pipeline went green but the environment wasn't verified The deploy looks done. It isn't. Misconfigured env vars, a failed migration, a routing issue — any of these produces a green pipeline and a broken product. Users (or the solo, the next morning) discover the failure instead of the deploy skill. Always verify the environment actively. Hit an endpoint. Check a health route. Confirm the deployed version. Pipeline + environment check = done. Pipeline alone = not done.
Deploying with other changes bundled in If the production branch has changes beyond what was in this phase — a hotfix merged during the phase, a dependency bump, exploratory commits — the deploy ships more than what was tested. The phase test covered a specific state of the codebase. Bundled changes aren't covered. Deploy only what was tested. The base branch should contain exactly what phase test validated. Extra changes should go through their own process — bug-fix or enhancement skills — before or after this deploy, not bundled into it.
Reopening QA when deploy verification fails Phase test confirmed the code is correct. A failed deploy verification doesn't invalidate that. Reopening QA wastes time re-testing code that was already confirmed to work — and it muddies the record for what phase test actually covers. Route the environment failure separately. Diagnose what's wrong in the environment — missing env var, wrong port, failed start command. Fix the environment. Re-verify. The code doesn't need to be retested.
🏁
When deploy is done, the phase lifecycle is complete. The backlog shows Deployed. The completion record has a date, a method, an environment, and a verification result. The work exists in production. Phase N is closed — Phase N+1 reads what was delivered and continues from there.
