What this phase is
The visual contract and decision surface — not a design deliverable

The design sprint is not a design deliverable. It is not a Figma translation exercise. It is a short, timeboxed collaborative session that produces an HTML artifact good enough to point at — and a conversation that surfaces what's actually in scope for Phase 1 versus what gets deferred.

This is the visual contract — the first thing everyone can point at and agree on — and a decision surface. Before this phase, your project exists as text: a discovery brief, process maps, intentions. After this phase, it exists as something everyone can see and react to. Engineers, stakeholders, PMs, and data people all respond differently to a real screen than they do to a list of requirements. Requirements get clarified because someone looks at a mockup and asks "where does this number come from?" Scope gets real because the question isn't "what are we building?" — it's "is this button Phase 1 or not?"

The sprint surfaces two categories of work that couldn't be seen before:

Visible scope
  • Every element on screen becomes a build/defer question
  • Buttons, filters, toggles, tabs — all become decision gates in the walk-through
  • Cannot be asked against wireframes or words — only against something people can see
  • Decisions organized by screen so they're traceable in planning
Implied infrastructure
  • Backend logic, data models, APIs that must exist before the UI connects to anything real
  • A simple-looking toggle may imply a historical data pipeline
  • A filter chip may imply a query that doesn't exist yet
  • The design doesn't show these — but it's what surfaces the question
The HTML artifact
  • Not a Figma to be translated into code later
  • The HTML artifact becomes the first frontend output
  • Collapses two steps into one — design and first code are the same thing
  • Serves as shared language for every phase that follows
Core principle
The deferred decisions log is the real deliverable. The screens are the tool that makes the conversation possible. And prd-to-plan cannot sequence phases correctly without it — because phases are built around what's approved and what's implied, not around what's technically comfortable to build first.

Skills in this phase
One phase skill, two supporting skills, one always-on cross-reference
design-sprint
Phase skill
Produces a visual HTML artifact for every screen before any code is written. Bridges the discovery brief to the build phase by creating something everyone can point at and make real decisions from. Runs as a collaborative warmer/colder loop — not a one-shot deliverable.
What it does
  • Reads discovery brief and to-be process maps before producing anything
  • Asks one aesthetic question to route skin direction before any HTML is written
  • Identifies the hero screen — the screen where the moment of value is most visible
  • Runs the warmer/colder loop on structure first, skin second
  • Extends from hero to all screens once hero is approved
Key disciplines
  • Three on-ramps: scratch, Figma import, reference-based
  • Warmer/colder loop: structure approved before skin touched
  • Deferred decisions log: both visible and implied infrastructure
  • Walk-through conversation: mandatory — it's the decision surface
  • Max three rounds per screen once structure is warm
Outputs
  • docs/design/sprint-[id].html — one per screen
  • docs/design/deferred-decisions.md — the gate artifact
  • To-be map annotated with screen references
  • Optional: stakeholder handoff package in docs/design/handoff/
frontend-design
Supporting skill
Aesthetic guidance for HTML artifact production. Not a phase skill — it is activated by design-sprint when the situation calls for it, particularly when skin direction is unclear or when the solo wants a strong aesthetic starting point rather than a utilitarian one.
When it activates
  • Solo asks for a specific aesthetic direction during skin pass
  • After three cold rounds on skin — indicates design system needs replacement, not incremental adjustment
  • When the build calls for distinctive visual quality beyond functional layout
What it provides
  • Production-grade aesthetic decisions — typography, spacing, color systems
  • Component patterns that can carry through all sprint screens
  • Design system consistency so secondary screens don't drift from the hero
Important
  • Does not replace the warmer/colder loop — it informs the skin pass
  • Design system established on the hero screen, inherited everywhere else
  • The solo's "warmer or colder" judgment is still the final word
awesome-design-md
Supporting skill
Reference-based aesthetic extraction. The skill behind On-ramp 3 — when the solo says "make it feel like ESPN" or shares a screenshot, this skill extracts the right design tokens and aesthetic constraints to drive the HTML production in that direction.
When it's used
  • On-ramp 3 — reference exists (product, app, screenshot, URL)
  • After the aesthetic question in Step 2 produces a named reference
  • When "make a call" isn't enough and a specific direction was given
What it extracts
  • Color palette and usage patterns from the reference
  • Typography choices — weights, sizing, hierarchy
  • Spacing and density signals — compact vs airy
  • Interaction and component patterns to carry forward
Output
  • Aesthetic constraints that feed directly into the HTML first pass
  • A DESIGN.md reference grounding subsequent iterations
  • Reduces warmer/colder rounds by starting closer to the right direction
process-mapper
Always-on
The always-on background skill that maintains process maps throughout the framework. During the design sprint, it cross-references every screen produced against the agreed to-be process map — and flags any gaps where a process step has no corresponding screen.

You don't invoke process-mapper. It runs continuously in guided and piloted modes. Its role during the design sprint is coverage enforcement: as each screen gets approved, it annotates the to-be map with the screen reference. Steps with no screen yet get surfaced as decisions.

Example gap surfacing
"Step 4 in the to-be map — 'system validates eligibility' — doesn't have a screen yet. Is this a background step the user never sees, or does it need an error/confirmation state on screen?"

These gaps are documented in the deferred decisions log, not silently assumed. Every to-be step either has a screen reference or a documented decision about why it doesn't need one.

The three on-ramps
All three end in the same place — HTML screens, walk-through, deferred decisions log
Routing question
At the end of the discover phase, one question determines the on-ramp: "Do you have a Figma, a reference, or are we starting from scratch?" The on-ramp decision is recorded in the discovery brief under design on-ramp.
On-ramp 1
From scratch
No visual exists. The design sprint opens with the discovery brief and produces an HTML artifact through a warmer/colder loop. The aesthetic question comes first — the solo either names a direction or the AI makes a call they'll react to.

Path: Aesthetic question → hero screen first pass → structure warm → skin direction question → skin pass → approved hero → extend to all screens.

This is the hardest path and the most common one. Expect more warmer/colder rounds when starting from a rough idea rather than a detailed brief.
On-ramp 2
Figma exists
Import via Figma MCP. The design sprint uses the Figma as the visual source of truth. The discovery brief provides the story layer — the "why" behind each screen. The warmer/colder loop confirms the Figma reflects the story correctly.

Path: Figma read via MCP → map screens to to-be map → confirm story alignment → convert to HTML → walk-through conversation.

Moves faster because skin direction is already established. Sprint rounds focus on story alignment and scope decisions, not aesthetic choices.
On-ramp 3
Reference exists
"Make it feel like ESPN" or "reference this screenshot." The awesome-design-md skill extracts the aesthetic constraints and the sprint produces HTML in that direction. Reference becomes the design system foundation.

Path: awesome-design-md extracts tokens → hero screen built from constraints → warmer/colder confirms alignment with reference → extend to all screens.

Reduces early warmer/colder rounds significantly — starting closer to the right aesthetic direction means fewer "rebuild the design system" moments.

All three on-ramps converge at the same execution flow once aesthetic direction is confirmed: hero screen first → warmer/colder until structure and skin are both warm → extend to all screens → walk-through conversation → deferred decisions log. The on-ramp affects how the first pass is generated, not the discipline applied to the loop that follows.

Step-by-step experience
The full sprint arc — from reading the brief to locking the deferred decisions log
1
Read the discovery brief and process maps
Required gate — nothing is produced until these are read

Before any HTML is written, two documents are required: docs/discovery-brief.md and docs/process/to-be-[name].md. If either is missing, the sprint stops and the appropriate upstream phase runs first.

The brief establishes the story — key moments, moment of value, personas, what's explicitly out of scope. The to-be map establishes the process — what the user does, in what order, across what steps. Both are active references throughout the sprint, not one-time reads.

process-mapper begins cross-referencing from this step forward. As each screen is produced, it annotates the to-be map with the screen reference. Steps with no screen get surfaced as decisions rather than silently assumed.

2
Ask the aesthetic question
On-ramp 1 and 3 only — routes skin direction before any HTML is written

Before writing any HTML, one question is asked:

"Is there a product, app, or aesthetic direction you want this to feel like — or should I make a call and you react to it?"
  • If they name something (ESPN, Linear, a screenshot): note it. That's the aesthetic constraint. Route to awesome-design-md if extraction is needed.
  • If they say "make a call": produce a reasonable starting point and state what direction was chosen. They'll correct it in the warmer/colder loop.
  • If they say "Figma exists": route to on-ramp 2 — no aesthetic question needed.

Do not skip this question. Establishing skin direction before the first pass saves multiple full rebuild rounds downstream.

3
Identify the hero screen
The screen where the system's core value is most visible

From the key moments in the discovery brief, one screen is identified: the moment of value from Zone 3 of the discovery conversation. This is the screen where the system's purpose is clearest — where a user sees the thing they came for.

The choice is stated explicitly: "The hero screen is [X] because [one sentence on why it shows the moment of value]."

Everything extends from this screen. Getting the hero right — structure and skin — before moving to secondary screens keeps the sprint focused and prevents aesthetic drift. Secondary screens inherit the design system the hero establishes. If the hero's design system is wrong, every other screen is also wrong.

4
Build the first pass — structure only, no skin
Wireframe-first: monochrome, system fonts, layout only — no color or typography decisions until structure is confirmed warm

Round one is structure only. The constraint is absolute: white background, dark text, gray borders, system fonts. No color decisions. No typography decisions. No skin of any kind. This is not "skin intentionally light" — it is no skin at all until the solo confirms structure is correct.

This rule applies to on-ramp 1 and on-ramp 3 only. On-ramp 2 (Figma source) has a defined visual system — skin discipline applies but the wireframe-first constraint does not.

  • White background, dark text, gray borders — no palette decisions
  • System fonts only — no typography decisions
  • Intentionally rough is fine. The question is: does this layout correctly represent what the person does here?
  • Use placeholder data that feels real — real-looking names, realistic numbers, plausible states
  • Include all interactive elements that are visible — every button, filter, toggle, tab becomes a gate question in the walk-through
  • Save to docs/design/sprint-[screen-id].html and serve so it opens in a browser

Element naming pass — runs immediately after producing the HTML: walk every interactive element in order and name the component type and expected behavior. Example: "Filter bar — clicking a chip applies it as an active filter. Search input — typing triggers results on submit, not keystroke. Export button — downloads the current view as CSV." The solo corrects anything misidentified before layout feedback begins. This pass prevents silent misclassification of interactive elements as decorative.

After the naming pass: "Here's the first pass. I built it this direction: [one sentence on structure decisions]. Warmer or colder on the layout before we touch the skin?"

5
The warmer / colder loop
Structure and skin are separate questions — never mix them in the same round

The warmer/colder loop is the core discipline of the sprint. It is not feedback gathering — it is a series of rounds that converge on something good enough to make real decisions from. The solo's judgment is the most important input.

Round discipline Max 3 rounds once structure is warm
Each round: (1) Solo gives warmer/colder on what's in front of them. (2) AI adjusts and states what changed and why. (3) Repeat.

Structure and skin are two separate passes. Structure warm → then skin direction question → then skin pass. Never adjust both in the same round.

Expected rounds by starting condition:

Starting condition Expected rounds on hero screen
Detailed discovery brief + wireframes 2–3 rounds
Strong discovery brief, no wireframes 3–5 rounds
Rough idea, early discovery 5+ rounds — may need to pause and deepen discovery first

Special cases in the loop:

  • Same element cold twice: Surface the underlying question — "You've been cold on the delta chart twice — is it the format, the data shown, or something else?"
  • Warmer on structure, cold on skin: Do a full skin pass. Ask one question before rebuilding: "What should this feel like?" Then rebuild the full design system — do not incrementally adjust colors.
  • Structure not warm after 3 rounds: Stop the design sprint. Go back to discover and deepen Zone 2 (the journey) — structure is unclear because the story isn't clear yet.
  • "Good enough, move on": Respect it. Note what's thin and move to the next screen.
6
Extend to all screens
Hero approved first — everything inherits from it

Once the hero screen is approved on both structure and skin, extend to remaining screens in this order:

  • The screens immediately before and after the hero in the user journey
  • Any screen explicitly called out in the discovery brief key moments
  • Remaining screens in navigation order

Each screen inherits the design system established in the hero. The skin does not change screen to screen — only content and layout patterns. Secondary screens should move significantly faster than the hero because design language is already settled.

For each screen: produce it, state what decisions were made, ask for warmer/colder. Move at pace. The goal is something good enough to point at and make decisions from — not pixel-perfect fidelity on every screen.

Every screen is saved to docs/design/sprint-[screen-id].html. process-mapper updates the to-be map annotation as each screen is approved.

7
The walk-through conversation
The decision-surface conversation — not a review, a series of real build/defer decisions

Once all screens are at acceptable fidelity, the walk-through runs. This is the most important conversation in the sprint. The goal is explicit build/defer decisions screen by screen, and surfacing the behind-the-scenes work the design implies but doesn't show.

For each screen, two questions:

"Is everything visible on this screen in scope for Phase 1, or are we deferring anything?"
"What has to exist behind this screen before it can work — data, APIs, logic — and is any of that a Phase 1 dependency or something we're not building yet?"

The AI proactively flags both categories:

  • Visible elements that imply multiple data sources or complex queries
  • UI controls that look simple but have significant backend implications
  • Features that require infrastructure not currently in scope
  • Anything in the "What We're Not Building Yet" section of the discovery brief

A good example of surfacing implied infrastructure: "The temporal lens chips — 2Y, 1Y, Season, L6, L3, Playoffs — are all visible here. Two questions: first, are all six in scope for Phase 1? Second, behind each lens is a historical aggregation query — do we have that data, and is building that query Phase 1 or later?"

Every decision — visible and implied — gets documented. The walk-through is mandatory. Skipping it means planning inherits scope that was never decided.

8
Produce the deferred decisions log
The gate artifact — feeds directly into prd-to-plan

After the walk-through, the deferred decisions log is written to docs/design/deferred-decisions.md. This is one of two required gate artifacts. prd-to-plan reads this log to sequence phases around what's approved and what's implied — not around what's technically comfortable to build first.

Structure of the log:

  • Organized by screen — every visible element gets a decision: Phase 1 or deferred
  • Deferred items include a trigger to revisit (what condition moves them back in)
  • Implied infrastructure documented alongside visible scope — same format, same screen grouping
  • Open questions section for anything that came up but couldn't be decided — with what each blocks

Treating this log as optional is the most common planning failure in the framework. Without it, phases get bloated with deferred scope that wasn't explicitly decided.

9
Ask about stakeholder handoff
One question — if no, skip. If yes, produce the package.

Before closing the sprint, one question: "Does this need to go to anyone outside the build — a stakeholder, client, approver — before design review begins?"

If no: skip. The sprint is complete. Gate artifacts exist. Design review can begin.

If yes: produce the stakeholder handoff package. See the Stakeholder Handoff section below for full details on what's produced.

📤
Stakeholder handoff (optional)
Step 9 output — only produced when external review is needed before design review begins

When the sprint work needs to go to someone outside the build — a client, stakeholder, or approver — before design review begins, a handoff package is produced at docs/design/handoff/<YYYY-MM-DD>/.

artifact.html
  • Standalone — all screens inlined into one file
  • Opens without a server — shareable as a single file
  • Every screen produced in the sprint included
  • No dependencies — no CDN, no external assets
artifact.pdf
  • PDF render of the HTML — for stakeholders who prefer PDF over a link
  • Generated via headless Chrome or equivalent
  • Useful for formal approval workflows or email-based review
feedback-template.md
  • Structured markdown the stakeholder fills in
  • Per-screen comment slots + overall reaction
  • Concerns/blockers section + approval status checkboxes
  • Returned as docs/stakeholder-feedback/<date>-<topic>.md
How feedback flows back
The solo shares whichever format the stakeholder prefers. Completed feedback returns as a file in docs/stakeholder-feedback/. The design-review skill reads any file in that directory as first-class input on the next round — per-screen comments become findings, and blockers gate the Ready status.

The gate
Design review cannot start without both artifacts — structural, not ceremonial
Gate: Design Sprint → Design Review
✓
sprint-[id].html exists — at least the hero screen is approved and saved. All other screens expected, but the gate checks for existence, not perfection.
✓
deferred-decisions.md exists — the walk-through ran and every visible element on every screen has a Phase 1 or deferred decision. Implied infrastructure is documented. Open questions are logged.
✓
to-be map is annotated — every step in the to-be map either has a screen reference or a documented decision for why no screen covers it. No silent assumptions.
Gate enforcement
framework-health checks for gate artifact existence on each session start. If deferred-decisions.md is missing and the project state suggests the sprint has run, it surfaces one recovery path: complete the walk-through and produce the log before continuing to design review.

Design Review reads the sprint output in two modes:

Round 1 is a full first pass across all screens — it defines the initial set of slices, identifies which slices can go straight to Ready and which need spikes first, and sets the review log. The deferred decisions log from the sprint is the primary input for determining what belongs in Phase 1 scope vs later.

Round N (subsequent rounds) refines slices, promotes them to Ready status, and updates the backlog. Design review continues in parallel with build for remaining slices — it does not have to complete before the first build work begins.

What to watch for
The most common ways this phase goes sideways — and why each one matters
Anti-pattern Why it happens The actual problem Instead
Jumping to skin before structure is warm Structure looks close enough — the impulse is to make it look good Structure and skin require separate warmer/colder questions. If skin is addressed while structure is cold, you're rebuilding both at the same time with neither converging. Always ask structure-only first. Get an explicit "warmer on layout" before asking about skin direction.
Building all screens before hero is approved Moving fast feels productive — ship all screens at once Every secondary screen inherits from the hero. If the hero's design system is wrong, all screens have to be rebuilt. The drift compounds on every screen. Hero screen approved on both structure and skin before the first secondary screen is produced. No exceptions.
Skipping the walk-through Screens look good, there's momentum — the walk-through feels like a slowdown Without the walk-through, no build/defer decisions are made. Planning inherits the entire visible scope as Phase 1 — plus implied infrastructure that was never surfaced. Phases get bloated. The walk-through is mandatory. It is the gate, not a nice-to-have. Screens are the tool that makes the conversation possible.
Treating the deferred log as optional "We talked through scope in the walk-through, that's enough" prd-to-plan reads the log to sequence phases. Without it, the planner has no structured input — phases default to technical convenience rather than product risk. The deferred items drift back into scope. Every visible element gets a decision in the log: Phase 1 or deferred. Every implied infrastructure item gets the same. No silent assumptions.
Incrementally adjusting colors when skin is cold One more tweak feels like the faster path Incremental color adjustments produce neither version correctly. You end up with a hybrid that doesn't commit to any direction. After two cold rounds on skin, the design system needs to be replaced, not tweaked. When skin is cold, ask one question: "What should this feel like?" Then rebuild from that direction. Change the design system, don't adjust it.
Aiming for pixel-perfect fidelity on every screen Perfectionism — wanting every screen to be presentable before moving on Wrong level of investment. The goal is something good enough to point at and make decisions from. Secondary screens don't need the same fidelity as the hero. Sprint time is better spent on the walk-through and deferred log than on visual polish. Good enough to point at and make real decisions from — not perfect. "Good enough, move on" is an acceptable signal. Note what's thin and continue.
Continuing a stuck sprint instead of going back to discover Sunk cost — the sprint has started, stopping feels like going backward If structure is still cold after 3 rounds, the story isn't clear enough. The design sprint cannot produce a good artifact from an unclear brief. More rounds won't fix it. After 3 cold rounds on structure: stop. Surface what's unclear. Go back to discover and deepen Zone 2 (the journey). Return to the sprint when the story is clearer.
Treating parallel secondary screens as a race condition In Claude Code, secondary screens fan out simultaneously after hero approval — multiple screens appear at once instead of one at a time This is intentional behavior, not a problem. The design system established on the hero applies to all screens in the same pass. Trying to intervene or sequence them manually defeats the parallelism and can produce inconsistent results if the design system hasn't fully propagated. Let the fan-out complete. Review all screens together as a set. If something looks wrong, address it in the next warmer/colder round on the affected screen — not mid-production.
Expecting a prompt before the preview pass runs in design review Step 1.5 is automatic in Claude Code — it fires between the state read and the specialist review without any activation. If you wait for a confirmation prompt before the preview runs, you'll miss findings that were already surfaced. The preview pass outputs its findings and flows directly into the specialist synthesis — it is not a separate step that requires a go signal. In Claude Code, let Step 1.5 run. If a project is non-visual or no HTML design artifacts exist, it skips silently — no intervention needed either way. In Cursor, the preview pass does not fire; the specialist review starts immediately.
Missing security process gaps in the process-mapper cross-reference The process-mapper reads the to-be map and annotates screens — but security gaps require reading the Security Classification from tech-context.md first. If that step is skipped or tech-context.md has no Security Classification section, the check runs as Standard and class-specific gaps go undetected. Security process gaps are process design failures, not code failures. A Personal-class project missing a data deletion path won't fail code review — it will ship without one. The design sprint is the only moment where these gaps can be caught before screens are approved and slices are written. In Step 1, confirm the Security Classification section exists in docs/tech-context.md before the process-mapper cross-reference runs. If it's missing and the project handles personal data, financial records, user credentials, or multi-tenant data — stop and add it first. The class takes two minutes to declare and changes what the mapper looks for.
