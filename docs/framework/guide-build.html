Solo Builder Framework — Phase Guide

Phase 4: Build

Two modes: guided (solo present at every gate) or autonomous (simulator at every gate, solo reviews at phase test). Every slice is anchored before it starts. Code-complete is not Done — it hands off to QA automatically.

Overview

What this phase is

Build executes the backlog, slice by slice, in the order a user actually travels through the product.

Every build session starts with a read, not a build. Before selecting any slice, the build skill reads the current state of six documents: the backlog (status and priority), the discovery brief (the agreed problem), the to-be process map (the agreed process), the tech context (stack and conventions), the data mapping (field sources), and the design sprint artifact (the visual contract). This is not optional and is not replaced by conversation history from a prior session. The first output is a context-loaded confirmation — what's in the backlog, what's Ready, what's next — before any slice work begins.

Build does not decide scope — that was settled in Design Review. It does not design — that was settled in the Design Sprint. Build executes what was decided, correctly, in the right order, with the right anchors confirmed before a single line is written.

The tracer bullet goes first: the thinnest path through the full user journey, end to end, before any individual screen is expanded. Once the core loop is proven to work, each step gets built out in full.

Before anchors are even checked, the slice must be in Ready status. Any other status — In Review, In QA, Deferred — means the slice does not enter build. No discussion, no partial starts, no "we can work on the design while we build." Ready is the gate before the gate. If there are no Ready slices, build reaches Build Active, no Ready slices status and routes to whatever skill will unblock them — it does not find workarounds.

Once status is confirmed, every slice requires four anchors confirmed before code starts: a design anchor, a data anchor, a done anchor, and a process anchor. Any missing anchor means the slice doesn't start. This is the gate, not a checklist.

When something surfaces mid-build — a design conflict, a missing data field, a dependency that wasn't fully resolved — it comes up immediately. It is never silently absorbed into the implementation and never resolved by quietly simplifying the design without a decision being made.

When a slice reaches code-complete, solo-build declares it and hands off to code-review-and-quality automatically. That skill chains to solo-qa on pass. Build does not declare Done. QA does.

Before starting any slice — the selection statement

✓
Slice is Ready in the backlog
✓
All dependencies are Done
✓
Next in journey order (or explicitly parallel)
✓
All four anchors confirmed

State it out loud before starting: "Building SL-[ID] — [Name]. It's next in journey order, dependencies are clear, and it has all four anchors."

The skill reads the plan and states the next slice — it does not ask. If the solo wants to divert from priority order, they say so. The skill confirms the diversion explicitly before proceeding.

Skills

Skills in this phase

Build has two mode skills — guided (solo-build) or autonomous (autopilot) — plus on-demand skills invoked at specific moments.

solo-build
Phase skill
The execution layer of the Build phase. Selects the right slice based on journey order and dependency state, confirms all four anchors before writing code, tracks discoveries, and hands off to code-review-and-quality when code-complete is declared. This skill runs the entire build phase — not a support tool, the primary driver.
Slice selection
  • Tracer bullet rule — thinnest path first
  • Journey order — first screen before second
  • Dependencies block — hard gate, not suggestion
  • Parallel exceptions named explicitly
During build
  • Build against the design screen directly
  • Pull data from mock layer — never hardcode
  • Surface conflicts immediately, don't absorb
  • Flag cross-slice impacts in the backlog
On code-complete
  • If CI/CD configured: generate tests/test_SL-[ID].py from done criteria, run full suite — green proceeds, red regression blocks commit
  • Commit: SL-[ID] code-complete — [name] (include test file if generated)
  • Update backlog: status → In QA, write test file path
  • Auto-invoke code-review-and-quality
  • Do not wait for solo to trigger the handoff
autopilot
Phase skill — autonomous mode
Alternative to solo-build. Builds the full product without solo participation after discovery or design sprint. Solo-simulator plays every gate. First human touch point is phase test. Refinement cycles handle delta iteration after phase test.
When to choose autopilot vs. solo-build
  • solo-build — solo wants to participate at every gate, catch drift in real time, and shape output slice by slice
  • autopilot — solo wants to reach a finished product fast and iterate on the whole rather than the slice. Accepts that first-pass output is spec-compliant, not preference-validated — Refinement closes that gap.
Entry points
  • Post-discover — leaner design anchor, more simulator judgment calls, higher Refinement likelihood
  • Post-design-review — richer anchor, fewer judgment calls, tighter first-pass output. Recommended for UI-heavy products.
Pre-flight check
  • Simulator reads all discovery docs before build begins
  • Surfaces judgment gaps — places where the spec doesn't decide
  • Solo answers each gap. Brief is locked. Build begins only when gate is clean.
  • Discovery quality check: thin to-be map flagged before gap questions
Refinement cycles
  • After phase test, issues classified: implementation (Refinement handles) or structural (guided re-entry required)
  • Delta build only — changed slices, not full product rebuild
  • Exit ramp to guided mode surfaced at every Refinement cycle completion
tdd
Technique skill
Test-driven development with a red-green-refactor loop. Invoked from solo-build when test-first development is the right approach for a slice. Tests verify behavior through public interfaces, not implementation details — so they survive refactors. One test, one implementation, one cycle at a time.
The loop
  • Red — write one failing test for one behavior
  • Green — minimal code to pass it
  • Refactor — clean up, tests must still pass
  • Never refactor while red
Good tests
  • Test behavior, not implementation
  • Use public interfaces only
  • Survive internal refactors
  • Read like a specification
Anti-patterns
  • Writing all tests first — horizontal slicing
  • Mocking internal collaborators
  • Testing private methods
  • Speculative tests for future behavior
principal-engineer
On demand
Architecture review lens for consequential decisions. Not always-on. Not a code reviewer. Not a design reviewer. Called when a mid-build discovery is structural — when the approach itself needs to change, not just the implementation. The PE makes one recommendation, explains the reasoning, names the tradeoff, and holds the position under preference pushback.
When to invoke
  • Mid-build discovery is architectural
  • Decision is hard to reverse
  • Blast radius if wrong is large
  • Solo feels uncertain about an approach
The four questions
  • Does this fit the platform?
  • What's the blast radius if wrong?
  • What debt does this create?
  • Is this built to expand to Phase 2?
Output format
  • One recommendation — not a menu
  • Reasoning grounded in tech-context
  • Named risks and named debt
  • Watch-for signals logged in backlog
Pre-build gate

The four anchors

Before any slice starts, all four anchors must be confirmed. If any anchor is missing, the slice doesn't start. This is the gate, not a checklist to be completed during build.

1
Design Anchor

The exact screen and specific element this slice builds. Not just the file — the exact component within it, and where it lives on the screen. Open the file. Look at the element. The implementation target must be unambiguous before a line is written.

Format
[screen-file] → [element name] → [location on screen]
Example: sprint-p1.html → Slot context card → top of overview body, below player tabs

When the design source is a Figma file: before writing any code, extract the exact node properties (spacing, color, typography) for every element in scope from Figma Dev Mode or MCP — not visual estimates. Then verify the interactive element inventory in the Notes field: every element classified Functional must have its logic wired; every element classified Deferred must render as an explicit non-interactive placeholder. If the inventory is missing, stop — the slice returns to design review before build continues.

2
Data Anchor

The specific mock data fields this slice consumes and where they live in the mock layer. The mock data layer is the data source during build — never hardcode values that should come from data. The data mapping doc shows where each field eventually comes from in production.

Format
[mock-file] → [field names] → eventual source per data-mapping.md
Example: data/mock/players.json → slot_tag, slot_target → MFL Rosters API
3
Done Anchor

The 2–3 criteria from the backlog that close this slice. These are what get handed to solo-qa. Vague criteria — "it works," "it looks right" — are not criteria. Sharpen them before starting. Every criterion must be specific and verifiable in a browser.

Format
Criteria listed exactly as they appear in the backlog
Example: WR2 slot tag renders from mock data / Slot target displays with correct formatting (14.2 pts) / Toggle is visible but non-functional in Phase 1
4
Process Anchor

The step in the to-be process map this slice implements. Read the to-be map and identify the step. If the slice doesn't map to any step in the agreed process, stop — this slice may be implementing something outside the agreed scope. That requires a decision, not silent build.

Format
[to-be map file] → [step name] → [position in flow]
Example: to-be-player-evaluation.md → Step 3: system displays slot context → main path, after player lookup
⛔

No anchor = slice doesn't start

This is not a formality. Each anchor exists because building without it produces a specific failure: building the wrong thing (design), building against hardcoded data (data), not knowing when done (done), or building outside the agreed process (process). Any missing anchor is a signal that the slice isn't ready — resolve it before building, not during.

Sequencing

Journey order

Build in the order a user travels through the product. First screen before second. Never build a screen that depends on state from an unbuilt screen.

Journey order means the sequence of screens as a real user experiences them — not as they appear in the backlog, not by technical complexity, not by personal preference. The first screen a user lands on gets built before the second. The second before the third.

⚠ Screen 4 is blocked until Screen 3 is Done. State this explicitly: "SL-007 depends on SL-005 which is still In Build. Building SL-005 first."
1

Journey order, not parallel

This is not about sequencing features for efficiency — it's about discovering integration problems while the codebase is still small enough to fix them cheaply. A problem found at Screen 2 costs far less to fix than one discovered after all five screens are built.

2

Dependencies block — hard gate, not suggestion

The backlog dependency field is enforced here. If SL-003 depends on SL-001, SL-001 must be Done before SL-003 can enter In Build. Surface it clearly rather than finding a workaround that creates hidden coupling.

3

Parallel exceptions — named explicitly

Screens with no shared infrastructure and no journey dependencies — settings, auth, help screens — can run alongside the main journey if they're Ready. Name them explicitly as parallel rather than letting them blur into the main sequence. "This is a parallel track, not next in journey order."

4

Dependency types matter

Not all dependencies are the same. Name the type when flagging a block: Slice (another slice must be Done), Infrastructure (service or DB must exist), Data (mock entity must be available), or Design (decision still open). The type determines what resolves the block.

Dependency type What it means What resolves it
Slice Another slice must be Done — shared component or shared state The blocking slice passes QA and reaches Done status
Infrastructure A backend service, API endpoint, or DB table must exist Infrastructure slice is built and verified
Data A specific mock entity or real data source must be available Mock data added to the data layer; data-scaffold updated
Design A design decision is still open — spike or review needed first Design review round closes the open decision; backlog updated
First slice

The tracer bullet

Before building any individual screen in full, build the thinnest path all the way through the user journey. Prove the core loop works end to end — even if every step is minimal.

⚡

Thinnest path. Full journey. Before expanding anything.

The tracer bullet is not a prototype. It's the first real code in the project.

What it is

The minimum that proves the core loop works end to end. Not the minimum that proves one screen works — the minimum that proves the user can travel from the first action to the moment of value, even if every step is minimal. Thin everywhere before deep anywhere.

  • Can the user reach the core interaction?
  • Does navigation between screens work?
  • Does data flow from source through the UI?
  • Does the journey reach the value moment?

Why it goes first

Building one screen to completion before touching the next hides integration problems until late. The tracer bullet forces every screen to exist in minimal form before any screen is expanded — so integration problems surface when the codebase is still small, not after five complete screens need to be rewired.

  • Integration failures surface early, cheaply
  • Core loop validated before building depth
  • Navigation and state flow proven before expansion
  • Something real to hand off to QA from day one
Tracer bullet approach
  • Screen 1 (thin) → Screen 2 (thin) → Screen 3 (thin)
  • Then Screen 1 full → Screen 2 full → Screen 3 full
  • Integration problems surface at Screen 2, not Screen 5
  • End-to-end verified before feature depth is added
Screen-by-screen approach
  • Screen 1 (full) → Screen 2 (full) → Screen 3 (full)
  • Integration tested for the first time at Screen 3
  • Rewrite risk is highest when the codebase is largest
  • Three complete screens may need rewiring when they meet

What "end-to-end" means in this context

End-to-end means from the first action a user takes — opening the app, landing on the first screen, triggering the first interaction — to the moment of value: the screen, result, or state the user came for. For a player evaluation tool, that might mean: user opens app → searches for a player → sees the player's core evaluation. Thin at every step, but all the way through.

During build

Mid-build discoveries

Build reveals things the design review didn't. How a discovery gets handled depends on its nature and scope — but it is always surfaced, never silently absorbed.

Minor Implementation detail

An implementation detail that doesn't affect the design, the agreed process, or any other slice. The kind of thing that comes up during build and has an obvious solution.

Action: Resolve it, note it in the slice detail in the backlog, continue building. No need to surface it or pause.
Design Gap Screen isn't buildable as shown

Something on the design screen can't be built as shown — missing state, an interaction that requires data that doesn't exist in the model, or a visual that implies logic not captured in the design.

Action: Stop. Surface it specifically with options. Return to design-review for a decision. Do not silently simplify the implementation.
Affects other slices Upstream or downstream impact

Building the current slice reveals something that will change how an upstream or downstream slice needs to be built — a shared component, a state assumption, a data model implication.

Action: Flag in the backlog immediately. Note which slices are affected and why. Continue current slice if possible — but make the impact visible before the next slice starts.
Blocks current slice Dependency not resolved

A dependency that was marked as resolved turns out to not be resolved — the service doesn't work as expected, the data isn't available, or a design decision that was supposed to be settled still isn't.

Action: Stop the current slice. Move it back to In Review with a note explaining the block. Pick the next appropriate slice while the blocker is resolved.
🚫
Never silently absorb a discovery into the code. If the implementation diverges from the design — for any reason — that divergence is a decision that needs to be made explicitly. Surfacing it takes one message. Silent absorption creates technical debt that won't be visible until QA, the phase test, or a solo two months later who wonders why the code doesn't match the design.
Handoff

The auto-chain to QA

Code-complete is not Done. It is the start of QA. The handoff from Build to QA is automatic — solo-build triggers it, the solo doesn't manage it.

Exact wording when code-complete is reached
"SL-[ID] is code-complete. Committing and running code-review-and-quality."

What "code-complete" means precisely

Code-complete means
  • ✓ Self-verification checklist run — builder confirmation populated with ✓/✗ results
  • ✓ Slice renders correctly against mock data
  • ✓ Visual matches the design screen
  • ✓ No hardcoded values where data should be
  • ✓ Committed with a traceable commit message
Code-complete does NOT mean
  • ✗ Done — QA hasn't run yet
  • ✗ Verified — AI testing hasn't happened yet
  • ✗ Accepted — solo browser sign-off hasn't happened yet
  • ✗ Deployable — phase test gate not yet open
1

Run the self-verification checklist

Before declaring code-complete, the builder runs the slice's self-verification checklist from the backlog record and populates the builder confirmation field with ✓/✗ results and observations. This is not optional — solo-qa reads the builder confirmation on arrival. An empty confirmation means the builder did not complete their self-check.

2

Commit the work

The build skill executes the commit directly — it is never handed to the solo to run. Commit message format: SL-[ID] code-complete — [slice name]. The slice ID makes every commit traceable in git history. Never use vague messages ("wip", "updates", "fix").

3

Update the backlog

Move the slice status from In Build to In QA. The backlog is a live document — if it's stale, the next session starts from the wrong place.

4

Invoke code-review-and-quality immediately

Do not wait for the solo to trigger it. The handoff is automatic. State code-complete, declare the invocation, and run the review. The solo manages the product — not the QA handoff chain.

Failure modes

What to watch for

The most common ways Build goes wrong. Each one is preventable — they fail because a constraint was relaxed once, not because the constraint is unworkable.

Starting build on a slice that isn't Ready

A slice in "In Review" has open design decisions. Starting build before those close means the code will contradict a decision made later — producing work that needs to be redone. When there are no Ready slices, build reaches Build Active, no Ready slices status — the right move is finishing design review and promoting slices, not finding a way to start build anyway. "We can work on the design while we build" is not an option.

Asking the solo what to work on next

The plan and priority already exist in the backlog. Asking "what would you like to work on?" hands a decision back to the solo that the framework already made. Read the backlog, identify the next Ready slice in priority order, and state it.

Building without all four anchors confirmed

The most common build failure. Starting with three anchors and resolving the fourth "as you go" means the fourth anchor gets resolved under time pressure, often incorrectly. Any missing anchor is a stop signal, not a caution.

Building out of journey order

Jumping ahead to a screen that looks easier, or building in parallel without naming it as parallel, hides integration problems. They surface later — after multiple screens are complete and rewiring is expensive.

Silently absorbing mid-build discoveries

When the implementation diverges from the design and nobody says anything, the design and the code are now different documents. QA checks against the design. The phase test checks against the discovery brief. Silent absorption always surfaces later — as a bug, a failed QA check, or a confused solo reading code that doesn't match the screens.

Skipping the tracer bullet

Going straight to full feature implementation on Screen 1 before touching Screen 2. The tracer bullet's job is to surface integration problems early. Skipping it trades a cheap early fix for an expensive late one.

Hardcoding values that should come from data

Any value that should eventually come from a real data source must come from the mock data layer during build — not from a hardcoded string in the UI. Hardcoded values break the data anchor and make the mock-to-real swap unreliable.

Treating code-complete as Done

Marking a slice Done before code review and solo QA sign-off bypasses the two checks that actually confirm the slice is complete. Code-complete is a state transition, not a destination. Done requires passing both stages of QA.

Vague done criteria accepted at anchor time

"It should look right" and "it should work" are not done criteria. If the done anchor is accepted with vague criteria, QA has nothing concrete to verify against. Sharpen every criterion to something that can be checked in a browser — specific, observable, pass/fail.

Stale backlog during a build session

The backlog is the solo's persistent context across sessions. Every status change is written immediately — before moving to the next action. Not at session end. Ready → In Build before the first line of code. In Build → In QA before triggering code review. In QA → In Test after solo sign-off. In Test → Done when phase-test gate opens. If it's stale at any point mid-session, the next session starts from the wrong place.

Starting a visual slice without re-reading the design file

The design anchor names the file — "design anchor check" means actually opening and reading that file before writing code, not recalling it from memory or a prior read. The required output is a verbatim quote: the exact string from the file, word for word. A paraphrase does not qualify — it's indistinguishable from recall. A design file that has changed since the last session, or one the builder remembers slightly wrong, produces implementation drift that compounds across slices. The verbatim quote makes the read verifiable.

Self-verification that skips quality contract items

Self-verification covers more than the happy path. Before declaring code-complete, the builder must walk every observable quality contract item in the running app: simulate a failure and confirm the error state renders, submit empty or invalid input and confirm validation fires at the right point with the right message, and for any security contract line, confirm user-supplied content is not rendered raw. A self-verification that only confirms the happy path will pass — and then fail at code review Check 8, which checks the same contract line by line with the same adversarial standard.

Self-verification that asserts instead of observes

"Data renders correctly" is a code assertion — it tells you what the code is supposed to do, not what the solo will actually see. Self-verification must be done against the running app. Every checklist item requires a statement of what was observed in the rendered output. The standard is identical to QA: named evidence only. If the builder cannot state what was seen on screen, the check has not been run.

Continuing past two failed attempts without stopping

After two attempts with no progress on the same problem, the builder stops and re-reads the full slice spec and design file from scratch before touching any code. Confidence that the next attempt will work is not a reason to continue — it's the same confidence that produced the two failed attempts. And if the solo asks to step back, that is an immediate directive, not a suggestion to evaluate.

Running past a deliverable boundary without updating the handoff

When a deliverable completes, the handoff should reflect it — what was just accepted, what's next. When a deliverable has more than four slices, the mid-deliverable checkpoint at slice 4 updates the handoff automatically. But context compression can still degrade build coherence in long sessions. The deliverable completion recommendation is a signal worth taking seriously: the work is coherent, the seam is clean, and the next deliverable starts fresher in a new session than it would continuing here.

Treating interactive Figma elements as window dressing

Filters, search boxes, pagination, and toggles visible in a Figma design are not decorative by default. Before build starts, each must be classified: Functional (logic wired in this slice), Deferred (logic in a future slice — companion slice created now), or Out of scope. There is no fourth option. A builder who silently decides an element is decorative during coding has made a product decision without the solo. An element rendered as a working-looking interactive component that accepts input and does nothing is a code review failure — it will fail Check 10c regardless of whether it "looks right" in the browser.

Ignoring a rising rework rate

The framework tracks rework automatically — every time a slice is returned to build after reaching In QA, the counter increments. A high rework rate across multiple slices is a signal the quality gates are catching things too late. The place to look: were the done criteria specific enough? Was the quality contract walked before presenting? A rework rate that keeps climbing without a response means the gates aren't changing the behavior that's causing the rework.

Rebuilding a completed slice without the rollback protocol

When a slice that has passed QA (In Test or Done) needs to be rebuilt, it is a rollback — not a continuation of active build work. The builder proposes the scope (targeted fix or full rebuild), explains why, and waits for solo confirmation before touching any status or discarding any code. Nothing changes before that confirmation. Skipping the protocol means the deliverable and phase records become inconsistent, the log entry is missing, and the solo has no visibility into what changed and why.

Treating code review findings as unexpected failures

Code review now runs as an isolated sub-agent with no access to the build context — it sees only the code, the slice spec, and the quality contract. This is intentional: the reviewer has no knowledge of the reasoning or decisions made during implementation. If code review surfaces something the build pass didn't flag, that's not a malfunction. The isolation removes the bias that comes from knowing why the code was written a certain way. Expect it to be more thorough than a review that shared build context.

Starting build on a core architecture slice without PE review

Any slice classified as Core architecture triggers a principal engineer review automatically before the branch opens — this is not optional and cannot be skipped. If the review didn't happen, the branch should not exist yet. The PE assesses whether the approach is right before a line of code is written. Outcome goes in the Notes field. A core architecture slice that enters build without a recorded PE outcome has bypassed a gate that exists because these changes ripple — catching a structural problem at review is cheap; catching it after three slices are built on top of it is not.

Invoking a comms cascade sub-agent without a complete brief

The comms cascade sub-agent requires a structured brief — version, date, summary, skills affected, watchfor items, and guide sections affected. This brief is the sub-agent's entire specification. A vague brief ("it's a big update, figure it out") produces vague docs: missing watchfor entries, shallow blog copy, skipped guide sections. The curator must provide every field before invoking. If a field genuinely doesn't apply, state that explicitly — "none" is a valid answer. Blank is not.

Using a static ANTHROPIC_API_KEY on a cloud-deployed project

If your project is deployed to a cloud platform (Railway, Render, AWS, GCP, Azure, Fly.io, Cloudflare) and you're currently authenticating to the Claude API with a static ANTHROPIC_API_KEY stored in an env var or secret manager, consider migrating to Workload Identity Federation. WIF uses the workload's existing cloud identity — an IAM role, service account, managed identity, or GitHub Actions OIDC token — to get a short-lived access token on demand. No static secret to store, rotate, or leak. Set it up at platform.claude.com → Settings → Workload identity. The deploy step handles the wiring at deploy time. Local Mac projects are unaffected.

Treating the preview check as a separate step after self-verification

The preview check is part of self-verification — not a step that runs after it. In Claude Code, after the quality contract walk and before declaring code-complete, the framework starts the app server, navigates to the review URL, takes a screenshot, checks console logs, checks for failed network requests, clicks through main interactive elements, and checks the mobile viewport. Console errors or failed network requests directly caused by the slice are ✗ items in builder confirmation and must be fixed before commit. This is not an optional check and it is not a post-commit review — it runs before the commit, inside the existing self-verification flow. Non-visual slices (review URL is None) and Cursor sessions skip silently with no change to behavior.

Check 9 running baseline-only on a project that handles personal, financial, or multi-tenant data

Check 9 reads docs/tech-context.md before it runs. If the Security Classification section is missing, it defaults to Standard — baseline checks only (9a–9e). That means class-specific checks for Personal, Financial, Authentication, Multi-tenant, Confidential, and Regulated never fire, even if the project clearly handles that kind of data. If your project stores PII, processes transactions, manages user credentials, or serves multiple tenants from shared infrastructure, add the Security Classification section to docs/tech-context.md now. The section is short — one to three lines stating the class or classes that apply. Once it's there, code review picks up the right checks automatically on the next slice.
