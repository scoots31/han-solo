What This Phase Is

Every build session starts with a read, not a build. Before selecting any slice, the build skill reads six documents: the backlog, the discovery brief, the to-be process map, the tech context, the data mapping, and the design sprint artifact. This is not optional and is not replaced by conversation history. The first output is a context-loaded confirmation — what's in the backlog, what's Ready, what's next — before any slice work begins.

Build does not decide scope — that was settled in Design Review. It does not design — that was settled in the Design Sprint. Build executes what was decided, correctly, in the right order, with the right anchors confirmed before a single line is written.

The tracer bullet goes first: the thinnest path through the full user journey, end to end, before any individual screen is expanded. Once the core loop is proven to work, each step gets built out in full.

Before anchors are even checked, the slice must be in Ready status. Any other status — In Review, In QA, Deferred — means the slice does not enter build. Ready is the gate before the gate. If there are no Ready slices, build reaches Build Active, no Ready slices status and routes to whatever skill will unblock them.

Once status is confirmed, every slice requires four anchors confirmed before code starts: a design anchor, a data anchor, a done anchor, and a process anchor. Any missing anchor means the slice doesn't start. This is the gate, not a checklist.

When a slice reaches code-complete, solo-build declares it and hands off to code-review-and-quality automatically. That skill chains to solo-qa on pass. Build does not declare Done. QA does.

Before starting any slice, state it out loud: "Building SL-[ID] — [Name]. It's next in journey order, dependencies are clear, and it has all four anchors." The skill reads the plan and states the next slice — it does not ask. If the solo wants to divert from priority order, they say so. The skill confirms the diversion explicitly before proceeding.

Skills in This Phase

solo-build — Phase skill

The execution layer of the Build phase. Selects the right slice based on journey order and dependency state, confirms all four anchors before writing code, tracks discoveries, and hands off to code-review-and-quality when code-complete is declared.

Slice selection
  • Tracer bullet rule — thinnest path first
  • Journey order — first screen before second
  • Dependencies block — hard gate, not suggestion
  • Parallel exceptions named explicitly
During build
  • Build against the design screen directly
  • Pull data from mock layer — never hardcode
  • Surface conflicts immediately, don't absorb
  • Flag cross-slice impacts in the backlog
On code-complete
  • Commit: SL-[ID] code-complete — [name]
  • Update backlog: status → In QA
  • Auto-invoke code-review-and-quality
  • Do not wait for solo to trigger the handoff

autopilot — Phase skill (autonomous mode)

Alternative to solo-build. Builds the full product without solo participation after discovery or design sprint. Solo-simulator plays every gate. First human touch point is phase test. Refinement cycles handle delta iteration after phase test.

solo-build vs. autopilot
  • solo-build — solo wants to participate at every gate, catch drift in real time, and shape output slice by slice
  • autopilot — solo wants to reach a finished product fast and iterate on the whole rather than the slice. Accepts that first-pass output is spec-compliant, not preference-validated.
Entry points
  • Post-discover — leaner design anchor, more simulator judgment calls, higher Refinement likelihood
  • Post-design-review — richer anchor, fewer judgment calls, tighter first-pass output. Recommended for UI-heavy products.
Refinement cycles
  • After phase test, issues classified: implementation (Refinement handles) or structural (guided re-entry required)
  • Delta build only — changed slices, not full product rebuild
  • Exit ramp to guided mode surfaced at every Refinement cycle completion

tdd — Technique skill

Test-driven development with a red-green-refactor loop. Invoked from solo-build when test-first development is the right approach for a slice. Tests verify behavior through public interfaces, not implementation details — so they survive refactors. One test, one implementation, one cycle at a time.

The loop
  • Red — write one failing test for one behavior
  • Green — minimal code to pass it
  • Refactor — clean up, tests must still pass
  • Never refactor while red
Good tests
  • Test behavior, not implementation
  • Use public interfaces only
  • Survive internal refactors
  • Read like a specification
Anti-patterns
  • Writing all tests first — horizontal slicing
  • Mocking internal collaborators
  • Testing private methods
  • Speculative tests for future behavior

principal-engineer — On demand

Architecture review lens for consequential decisions. Not always-on. Not a code reviewer. Called when a mid-build discovery is structural — when the approach itself needs to change, not just the implementation. Makes one recommendation, explains the reasoning, names the tradeoff, and holds the position under preference pushback.

When to invoke
  • Mid-build discovery is architectural
  • Decision is hard to reverse
  • Blast radius if wrong is large
  • Solo feels uncertain about an approach
The four questions
  • Does this fit the platform?
  • What's the blast radius if wrong?
  • What debt does this create?
  • Is this built to expand to Phase 2?
Output format
  • One recommendation — not a menu
  • Reasoning grounded in tech-context
  • Named risks and named debt
  • Watch-for signals logged in backlog

The Four Anchors

Before any slice starts, all four anchors must be confirmed. If any anchor is missing, the slice doesn't start. This is the gate, not a checklist to be completed during build.

1 — Design Anchor

The exact screen and specific element this slice builds. Not just the file — the exact component within it, and where it lives on the screen. Open the file. Look at the element. The implementation target must be unambiguous before a line is written.

Format: [screen-file] → [element name] → [location on screen]

2 — Data Anchor

The specific mock data fields this slice consumes and where they live in the mock layer. The mock data layer is the data source during build — never hardcode values that should come from data.

Format: [mock-file] → [field names] → eventual source per data-mapping.md

3 — Done Anchor

The 2–3 criteria from the backlog that close this slice. These are what get handed to solo-qa. Vague criteria — "it works," "it looks right" — are not criteria. Every criterion must be specific and verifiable in a browser.

Format: Criteria listed exactly as they appear in the backlog

4 — Process Anchor

The step in the to-be process map this slice implements. If the slice doesn't map to any step in the agreed process, stop — this may be implementing something outside the agreed scope. That requires a decision, not silent build.

Format: [to-be map file] → [step name] → [position in flow]

No anchor = slice doesn't start. Each anchor exists because building without it produces a specific failure: building the wrong thing (design), building against hardcoded data (data), not knowing when done (done), or building outside the agreed process (process). Any missing anchor is a signal that the slice isn't ready — resolve it before building, not during.

Journey Order

Build in the order a user travels through the product. First screen before second. Never build a screen that depends on state from an unbuilt screen.

RuleWhat it means
Journey order, not parallel Build in the sequence a real user experiences them. A problem found at Screen 2 costs far less to fix than one discovered after all five screens are built.
Dependencies block — hard gate The backlog dependency field is enforced. If SL-003 depends on SL-001, SL-001 must be Done before SL-003 can enter In Build. Surface it clearly rather than finding a workaround that creates hidden coupling.
Parallel exceptions named explicitly Screens with no shared infrastructure and no journey dependencies can run alongside the main journey if they're Ready. Name them explicitly as parallel — not next in journey order.
Dependency types matter Slice (another slice must be Done), Infrastructure (service or DB must exist), Data (mock entity must be available), or Design (decision still open). The type determines what resolves the block.

The Tracer Bullet

Before building any individual screen in full, build the thinnest path all the way through the user journey. Prove the core loop works end to end — even if every step is minimal.

What it is

The minimum that proves the core loop works end to end. Not the minimum that proves one screen works — the minimum that proves the user can travel from the first action to the moment of value, even if every step is minimal. Thin everywhere before deep anywhere.

  • Can the user reach the core interaction?
  • Does navigation between screens work?
  • Does data flow from source through the UI?
  • Does the journey reach the value moment?
Why it goes first

Building one screen to completion before touching the next hides integration problems until late. The tracer bullet forces every screen to exist in minimal form before any screen is expanded — so integration problems surface when the codebase is still small.

  • Integration failures surface early, cheaply
  • Core loop validated before building depth
  • Navigation and state flow proven before expansion
  • Something real to hand off to QA from day one
Tracer bullet approachScreen-by-screen approach
Screen 1 (thin) → Screen 2 (thin) → Screen 3 (thin), then expand each Screen 1 (full) → Screen 2 (full) → Screen 3 (full)
Integration problems surface at Screen 2, not Screen 5 Integration tested for the first time at Screen 3
End-to-end verified before feature depth is added Three complete screens may need rewiring when they meet

Mid-Build Discoveries

Build reveals things the design review didn't. How a discovery gets handled depends on its nature and scope — but it is always surfaced, never silently absorbed.

TypeWhat it isAction
Minor — implementation detail Doesn't affect the design, the agreed process, or any other slice. Has an obvious solution. Resolve it, note it in the slice detail in the backlog, continue building.
Design gap Something on the design screen can't be built as shown — missing state, an interaction that requires data that doesn't exist, or a visual that implies logic not captured. Stop. Surface it specifically with options. Return to design-review for a decision. Do not silently simplify the implementation.
Affects other slices Building the current slice reveals something that will change how an upstream or downstream slice needs to be built — a shared component, a state assumption, a data model implication. Flag in the backlog immediately. Note which slices are affected and why. Continue current slice if possible.
Blocks current slice A dependency that was marked as resolved turns out to not be resolved — the service doesn't work as expected, the data isn't available, or a design decision still isn't settled. Stop the current slice. Move it back to In Review with a note explaining the block. Pick the next appropriate slice while the blocker is resolved.
Never silently absorb a discovery into the code. If the implementation diverges from the design — for any reason — that divergence is a decision that needs to be made explicitly. Silent absorption creates technical debt that won't be visible until QA, the phase test, or a solo two months later who wonders why the code doesn't match the design.

The Auto-Chain to QA

Code-complete is not Done. It is the start of QA. The handoff from Build to QA is automatic — solo-build triggers it, the solo doesn't manage it.

Exact wording when code-complete is reached: "SL-[ID] is code-complete. Committing and running code-review-and-quality."
Code-complete means
  • Self-verification checklist run with ✓/✗ results
  • Slice renders correctly against mock data
  • Visual matches the design screen
  • No hardcoded values where data should be
  • Committed with a traceable commit message
Code-complete does NOT mean
  • Done — QA hasn't run yet
  • Verified — AI testing hasn't happened yet
  • Accepted — solo browser sign-off hasn't happened yet
  • Deployable — phase test gate not yet open

What to Watch For

Failure modeWhy it matters
Starting build on a slice that isn't Ready A slice in "In Review" has open design decisions. Starting build before those close means the code will contradict a decision made later. When there are no Ready slices, the right move is finishing design review — not finding a way to start build anyway.
Asking the solo what to work on next The plan and priority already exist in the backlog. Read the backlog, identify the next Ready slice in priority order, and state it.
Building without all four anchors confirmed Starting with three anchors and resolving the fourth "as you go" means the fourth anchor gets resolved under time pressure, often incorrectly. Any missing anchor is a stop signal, not a caution.
Building out of journey order Jumping ahead to a screen that looks easier hides integration problems. They surface later — after multiple screens are complete and rewiring is expensive.
Silently absorbing mid-build discoveries When the implementation diverges from the design and nobody says anything, QA checks against the design, the phase test checks against the discovery brief, and the mismatch surfaces late.
Skipping the tracer bullet Going straight to full feature implementation on Screen 1 before touching Screen 2 trades a cheap early integration fix for an expensive late one.
Hardcoding values that should come from data Any value that should come from a real data source must come from the mock data layer during build — not from a hardcoded string. Hardcoded values break the data anchor and make the mock-to-real swap unreliable.
Treating code-complete as Done Marking a slice Done before code review and solo QA sign-off bypasses the two checks that actually confirm the slice is complete.
Vague done criteria accepted at anchor time "It should look right" is not a done criterion. If the done anchor is accepted with vague criteria, QA has nothing concrete to verify against. Sharpen every criterion to something observable and pass/fail.
Starting a visual slice without re-reading the design file The design anchor means actually opening and reading the file before writing code. A design file that has changed since the last session, or one the builder remembers slightly wrong, produces implementation drift that compounds across slices.
Continuing past two failed attempts without stopping After two attempts with no progress, stop and re-read the full slice spec and design file from scratch before touching any code. Confidence that the next attempt will work is not a reason to continue.
Ignoring a rising rework rate When multiple slices are returning to In Build after QA fails, or multiple mid-build discoveries surface in sequence, it's a signal that something earlier wasn't settled correctly. Stop and diagnose before continuing.
