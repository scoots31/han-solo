Discover + Tech Context

The foundation everything else builds from. Discover surfaces the full story before any design decisions are made. Tech Context locks the stack before any screens are drawn. Together they prevent the most expensive mistakes in the framework.

Visual artifact. Only begins when both gates above are passed.

Discover is the day-one conversation. It is not requirements gathering and it is not a scoping session. It is the conversation a good engineer or UX person has before they touch a keyboard — the one where they listen, ask the right questions, and build a shared picture of what you're actually trying to create. By the end, everyone would tell the same story. That story feeds the design sprint, not an engineer's backlog.

The four zones of the Discover conversation — who, the journey, the moment it matters, and what done looks like — surface the full use case from beginning to end. Each zone is explored at a depth calibrated to the complexity of what you're building. A simple internal tool gets five questions. A greenfield product with multiple users and branching flows gets a multi-round conversation.

Tech Context runs immediately after Discover, before Design Sprint begins. It establishes what you're building on and what constraints that creates — platform, stack, architecture rules, engineering principles. This matters because every downstream decision is technically constrained: the design can only show what the stack can produce, the data scaffold is tied to the data layer, the plan's done criteria are written in terms of the framework in use. A design sprint that runs without tech context produces screens that may be unbuilable on the actual platform.

Together these two phases are the foundation. If either is skipped or incomplete, every phase downstream either makes wrong decisions or has to backtrack. The gates exist precisely because of this — there is no way into Design Sprint without them both.

Skills in this phase
discover
Phase Skill
Conversational discovery that builds a shared understanding of what you're trying to create — beginning to end. Not requirements gathering. Not a spec. A story-based brief that feeds the design sprint. The core question: "Do we understand this well enough to make something that shows what it could be?"
What it does
  • Receives your high-level overview without interruption
  • Confirms shared understanding with one or two clarity questions
  • Conducts the narrative conversation across four zones
  • Asks clarifying questions to complete the as-is picture for process-mapper
  • Presents and validates the as-is and to-be process maps
  • Produces the discovery brief as a short narrative document
  • Ends with the design on-ramp question to route what comes next
The four zones
  • Zone 1 — Who — real picture of the person and their context when they use this
  • Zone 2 — The Journey — walk through every step from open to close; no skipping ahead
  • Zone 3 — The Moment It Matters — where this becomes genuinely useful; the payoff moment
  • Zone 4 — What Done Looks Like — what they accomplished; what would make them come back
Outputs
  • docs/discovery-brief.md
  • docs/process/as-is-[name].md
  • docs/process/to-be-[name].md
  • Design on-ramp decision (Figma / reference / from scratch) documented at bottom of brief
tech-context
Phase Skill
Establishes the technical stack, architecture constraints, and engineering principles before design review, planning, or build. Created once per project — never recreated, only updated when the stack changes. Referenced by every downstream skill for the life of the project.
Two routing paths
  • Known profile — organizational project with an established stack (e.g. Bayer Aurora). Two questions: org or solo? Which platform? Loads profile, confirms key constraints, produces the doc.
  • General solo — no established profile. Asks questions one at a time: deployment target, frontend, backend/data, component library, API communication, auth, hosting, engineering principles.
  • Either path ends with a summary confirmation before writing the document
What it captures
  • Full stack table: frontend, component library, API comms, backend, auth, database, deployment, CI/CD, linting, testing
  • Architecture constraints — things design and build must account for beyond the stack
  • Engineering principles — standards and rules for this project
  • Branching and delivery model
  • Infrastructure slices — prerequisites that must be Done before any feature slice starts; identified here, not in Design Review
  • Secrets and config management pattern
Downstream impact
  • Design Sprint — architecture constraints shape what can be designed; Aurora module constraints mean no login screen
  • Data Scaffold — stack determines mock data format and data layer pattern
  • Plan — done criteria written with real technical specificity, not vague behavior descriptions
  • Build — the build bible. Stack, principles, branching, secrets — all here
process-mapper
Always-On (piloted / guided)
Activates during Discover in piloted and guided modes. Documents the as-is process and produces the agreed to-be map. The to-be map is the process contract — its agreed structure does not change without an explicit decision.
What it does in this phase
  • Documents the as-is process exactly as described — including manual steps, workarounds, and exceptions outside any system
  • Presents as-is for validation before proceeding: "Does this accurately reflect what happens today — including the exceptions and manual steps?"
  • Drafts the to-be map from the journey conversation — every step, every decision point, every outcome; annotated with which zone it came from
  • Presents to-be for agreement before Design Sprint begins: "Does this map reflect what we're building? Are there decision points or branches we haven't covered?"
  • Produces both maps as markdown files in docs/process/
The process contract
  • The to-be map agreed here is the contract that every downstream skill works against
  • Design Sprint cross-references screens against the to-be map — gaps flagged and decided
  • Design Review requires a process anchor on every slice before it reaches Ready
  • Phase Test validates that what was built matches the to-be map intent
  • The map is updated as screens and slices are added — but its agreed structure does not change without an explicit decision
📋
Step-by-step experience
1
Discover
You give the high-level overview

Describe what you want to build at a high level. This is your opening — the discover skill does not interrupt it or redirect it. It lets you tell it your way. After you finish, it scans for who this is for, what problem or opportunity it addresses, and any existing context in the project (prior docs, blueprints, earlier briefs).

2
Discover
Clarity check (one round, at most)

One or two questions at most to confirm shared understanding before going deeper. The right kind: "When you say [X], do you mean [A] or [B]?" — not challenges, not premise questions, not ROI questions. If the overview was clear, this step is skipped and the conversation moves directly to the zones.

3
Discover
Narrative conversation — Zone 1: Who

Not a demographic. A real picture of the person using this, their context when they arrive, what triggered them to come here, and what they already know versus what they don't. One or two questions per round — never a barrage. If an answer is vague, go one level deeper before moving on.

Zone 1 — Who
The user in context
"Who is this person and what's their context when they use this?" — "What are they doing before they open this?"
Zone 2 — The Journey
Every step, in order
"Walk me through what someone does from the moment they open it." — "And then what happens?" Don't skip ahead.
4
Discover
Narrative conversation — Zones 3 & 4
Zone 3 — The Moment It Matters
The payoff
"Where does this actually become valuable to them?" — "What's the payoff moment — the thing they'd tell someone else about?" This often becomes the hero screen.
Zone 4 — What Done Looks Like
Outcomes, not features
"When they're done and it worked — what did they accomplish?" — "What would make them come back?" Grounds the design in real outcomes.

Depth is calibrated: simple use case = 3–5 questions total; moderate complexity = 5–10; vague greenfield = multi-round conversation. If you say "that's enough" — the skill respects it, notes where clarity is thin, and proceeds.

5
process-mapper
As-is and to-be process maps

Before writing the discovery brief, process-mapper activates. It asks any remaining clarifying questions to document how the process works today — including manual steps, workarounds, and exceptions that happen outside any system. The as-is may be entirely offline: emails, spreadsheets, phone calls, decisions in someone's head.

The as-is map is presented for validation. Once confirmed, the to-be map is drafted from the journey conversation and presented for explicit agreement. This agreement is the process contract. Both maps are saved to docs/process/.

6
Discover
Discovery brief written + design on-ramp decided

A short narrative document — not a spec, not bullet-point requirements. The story, key moments in the journey, open threads that don't need resolving before design starts, and what's explicitly out of scope. Small on purpose: a good designer doesn't need 40 pages to start sketching.

The brief is saved to docs/discovery-brief.md. Then one final question: "Do you have an existing Figma file, a product you'd like to reference for look and feel, or are we starting from scratch?" The answer is documented at the bottom of the brief and routes the design sprint.

7
tech-context
Stack established via profile or questions

Two paths. If this is an organizational project (e.g. Bayer Aurora), a profile is loaded and key constraints are confirmed. If it's a general solo project, questions are asked one at a time — deployment target, frontend, backend, component library, API communication, auth, hosting, engineering principles. Never asked all at once.

When enough is known, the stack is summarized for confirmation: "Based on what you've shared: [summary]. Before I write this up — anything else that would constrain how we build?"

8
tech-context
Infrastructure slices identified

As part of writing docs/tech-context.md, infrastructure slices are called out — these are the prerequisite slices that must be Done before any feature slice can start. They are identified from the tech stack and platform constraints now, not later during Design Review. A project on Aurora that calls an existing API has infrastructure slices. A greenfield solo project starting with a mock data layer has infrastructure slices. These go directly into the backlog when Design Review begins.

Documents produced
docs/discovery-brief.md
discover
Discovery Brief
The story in narrative form — not requirements, not a spec. Contains: the story (2–4 paragraphs), key moments in the journey, open threads that don't block design, what's explicitly out of scope, and the design on-ramp decision at the bottom. Written so that anyone who reads it can picture the product.
Read by: design-sprint (the story layer), phase-test (acceptance reviewer checks against this), product-continuity throughout
docs/process/as-is-[name].md
process-mapper
As-Is Process Map
How the process works today, exactly as described — including manual steps, workarounds, offline decisions, and steps that happen outside any system. The as-is may not involve any software at all. Validated with you before finalization. [name] is the process name (e.g. as-is-onboarding.md, as-is-order-review.md).
Read by: process-mapper (cross-reference during design review and phase test)
docs/process/to-be-[name].md
process-mapper
To-Be Process Map
The agreed target process — how this will work when the product is built. Every step from the journey conversation, every decision point, every outcome. Annotated with which zone each step came from. Explicitly agreed before Design Sprint begins. This is the process contract: its structure does not change without an explicit decision. Updated as screens and slices are added, annotated with screen references during Design Sprint.
Read by: design-sprint (gaps flagged), design-review (process anchor required per slice), phase-test (acceptance coverage)
docs/tech-context.md
tech-context
Tech Context
Created once per project. Contains: what's being built (one paragraph), full stack table, architecture constraints, engineering principles, branching and delivery model, infrastructure slices required, secrets and config pattern, and profile reference. Never recreated — only updated when the stack changes.
Read by: design-sprint (architecture constraints), data-scaffold (stack determines data layer pattern), plan (done criteria specificity), solo-build (build bible), deploy (deploy method entirely determined by stack)
🔒
The gates

Two gates stand between this phase and Design Sprint. Both must be passed. There is no partial credit — skipping or soft-passing either gate causes downstream phases to make wrong decisions or forces backtracking.

🗺
Gate 1 — Discover complete
After Discover
Required artifacts
  • docs/discovery-brief.md exists and contains: story, key moments, open threads, out of scope, design on-ramp decision
  • docs/process/as-is-[name].md exists and was validated with you
  • docs/process/to-be-[name].md exists and was explicitly agreed — not just drafted
Quality check
  • The story is specific enough that a designer could picture the product and identify a hero screen candidate
  • The to-be map covers all steps and decision branches that were described in the journey conversation
  • Out of scope items are explicitly listed — not just implied
  • The design on-ramp is documented (Figma / reference / from scratch)
Not met: Return to the discover conversation. The most common issue is a to-be map that was drafted but not actually validated — revisit the agreement step with the process-mapper output visible.
⚙️
Gate 2 — Tech Context complete
After Tech Context
Required artifact
  • docs/tech-context.md exists in the project
  • Stack table is complete — no empty rows for layers that apply to this project
  • Architecture constraints section populated — even if minimal
  • Infrastructure slices table populated (or explicitly noted as "none required")
What passes this gate
  • A known profile was loaded, key constraints confirmed
  • Or: questions were answered and the summary was confirmed before the document was written
  • Engineering principles documented — even if they are "TBD, to be established in first build slice"
Not met: Run tech-context before opening the design sprint. The design sprint skill checks for this file — if it does not exist, downstream skills cannot route correctly.

Handoff to Design Sprint
What Design Sprint reads from this phase
Three documents. Three distinct purposes. All required.
The story
  • Reads docs/discovery-brief.md as the narrative foundation
  • The story gives the design sprint context for every decision: what screen matters most, what the payoff moment looks like, what's out of scope
  • Zone 3 (the moment it matters) is the hero screen candidate — the design sprint opens here
  • The design on-ramp at the bottom of the brief determines the starting point: from scratch, Figma import, or reference-based
What to build against
  • Reads docs/process/to-be-[name].md as the build target
  • process-mapper cross-references every screen in the sprint against the to-be map — steps not covered by any screen are surfaced as gaps for you to decide on
  • Annotates the to-be map with screen references as screens are approved
  • The to-be map is the process contract — no step is designed around without a decision
Aesthetic + stack constraints
  • Reads docs/tech-context.md for what can and cannot be designed
  • Aurora module constraint = no login screen, no global nav in the designs
  • Component library = design uses those components, not custom ones
  • Mobile target = layout decisions are made for that form factor from frame one
  • No-backend constraint = no screens that imply server-side logic that isn't there
Design Sprint stops if either gate is missing. The discovery brief is how the sprint knows what to design. The tech context is how it knows what it can design. Without both, the artifact produced may be a complete mismatch with what you can actually build.
⚠️
What to watch for
Issue
Why it matters
What to do instead
To-be map agreed too quickly The map was drafted and you said "looks fine" without real scrutiny of whether it covers all the decision branches you described in the journey.
This is the process contract. Every downstream decision — screen design, slice definition, done criteria, phase test — is anchored to it. A vague or incomplete to-be map produces slice boundaries that don't match the real process, which surfaces as confusion in Design Review and rework in Build.
Go back to Zone 2 of the conversation. Walk the journey again step by step. Confirm that every step you described is represented in the to-be map. Check: are there branches? Edge cases you mentioned? Multiple user types doing different things? The map must cover them.
Tech Context skipped or deferred "We'll figure out the stack later" or "let's just get into design first."
Downstream skills cannot route correctly. The design sprint may produce screens for a mobile app when you're building a web module. The data scaffold defaults to the wrong data layer. Plan done criteria are written in the abstract. Infrastructure slices are never identified — and feature work begins with no foundation underneath it.
Run tech-context before opening design-sprint. It takes 5–10 minutes for a solo project and less for a known organizational profile. This is not optional overhead — it is the routing mechanism for everything that follows.
Discovery brief too vague to produce a hero screen The story section is abstract — describes a type of product without a clear picture of who uses it and what they do.
Zone 3 (the moment it matters) is supposed to surface the hero screen candidate. If the brief doesn't have a specific payoff moment, the design sprint opens without a starting target. The warmer/colder loop becomes untethered. Screens get designed that nobody can evaluate against the brief.
Before writing the brief, check: can you name the specific moment in the journey where this becomes genuinely useful? If not, go deeper in Zone 3. Ask: "What's the thing they'd tell someone else about?" If you still can't name it, the Zone 2 journey may not be specific enough — go back and sharpen it first.
As-is process skipped because "there is no current process" Assuming the process doesn't exist because there's no system today.
There is always an as-is process. It may be entirely manual — an email chain, a spreadsheet, a phone call, a decision someone makes in their head. Skipping it means the to-be map is designed without accounting for the manual workarounds and exception-handling steps that the new product must replace or improve on.
Ask specifically: "How does this happen today — even if there's no system?" and "What's the most common place this goes wrong today?" Then document exactly what was described. Even "someone sends an email and waits" is a valid as-is step.
Infrastructure slices not identified during Tech Context Tech Context is completed but the infrastructure slices table is left empty or marked TBD.
Infrastructure slices are the prerequisites before any feature slice can start — authentication setup, API client wiring, database schema, environment configuration. If they're not identified here, feature slices go into the backlog without a foundation to build on. The first feature build stalls waiting for infrastructure work that wasn't planned.
When producing docs/tech-context.md, ask: "What must exist before a single feature slice can start?" For a React app: Can the project boot? Is the component library installed? Is auth wired? For an Aurora module: Is the shell integration established? These become infrastructure slices in the backlog.
