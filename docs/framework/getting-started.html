Getting Started

Everything you need to know to start your first project — which path to take, how to enter the framework, and what a working session looks like.

Is This a Product or a Spike?

Before you pick a mode or invoke anything, answer one question: will other people use this, or is it just for you? That answer routes you to one of two systems.

Use the Framework

Product

Other people will use this. Lifespan is indefinite. There are real users, real requirements, and decisions made now that will matter later.

Say "let's start" or describe what you're building.

Use Workshop

Spike or personal tool

Exploratory work, a script for yourself, or something timeboxed to hours or days. Outcome is learning or a small artifact — not a shipped product.

Invoke /scope-check to enter Workshop.

Spikes can graduate. If a personal tool turns into something you want to ship, Workshop's Promote outcome hands it to the framework at the right phase. You don't have to decide forever today.

Three Ways to Enter

All three paths start the same way — say "let's start" or describe what you have. The routing skill reads what you brought and routes you automatically. No flags, no setup.

Entry 1

You know what you want to build

Describe the idea, even briefly. You'll go through a full narrative conversation — what it does, who it's for, what you're explicitly not building — and come out with a discovery brief and a to-be process map.

Route: start → discover

Entry 2

You're exploring an idea

Something isn't fully formed yet. Say "let's start" or "I'm thinking about…" and you're routed to Brainstorming first — a genuine collaborative exploration that produces directional clarity and a rough sketch before committing to Discover.

Route: start → brainstorming → discover

Entry 3

You have prior work

Notes, decisions, maybe some code already written. Describe what you have. The onboard skill reads your project directory, maps prior work to framework artifacts, identifies gaps, and places you at the right phase — without making you re-do work you've already done.

Route: start → onboard → [correct phase]

The front door: Always start with "let's start" or just describe what you're working on. The routing skill reads your opening message and routes you — you don't need to know which path you're on before you begin.

Choose a Session Mode

Modes control which skills load automatically. Set your mode at the start of a session by saying "guided mode" or "piloted mode." Default is bare. You can change modes mid-project.

Default

Bare

Nothing loads until you invoke a skill. Zero overhead.

Right for: focused work, single-skill sessions, builders who want full control over what runs.

Active

Piloted

You're flying it. The four always-on skills load at session start — memory, continuity, process accountability, and health stay active. You choose which phase to run next.

Right for: builders who know the framework and want the instruments live while staying in control of the course.

Guided

Guided

The framework leads, you decide. Always-on skills plus the full phase chain — phases flow forward automatically. You're in every conversation and every sign-off, but not managing the navigation.

Right for: new projects, unfamiliar phases, or when you want to stay in the work rather than managing the framework.

When in doubt: Start bare. If you find yourself invoking the same skills every session, switch to piloted. Guided pays for itself on new projects — it costs more context but the framework carries you through the phase sequence automatically.

What a First Session Looks Like

For a brand-new product idea, here's the practical sequence from open to close.

StepWhat happens
1. Open Claude Code or Cursor Navigate to an empty directory or create one. This becomes your project root — all framework artifacts live here.
2. Say "let's start" and describe what you want to build It doesn't have to be polished. A sentence or a paragraph. If you want a specific mode, say it here: "guided mode, let's start."
3. Routing reads your opening and routes you You'll see a brief orientation — which phase you're entering and why — and then the Discovery conversation begins. No setup, no files to create manually.
4. Discovery conversation runs You'll walk through the idea from beginning to end: who it's for, key moments in the experience, what success looks like, and what you're explicitly not building. Expect it to surface things you hadn't articulated yet.
5. Gate outputs are written Discover writes two artifacts: a discovery brief (docs/discovery-brief.md) and a to-be process map (docs/process/).
6. Gate check: both artifacts must exist Before the Design Sprint begins, both outputs must be present. The gate exists because Design Sprint reads them. If you close out before they're written, the next session can resume from this phase.

The Phase Sequence

The framework moves through phases in order. Each phase has a required output and a gate that the next phase checks before starting. You can't skip gates — but you can pause at any gate and resume cleanly.

PhaseProducesGate (next phase requires)
Brainstorming (optional) Directional clarity, rough sketch, a defined idea worth discovering Decision to proceed to Discover
Discover discovery-brief.md · process maps Both files exist and are complete
Tech Context tech-context.md Stack, constraints, and infra decisions documented
Design Sprint HTML screen artifacts for every screen before any code is written All screens reviewed and approved
Data Scaffold + Design Review Realistic fake data wired to screens; design review passed Screens match approved designs with real-shaped data
Plan prd.md sequenced into slices with acceptance criteria Plan reviewed; slices are independently buildable
Build Working code, slice by slice. Say "show progress" for the active deliverable · "show plan" for the full build. All slices complete; QA can run end-to-end
QA Verified behavior, triage log for anything unexpected No open critical issues; phase-test can run
Phase Test Full-phase test run against acceptance criteria All phase criteria pass
Deploy Production deployment, launch checklist complete App live; retrospective captured

Session Hygiene

The framework is designed so sessions are self-contained. Gates are the natural close points. A fresh session that starts with gate output has full context — you don't lose anything by stopping at a gate.

End at gates, not mid-phase

Each gate output is a complete, readable artifact. The next phase reads it cold and picks up with full context. Stopping mid-phase means the next session has to reconstruct partial work.

One phase per session — roughly

Short phases (Discover, Tech Context, Plan) each fit in a session. Long phases like Build should be broken into sprint slices — complete one slice per session, commit at the end.

To close out: say "let's close out"

The continuity skill captures current state, updates the handoff document, and generates a resume prompt. Copy the resume prompt — paste it to start the next session.

To resume: paste the prompt or say the name

Option 1 — Resume prompt: Open a new session, paste the resume prompt. The framework reads it and continues from exactly where you left off.

Option 2 — Named resume: Add your project to projects.md in the playbook root. Then say "guided on [project name]" — no saved prompt needed.

The pattern that works: Open session → "guided on [project]" or paste resume prompt → work to the next gate → say "let's close out" → close. Repeat. The framework carries state between sessions so you don't have to.

Workshop — For Non-Product Work

Workshop is the companion system for spikes, personal tools, and exploratory builds. It's lighter, faster, and designed for work that has a clear endpoint. Enter with /scope-check.

Invoke /scope-check. It asks one question: who is this for and how long will it live? The answer routes you into the right Workshop track.

Toss

The work is done and doesn't need to live on. Workshop captures the key learning as a concise note and archives the artifact. You leave with something worth keeping, without a codebase to maintain.

Keep

The tool works and you'll keep using it — just not as a product. Workshop marks it as a personal tool: no framework gates, no shipping requirements. It lives in your local environment.

Promote

The spike graduated into something real — other people will use it, or you want to build it properly. Workshop hands off to the framework at the correct phase, carrying over what's already been done so you don't re-run discovery on something you already understand.

Don't force things. Not every good idea needs to be a product. If you find yourself in Workshop, take the Toss or Keep outcome seriously. The cost of prematurely promoting a spike into the full framework is real — you get all the overhead before you're sure the idea is worth it.
