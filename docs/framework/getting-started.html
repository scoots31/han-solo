Solo Builder Framework

Getting Started

Everything you need to know to start your first project — which path to take, how to enter the framework, and what a working session looks like.

01
Is this a product or a spike?

Before you pick a mode or invoke anything, answer one question: will other people use this, or is it just for you? That answer routes you to one of two systems.

Use the Solo Builder Framework
Product
Other people will use this. Lifespan is indefinite. There are real users, real requirements, and decisions made now that will matter later.

→ Start here. Say "let's start" or describe what you're building.
Use Workshop
Spike or personal tool
Exploratory work, a script for yourself, or something timeboxed to hours or days. Outcome is learning or a small artifact — not a shipped product.

→ Invoke /scope-check to enter Workshop.
Note
Spikes can graduate. If a personal tool turns into something you want to ship, Workshop's Promote outcome hands it to SBF at the right phase. You don't have to decide forever today.
02
Three ways to enter SBF

All three paths start the same way — say "let's start" or describe what you have. start reads what you brought and routes you automatically. No flags, no setup.

1
You know what you want to build
Describe the idea, even briefly. start routes you to Discover. You'll go through a full narrative conversation — what it does, who it's for, what you're explicitly not building — and come out with a discovery brief and a to-be process map.
→ start → discover
2
You're exploring an idea
Something isn't fully formed yet. You have a direction but not a definition. Say "let's start" or "I'm thinking about…" and start routes you to Brainstorming first. That skill is a genuine collaborative exploration — not a requirements dump — and produces directional clarity and a rough sketch before you commit to Discover.
→ start → brainstorming → discover
3
You have prior work
Notes, decisions, maybe some code already written. Describe what you have. start routes to onboard, which reads your project directory, maps prior work to framework artifacts, identifies gaps, and places you at the right phase — without making you re-do work you've already done.
→ start → onboard → [correct phase]
The front door
Always start with "let's start" or just describe what you're working on. start reads your opening message and routes you — you don't need to know which path you're on before you begin.
03
Choose a session mode

Modes control which skills load automatically. Set your mode at the start of a session by saying "guided mode" or "piloted mode." Default is bare. You can change modes mid-project.

bare default
Nothing loads until you invoke a skill. Zero overhead.
Right for: focused work, single-skill sessions, builders who want full control over what runs.
piloted
You're flying it. The four always-on skills load at session start — memory, continuity, process accountability, and health stay active. You choose which phase to run next.
Right for: builders who know the framework and want the instruments live while staying in control of the course.
guided
The framework leads, you decide. Always-on skills plus the full phase chain — phases flow forward automatically. You're in every conversation and every sign-off, but you're not managing the navigation.
Right for: new projects, unfamiliar phases, or when you want to stay in the work rather than managing the framework.
When in doubt
Start bare. If you find yourself invoking the same skills every session, switch to piloted. Guided pays for itself on new projects — it costs more context but the framework carries you through the phase sequence automatically.
04
What a first session looks like

For a brand-new product idea, here's the practical sequence from open to close.

1
Open Claude Code or Cursor
Navigate to an empty directory or create one. This becomes your project root — all framework artifacts live here.
2
Say "let's start" and describe what you want to build
It doesn't have to be polished. A sentence or a paragraph. start reads it and routes you — no configuration required. If you want a specific mode, say it here: "guided mode, let's start."
3
start reads your opening and routes you
You'll see a brief orientation — which phase you're entering and why — and then the Discovery conversation begins. No setup, no files to create manually.
4
Discovery conversation runs
You'll walk through the idea from beginning to end: who it's for, key moments in the experience, what success looks like, and — just as importantly — what you're explicitly not building. This conversation is generative, not an interview. Expect it to surface things you hadn't articulated yet.
5
Gate outputs are written
Discover writes two artifacts: a discovery brief and a to-be process map.
docs/discovery-brief.md docs/process/
6
Gate check: both artifacts must exist
Before the Design Sprint begins, both outputs must be present. The gate exists because Design Sprint reads them. If you close out before they're written, the next session can resume from this phase using the resume prompt.
05
The phase sequence

SBF moves through phases in order. Each phase has a required output and a gate that the next phase checks before starting. You can't skip gates — but you can pause at any gate and resume cleanly.

Brainstorming optional Directional clarity, rough sketch, a defined idea worth discovering Decision to proceed to Discover

QA Verified behavior, triage log for anything unexpected No open critical issues; phase-test can run
Phase Test Full-phase test run against acceptance criteria All phase criteria pass
Deploy Production deployment, launch checklist complete App live; retrospective captured
06
Session hygiene

The framework is designed so that sessions are self-contained. Gates are the natural close points. A fresh session that starts with gate output has full context — you don't lose anything by stopping at a gate.

End at gates, not mid-phase
Each gate output is a complete, readable artifact. The next phase reads it cold and picks up with full context. Stopping mid-phase means the next session has to reconstruct partial work. Stop at gates.
One phase per session — roughly
Short phases (Discover, Tech Context, Plan) each fit in a session. Long phases like Build should be broken into sprint slices — complete one slice per session, commit at the end.
To close out: say "let's close out"
product-continuity captures current state, updates the handoff document, and generates a resume prompt. Copy the resume prompt — you'll paste it to start the next session.
To resume: paste the resume prompt — or just say the name
Option 1 — Resume prompt: Open a new session, paste the resume prompt. The framework reads it and continues from exactly where you left off.

Option 2 — Named resume: Add your project to projects.md in the playbook root (one line: name → path). Then say "guided on [project name]" — no saved prompt needed. The framework finds the project, reads the handoff, and picks up from there.
The pattern that works
Open session → "guided on [project]" or paste resume prompt → work to the next gate → say "let's close out" → close. Repeat. The framework carries state between sessions so you don't have to.
07
Workshop — for non-product work

Workshop is SBF's companion system for spikes, personal tools, and exploratory builds. It's lighter, faster, and designed for work that has a clear endpoint. Enter with /scope-check.

Entry
Invoke /scope-check. It asks one question: who is this for and how long will it live? The answer routes you into the right Workshop track.
Toss
The work is done and doesn't need to live on. Workshop captures the key learning as a concise note and archives the artifact. You leave with something worth keeping, without a codebase to maintain.
Keep
The tool works and you'll keep using it — just not as a product. Workshop marks it as a personal tool: no SBF gates, no shipping requirements. It lives in your local environment.
Promote
The spike graduated into something real — other people will use it, or you want to build it properly. Workshop hands off to SBF at the correct phase, carrying over what's already been done so you don't re-run discovery on something you already understand.
Don't force things
Not every good idea needs to be a product. If you find yourself in Workshop, take the Toss or Keep outcome seriously. The cost of prematurely promoting a spike into SBF is real — you get all the overhead before you're sure the idea is worth it.
