Data Scaffold + Design Review

The phase that converts design screens into a defined, executable backlog. Two skills run in parallel — one builds the data foundation, the other defines every unit of work. Build cannot start without it.

What this phase is
Phases 2.5 and 2.6 run in parallel — not in sequence

This phase sits between the Design Sprint and Planning. Its job is to take the design artifact — the HTML screens that came out of the sprint — and do two things simultaneously: build a data foundation under them, and define every unit of work well enough to build.

Data Scaffold (phase 2.5) generates realistic mock data for every piece of information that appears on a screen, before a single real API exists. The output is a data/mock/ layer — one place where all fake data lives — and a docs/data-mapping.md document that maps every mock field to its eventual real source. That document is the first draft of the API contract. Design review that happens against real-looking data produces dramatically better decisions than design review against empty placeholders.

Design Review (phase 2.6) is an iterative loop. Each round applies three specialist lenses — PM, Tech Lead, and Data — to every screen. Findings become slice definitions or spike triggers. Slices move through states. When enough slices have reached Ready, build can start. Review then continues in parallel on the remaining slices.

These two skills inform each other. The data-scaffold output is what the data lens in design-review reads. The design-review data lens surfaces fields that haven't been scaffolded yet. Running them separately, in sequence, defeats the value of both. Run them together.

The phase ends when there's enough to learn from, not when everything is defined. Build starts when the first coherent set of Ready slices exists — the thinnest path through the system that proves the core loop works. Design review continues on remaining slices while build is in progress.

Skills in this phase
Two phase skills run in parallel; one supporting skill triggered on demand
data-scaffold
Phase skill 2.5
Generates realistic fake data for screens that don't have real APIs yet. Makes design review actually useful — screens with real-looking data produce better decisions than screens with placeholder text. Fake data lives in one mock layer, not scattered through the UI, so swapping to real data later is a replacement, not a rewrite. The mapping document is the first draft of the API contract.
When to run
  • Before or during first design review round
  • When a screen has placeholder text where data should be
  • When design review surfaces a field that doesn't exist yet
  • When build starts and real APIs aren't ready
Outputs
  • data/mock/[entity].json one file per data entity
  • data/mock/mock-index.js exports all mock data
  • docs/data-mapping.md proto-API contract
  • Mock indicator badge on all screens
  • Single swap seam in UI code
Key rules
  • Domain-accurate values — realistic ranges, not arbitrary numbers
  • Structurally honest — field names match plausible API shape, not UI labels
  • Edge cases included — always: a zero score, a missing field, a bye week
  • One seam — UI calls a function; mock or real lives underneath
  • Labeled everywhere — mock badge + comment pattern, no ambiguity
design-review
Phase skill 2.6 — iterative
Iterative review loop between the design artifact and the build decision. Each round adds knowledge, refines the design, defines slices, and updates the backlog. A slice reaches Ready when it's defined well enough to build. Build starts when enough slices are Ready. Review continues in parallel. The backlog is the persistent document that makes all of this visible.
Each round does
  • Applies PM, Tech Lead, and Data lenses to every screen
  • Names and classifies each finding immediately
  • Runs process coverage check against to-be map
  • Defines or refines slices
  • Moves slices through states
  • Updates docs/backlog.md in real time
Gate inputs required
  • docs/design/sprint-[id].html
  • docs/design/deferred-decisions.md
  • To-be process map — non-negotiable. No to-be map means no slice review.
  • docs/data-mapping.md (when available)
  • Any spike results in docs/spikes/
Outputs
  • docs/backlog.md created Round 1, updated every round
  • Slice definitions with IDs, process anchors, and done criteria
  • Process coverage map per round
  • Review log appended after each round
  • Spike triggers for knowledge gaps
research-spike
Supporting skill
Timeboxed investigation of a specific unknown that's blocking a decision. Triggered when design-review finds a knowledge gap on a slice — a question that can't be answered from what's known, and where the answer changes what gets built. The output is a named, scoped answer that unblocks the slice.
When triggered
  • Design review finds a field whose source is unknown
  • A slice depends on a technical approach that hasn't been validated
  • A process step is unclear and the design assumes it
  • An API's shape or availability is uncertain
How it integrates
  • Slice moves to Blocked when spike is triggered
  • Spike result lives in docs/spikes/
  • Next design-review round reads the result
  • Slice unblocks and can be promoted to Ready
Key constraint
  • Timeboxed — one focused session, specific question
  • Must produce a decision, not just more research
  • Every blocked slice must name the spike question explicitly — no vague blockers
  • Spikes are tracked in the backlog At a Glance

The design-review loop
How rounds work — from first pass to build signal
Round 1 — Full first pass
Establish baseline

The first round reviews every screen. Every element gets all three specialist lenses applied. Nothing skips review. Nothing starts as Ready — slices need at least one round of review before they can be promoted.

The main job of Round 1: produce the full coverage map against the to-be process, trigger spikes for anything that blocks definition, and create docs/backlog.md with every slice named and in an initial state. No slice reaches Ready on Round 1 unless it has a clear design reference, defined done criteria, no open spike, and a confirmed process anchor. That's rare on a first pass.

End Round 1 with an explicit statement: what's In Review, what's Blocked (with named spikes), what's Deferred, and what the next round needs to resolve. The coverage map is a deliverable — every uncovered to-be step is a decision, not a silent gap.

Round 2+ — Refine and promote
Move slices forward

Each round after the first is driven by what changed: spike results returned, design updates made, new screens added, stakeholder feedback received. Don't try to review everything equally every time — let the findings from the previous round drive the focus.

For each slice In Review: can it be promoted to Ready? Apply the five Ready conditions (see below). If yes, promote it. If not, name the specific blocker and carry it to the next round.

The process coverage check runs every round — not just Round 1. As slice definitions sharpen and process understanding deepens, gaps that weren't visible before may appear. Run the full coverage map every round and surface anything unresolved.

The three specialist lenses
PM Lens
Is this solving the right problem for the right person? Does the flow make sense from the user's perspective? What's missing from the story — what would a user hit that we haven't designed for? Does this screen earn its place in Phase 1?
Tech Lead Lens
Is this buildable with the stack we have? What are the real dependencies — not just the stated ones? What has to exist before this slice can start? Where is the complexity risk? Are we extending existing patterns or creating new ones?
Data Lens
Where does every visible field come from? Does the data model support what the design implies? What API calls are behind this screen? What doesn't exist yet that the design assumes? This is the lens that connects design-review to data-scaffold — every un-scaffolded field found here should trigger a data-scaffold pass.
The build signal
Move to Plan

Build starts when enough slices are Ready — not when everything is Ready. "Enough" means: is there a meaningful set of Ready slices that form a coherent starting point? Enough to learn something real from building them?

This is usually the first vertical slice through the system: the simplest path that produces something working end to end. Not the most feature-complete. Not the easiest. The one that proves the core loop works.

When that set exists, state it clearly: "These [N] slices are Ready and form a coherent Phase 1 starting point. Build can begin. Review continues on the remaining [N] slices in parallel." Build and review run in parallel from this point. The backlog is what keeps both tracks visible.

⬡
Slice states
Every slice in the backlog has one of these states at all times
✅
Ready
All five Ready conditions are met. Design reference is unambiguous, done criteria are defined, dependencies known, no open spike, process anchor confirmed. Can be built now.
Moves to: In Build when work starts.
🔄
In Review
Identified — named and tracked in the backlog — but not yet fully defined. Needs more refinement: design clarity, missing done criteria, or unresolved questions that don't yet warrant a spike.
Moves to: Ready when all five conditions are met. Blocked when a spike is triggered.
🔬
Blocked
Waiting on a specific spike result. The spike question must be named explicitly in the backlog — "Blocked: spike on [question]" is the only valid format. Vague blockers are not allowed.
Moves to: In Review (or Ready) when the spike result returns and the question is answered.
⏸
Deferred
Explicitly set aside — not in current scope. A trigger condition is noted (what would bring it back). This is a decision, not an oversight. Deferred slices appear in the backlog so nothing disappears silently.
Moves to: In Review when the trigger condition is met and scope changes.
🔨
In Build
Currently being worked on by the builder (solo-build skill). All four build anchors are confirmed before work starts: design anchor, data anchor, done anchor, and process anchor.
Moves to: Done when QA passes and the solo confirms in the browser.
✓
Done
Complete and verified. Code review passed, AI tested with evidence, solo confirmed in the browser. The slice's done criteria were verified — not assumed. Solo sign-off is required; code-complete alone is not Done.
Terminal state unless a regression is found, which routes through qa-triage.
✅
What "Ready" actually means
Five conditions — all five must be true. No exceptions.
Ready is not "I think we can build this." It's a checklist. The solo confirms it's clear enough to hand to a builder — which in a solo context means: clear enough to build in a separate session, or to hand off to a collaborator, without any follow-up questions required.
1
Design reference is unambiguous
The slice points at a specific screen file and a specific element within it. "The player card on sprint-001.html — the weekly performance row" is a design reference. "The player screen" is not. If you can't point at the exact pixel, the slice isn't Ready.
2
Done criteria are defined — 2 to 3 specific, verifiable statements
"It works" is not a criterion. "The weekly performance row renders the correct score for each week, including a zero for bye weeks, and the delta column shows positive/negative correctly" is a criterion. Each condition in "done looks like" should be something a person can check in a browser and give a yes/no answer to. Vague criteria produce failed QA and rework.
3
Dependencies are identified and not blocking
Every slice that must exist before this one is named. If a dependency isn't Done yet, the question is: does it block this slice from starting? If it does, the slice stays In Review until the dependency is in build or Done. If it doesn't block the start (only a specific later step), that's noted and the slice can be Ready.
4
No open spike on this slice
If there's a spike open on a question that affects what gets built, the slice is Blocked, not Ready. A closed spike result (in docs/spikes/) that has been incorporated into the slice definition is fine — the spike is closed. An active spike where the answer is still unknown means the slice cannot be Ready. Promote after the spike returns.
5
Process anchor is set
The slice is connected to a specific step in the to-be process map — or explicitly documented as infrastructure or background logic that doesn't map to a user-facing step. This is what allows the Plan phase to sequence slices by process order rather than just by dependency. A slice without a process anchor can't be Ready. Slices that don't map to any process step need an explicit decision: infrastructure, deferred feature, or scope creep.

Stakeholder feedback incorporation
When feedback files exist, they're first-class input to design review

If files exist at docs/stakeholder-feedback/*.md, design-review reads each one before the review round begins and surfaces it explicitly. The format for opening a round with stakeholder feedback: "Stakeholder feedback from [reviewer, date] is in. Overall reaction: [summary]. Approval status: [status]. Let's work through each per-screen comment."

Every per-screen comment becomes a finding, classified using the same taxonomy as any other design-review finding: design gap, knowledge gap, slice definition, deferral confirmation, or ready signal. Blockers listed by the stakeholder must be resolved before any affected slice can reach Ready — they function exactly like an open spike on that slice.

Approval status drives the next action:

Approved to build
Log the approval in the Review Log. If there are no per-screen changes, slices can proceed to Ready on their merits. Continue with the current round.
Needs another round
Work each per-screen comment as a finding. Apply changes. Schedule another handoff after this round's changes land. No slice that was flagged by the stakeholder reaches Ready until their feedback is addressed.
The stakeholder handoff package

The Design Sprint phase can produce a shareable handoff package at docs/design/handoff/<date>/ — an HTML artifact, PDF export, and a feedback template. When a stakeholder returns a completed feedback template, it lands in docs/stakeholder-feedback/ and becomes available to the next design-review round.

This is the loop: sprint produces the handoff package → stakeholder reviews and returns feedback → design-review reads the feedback as first-class input → slices are updated accordingly → another handoff cycle if needed. Stakeholder approval is a condition for a clean phase gate, not a nice-to-have.

Documents produced
Three persistent artifacts created and maintained during this phase
data/mock/[entity].json
Mock data layer
One JSON file per data entity, with a _mock: true header and a reference to docs/data-mapping.md. Contains domain-accurate values, enough records to show real list states, and at least one edge case per entity. The mock layer is a seam — UI code doesn't change when real data arrives, only the layer underneath it.
Created by data-scaffold. Updated when new entities are discovered during design review.
docs/data-mapping.md
Data mapping / proto-API contract
Every mock field mapped to its eventual real data source. Includes connection status per entity (Pending / Connected), field-level mapping table with mock example values and real field names, and the instructions for swapping to real data when each source is ready. This document is what makes the mock layer valuable beyond the immediate session.
Created by data-scaffold. Updated as real sources get connected or new fields are discovered.
docs/backlog.md
Backlog
Created Round 1, updated every round. At a Glance summary with counts by state. Full slice table with IDs, names, phases, and statuses. Slice detail per slice: design reference, description, process anchor, done criteria, dependencies, and notes. Review log appended each round. Readable by anyone who wasn't in the room.
Created Round 1 of design-review. Every subsequent round appends to — never recreates — this document.

The gate
What must be true before the Plan phase can begin
Phase Gate 2.6 → 3
Build cannot start without a backlog.
The gate for this phase is not "design is done" or "all slices are Ready." It's a specific document with a specific shape. Until this document exists and contains what's required, Plan cannot start and Build cannot start.
docs/backlog.md exists with a coherent set of Ready slices for Phase 1

"Coherent set" means the Ready slices form a meaningful starting point — they aren't random isolated slices, but enough of the system's core loop to build something that can actually be tested. The thinnest path through the full user journey, built as a tracer bullet, is the right mental model.

The process coverage check is part of gate readiness. Every step in the to-be process map must have a slice assigned to it, or an explicit documented decision that it's deferred, handled inside another slice, or outside scope. Silent gaps in process coverage are a gate hold condition.

The framework-health skill checks this gate signal at session start. If docs/backlog.md doesn't exist and the project is past the design sprint phase, it will surface that as the recovery path.

Handoff to Plan
What the Plan phase reads when it starts
Plan phase reads these three things when it starts
The backlog
docs/backlog.md — specifically the Ready slices for Phase 1. Each one has: a design reference, done criteria, dependencies, and a process anchor. These are the units the Plan phase sequences into a build order. The prd-to-plan skill reads the backlog and sequences by risk and process order, not by comfort or convenience.
The deferred decisions log
docs/design/deferred-decisions.md — the scope decisions made during the design sprint and carried through design review. These are explicit choices about what's in and what's out of Phase 1. The Plan phase uses this to sequence phases: Phase 1 is what's Ready now, later phases are what's Deferred with trigger conditions.
The design screens
docs/design/sprint-[id].html — the build reference. Every slice's design reference points into this file. The Plan phase confirms that the HTML screens are accessible and that each slice's design reference is resolvable before GitHub issues are created. The screens are what the builder uses during Build — the backlog references them, not descriptions of them.

What to watch for
Failure modes that are common in this phase
Anti-pattern Why it's a problem Instead
Waiting for ALL slices to be Ready before starting build Waterfall — the entire value of the iterative loop is lost. You end up doing all the work upfront that the process is designed to distribute over time. Build starts learning late, feedback arrives late. Build starts when the first coherent set of Ready slices exists. Design review continues in parallel on the remaining slices. The backlog makes both tracks visible simultaneously.
Vague done criteria — "it works," "it looks right," "it functions" Nobody can verify these. They're not criteria — they're intentions. The QA phase will fail because there's nothing specific to check against. Rework happens after the fact. 2–3 specific, verifiable statements per slice. Each one answerable with yes or no from a browser. If you can't write it as a checkbox a person can check, it's not ready.
Skipping the data lens Design implies data infrastructure that doesn't exist yet. Build starts, then discovers mid-build that a required API doesn't exist, a computed field has no source, or a data model doesn't support what the screen assumes. Every field on every screen gets a "where does this come from" pass. Unknown sources become spikes or data-scaffold entries. No field is assumed — every one is either mapped or flagged.
Skipping the process coverage check Slices pile up around screens and UI elements, but whole process steps go unbuilt. The system ships screens that look right but don't implement the actual business process. Phase test fails on use cases that should be basic. Run the coverage map every round — every to-be process step must have a slice, or an explicit decision that it's deferred, handled inside another slice, or out of scope. This check is not optional.
Letting blocked slices sit without a named spike The blocker becomes invisible. It shows up in the backlog as "Blocked" with no context. The next session can't determine what's blocking it or how to unblock it. Blockers accumulate without resolution paths. Every blocked slice names the specific spike question. "Blocked: spike on whether MFL API returns historical scores by player" is a blocked slice. "Blocked: data question" is not. Name it or it will never get resolved.
Defining slices without checking for existing patterns in the codebase Silent duplicates — two slices end up doing the same thing, neither done well. Or a new slice creates a naming collision with something the codebase already expresses. Build session is surprised by the divergence. Coherence check before every new slice definition. Is there already a pattern in the codebase for this? Are we extending or creating? If creating, name the divergence explicitly in the slice notes.
Running data-scaffold after design review instead of alongside it Design review happens against empty placeholders. The data lens can't evaluate whether the data model supports what the design assumes. Edge cases don't surface. The screens that come out of review aren't grounded in realistic data shape. Run data-scaffold before or during first design review round. The value of data-scaffold is precisely that design review has real-looking data to react to. Scaffold first, review against populated screens.
