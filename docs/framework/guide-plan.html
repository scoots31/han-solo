What This Phase Is

Planning converts the Ready backlog — slices that survived Design Review — into a sequenced, multi-phase implementation plan, then into executable GitHub issues. It's the bridge between "we know what to build" and "we know what to do today."

The defining principle of this phase: sequence by risk, not by comfort. The hardest, most uncertain work goes first. Not because it's the most exciting, but because it's the work that could invalidate everything else if it fails. If an assumption buried in Phase 3 turns out to be wrong, you've built two phases on a broken foundation. Planning forces that reckoning early.

The input is the Ready backlog plus the deferred decisions log from the Design Sprint. The deferred decisions log is critical — it contains both visible scope decisions (what was explicitly put out of Phase 1) and implied infrastructure decisions (backend work that must exist before certain screens can function). Both feed into sequencing.

Why the to-be process map is a prerequisite. prd-to-plan requires the to-be process map to exist before it starts. Without it, slices get sequenced by what's comfortable to build rather than by the process they implement. The process map is the anchor that connects slice order to user flow order — and it surfaces uncovered steps that would otherwise be discovered mid-build.

Skills in This Phase

prd-to-plan — Phase skill

Turns the Ready backlog and deferred decisions log into a sequenced, multi-phase implementation plan. Maps each slice to a step in the to-be process map, identifies integration seams, designs tracer-bullet vertical slices, and sequences by risk order — with explicit conflict resolution when risk order and process order diverge.

Reads
  • docs/backlog.md — Ready slices + dependencies
  • docs/process/to-be-*.md — agreed process maps
  • docs/design/deferred-decisions.md — scope + infrastructure decisions
  • docs/tech-context.md — stack + infrastructure slices
Produces
  • Multi-phase implementation plan with slice-to-process-step mapping
  • Process anchors on every Ready slice in the backlog
  • Flagged uncovered process steps requiring resolution
  • Open questions list for pre-build decisions
Each slice gets
  • Design anchor — screen + element from design sprint
  • Data anchor — mock data fields it reads
  • Done anchor — 2–3 verifiable criteria
  • Process anchor — which to-be step it implements

grill-me — On-demand skill

Stress-tests a plan by relentlessly interviewing the solo through every branch of the design tree, one question at a time. Can run before prd-to-plan to validate the approach before committing to it, or during planning when assumptions surface. Each question comes with Claude's recommended answer.

When to use
  • Before planning when the approach has significant unknowns
  • When the deferred decisions log has a lot of open questions
  • When Phase 1 touches an integration seam you haven't built through before
  • Any time you feel uncertain but can't name exactly why
What it surfaces
  • Assumptions that are load-bearing but haven't been stated
  • Decision dependencies — question B can't be answered until question A is resolved
  • Scope drift — things that feel small but imply larger work
  • Sequencing risks — work that appears parallel but actually isn't
How it runs
  • One question at a time — no batching
  • Explores codebase if a question can be answered empirically
  • Provides recommended answer with each question
  • Walks the full decision tree before stopping

to-issues — Phase skill

Converts the approved phase plan into self-contained GitHub issues in dependency order — blockers first. Each issue contains everything a builder needs to start the slice: design reference, process anchor, data anchor, done criteria, and dependency links. Creates labels by phase and status. Does not modify the plan — it converts, not designs.

Prerequisites
  • Plan explicitly approved (not still under discussion)
  • GitHub repo exists + gh CLI authenticated
  • Backlog has process anchors on every Ready slice
Issue structure
  • Title: [SL-XXX] Slice Name
  • What this builds — one sentence from backlog
  • Design reference — screen + specific element
  • Process anchor — to-be map step
  • Data anchor — mock data fields
  • Done looks like — acceptance criteria
  • Depends on — issue number link
Does NOT do
  • Create issues for Deferred slices
  • Modify or redesign the plan
  • Set assignees or milestones
  • Create infrastructure slices not already in the backlog

to-prd — On-demand skill

Synthesizes the current conversation context — discovery notes, design sprint output, research findings — into a structured PRD and submits it as a GitHub issue. Use this when the context exists in conversation but hasn't been formalized into a document that prd-to-plan can read. Most Plan sessions won't need this — it's for cases where you're entering planning with context that never got written down.

Sequencing by Risk, Not Comfort

The natural instinct when sequencing work is to start with what you know how to build. Confidence is mistaken for priority. The framework inverts this. Risk order wins for infrastructure and foundation slices. The first slice should prove the riskiest assumption — the thing that, if it turns out to be wrong, makes other work irrelevant or forces a rebuild.

Risk levelType of workWhy it goes first
High Technical unknowns at integration seams API boundaries, auth, external services — assumptions made early are painful to undo. A screen that looks simple may require a data pipeline that doesn't exist yet.
High External dependencies Third-party APIs with unknown rate limits, data contracts with other teams, infrastructure not yet stood up. These can't be parallel-worked around — they block everything downstream.
Medium Work that validates other work Slices whose outcome determines whether subsequent slices are even the right thing to build. Getting the data model wrong in Phase 1 means retrofitting every slice that reads it.
Lower UI and presentation work Once the data layer and core logic are proven, UI work is comparatively low risk. It can be wrong and corrected without structural consequences.

The tracer bullet

The first slice of Phase 1 is the tracer bullet — the thinnest complete path through the full system. It touches every integration layer: data, business logic, API, UI. It delivers something demoable. And it goes first not because it's the easiest, but because it proves that the core loop actually works end to end.

If the tracer bullet reveals a problem — a data model that won't support the use case, an API that doesn't behave as expected, an infrastructure gap — it reveals it before you've built anything on top of those assumptions. The tracer bullet is an early warning system, not a warmup exercise.

When risk order and process order conflict: prd-to-plan resolves this explicitly. Risk order wins for infrastructure and foundation slices. Process order wins for feature slices once infrastructure is in place. The skill names the conflict when it occurs rather than silently resolving it — so the sequencing decision is visible and intentional.

Step-by-Step Experience

1. Read the backlog and deferred decisions log
2. Map every Ready slice to a process step
3. Identify integration seams and flag high-risk ones
4. Optionally run grill-me
5. Sequence phases, design tracer-bullet slices
6. Present, discuss, and get explicit approval
7. Run to-issues to create GitHub issues in dependency order

Step 1. prd-to-plan reads the backlog, the to-be process map, the deferred decisions log, and docs/tech-context.md. If the to-be process map doesn't exist, it stops and asks — that document is the sequencing foundation.

Step 2. Each slice gets a process anchor: which step in the to-be map it implements, and its position in the flow (main path, branch, or exception). Uncovered steps — process steps with no slice assigned — are surfaced explicitly before sequencing continues.

Step 3. Every layer the slices cut through is listed: data model, business logic, API endpoints, UI, external integrations, infrastructure. High-risk seams are flagged explicitly.

Step 4. If the plan has significant assumptions, or if this is the first time building through a particular integration seam, grill-me runs here. Not required — but strongly recommended when Phase 1 has unknowns.

Step 5. prd-to-plan produces a multi-phase plan. Each phase shows its slices with process anchors, what process steps it completes, what it proves or de-risks, what blocks it, and its definition of done.

Step 6. The full plan is shown. Questions asked: Does the phase sequencing respect both risk and process order? Are there slices to merge, split, or reorder? Are uncovered steps accounted for? The plan iterates until explicitly approved. Process anchors are written to the backlog before proceeding.

Step 7. to-issues converts the approved plan into GitHub issues — blockers first, so dependent issue numbers can be referenced in subsequent issues. Labels are applied: Phase 1/2/etc., Ready, Infrastructure, Blocked. Build starts from these issues.

What grill-me Does Here

In a real product team, before engineering commits to a plan, a PM or product lead asks hard questions. Not to be difficult — to catch the things that look obvious until someone interrogates them. grill-me is that role. The questions go after load-bearing assumptions — the things you've been taking for granted because you've been close to the problem.

Questions it asks in Plan context
  • Is this solving the right problem — or is the PRD scope drifted from the original need?
  • What assumptions are baked into Phase 1 sequencing?
  • What could fail in Phase 1 that would require rearchitecting Phase 2?
  • Which integration seam is least understood, and is it Phase 1 or Phase 2?
  • What deferred decisions, if reversed, would invalidate the plan structure?
  • Are there slices that look thin but imply substantial backend work not yet captured?
When to skip it
  • You've built this exact stack before and there are no new integration seams
  • The plan has zero open questions and no external dependencies
  • This is a small, contained feature with no architectural implications
When it's essential
  • Phase 1 touches an external API or service for the first time
  • The deferred decisions log has more than a few open items
  • The plan feels right but you can't fully explain why
  • Multiple plausible Phase 1 sequences exist and you're not sure which is right

The Gate

Gate: GitHub issues created for Phase 1 Ready slices.

Build cannot start without issues to work from. The gate is not "the plan is approved" — approved plans can sit in documents indefinitely. The gate is the moment the plan becomes a tracked set of issues in GitHub: concrete, assigned to a phase, labeled, and self-contained enough for Build to start.

Required: Phase 1 slices have issues in dependency order · Each issue has a design reference · Each issue has a process anchor · Each issue has verifiable done criteria · Process anchors are written to the backlog.

Handoff to Build

What to build

GitHub issues

The Phase 1 issues are the build queue. solo-build selects the next slice from the open Phase 1 issues, reads the issue, and starts. No other document is required to begin a slice — the issue is self-contained.

What it looks like

Design screens

Each issue references a specific screen and element in the design sprint artifact. Build reads the design reference to understand the visual target. The HTML artifact is the source of truth — not a translation of it.

Status tracking

Backlog

The backlog tracks slice status across phases. As Phase 1 issues close, slice status updates. The backlog shows overall progress and what's in queue for Phase 2. Build doesn't read it for instructions — it reads it for context.

What to Watch For

PatternWhy it happensCorrection
Comfort-first sequencing The familiar slice feels tractable; the uncertain slice feels big. Starting with what you know is psychologically easier. Name the riskiest assumption explicitly. If the first slice doesn't test it, ask why not. prd-to-plan is designed to flag this — don't override it without a stated reason.
Issues without design references The slice seems obvious — everyone knows what it should look like. The design reference step feels redundant. Every issue must reference a specific screen and element from the design sprint artifact. "Obvious" slices are where design drift is most common — precisely because no one double-checks them.
Skipping grill-me on assumption-heavy plans Planning took effort to produce. Subjecting it to scrutiny feels like backtracking. Count the assumptions in the plan. If the deferred decisions log has more than a handful of open items, or if Phase 1 touches an integration seam you haven't proven, run grill-me first.
Plan approved but process anchors not written to backlog The plan was discussed and approved in conversation. Writing it back to the backlog feels like admin work. prd-to-plan explicitly requires this. Process anchors must be in the backlog before Build starts — not just in the plan document.
Infrastructure slices sequenced after feature slices Infrastructure work is invisible in the UI. Easy to delay in favor of screens that demonstrate visible progress. Infrastructure slices always sequence before feature slices, regardless of process order. This is a hard rule in prd-to-plan.
Slices approved without an Architecture type The field is new. Easy to treat it as optional on slices that feel obviously self-contained. Architecture type is required for every slice before the plan is approved. Two values only: Leaf node (nothing else depends on it) or Core architecture (touches data models, shared logic, or infrastructure).
