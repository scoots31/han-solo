What this phase is
Where the Ready backlog becomes a build plan

Planning converts the Ready backlog — slices that survived Design Review — into a sequenced, multi-phase implementation plan, then into executable GitHub issues. It's the bridge between "we know what to build" and "we know what to do today."

The defining principle of this phase: sequence by risk, not by comfort. The hardest, most uncertain work goes first. Not because it's the most exciting, but because it's the work that could invalidate everything else if it fails. If an assumption buried in Phase 3 turns out to be wrong, you've built two phases on a broken foundation. Planning forces that reckoning early.

The input is the Ready backlog plus the deferred decisions log from the Design Sprint. The deferred decisions log is critical — it contains both visible scope decisions (what was explicitly put out of Phase 1) and implied infrastructure decisions (backend work that must exist before certain screens can function). Both feed into sequencing.

grill-me can run before planning to stress-test the approach. It's optional but valuable when the plan involves significant assumptions. to-issues runs after the plan is approved to convert it into GitHub issues in dependency order. Build cannot start without those issues.

⚡

Why the to-be process map is a prerequisite

prd-to-plan requires the to-be process map to exist before it starts. Without it, slices get sequenced by what's comfortable to build rather than by the process they implement. The process map is the anchor that connects slice order to user flow order — and it surfaces uncovered steps that would otherwise be discovered mid-build.

Sequencing by risk, not comfort
The core discipline of this phase

The natural instinct when sequencing work is to start with what you know how to build. There's a familiar UI pattern, a clear data model, an established pattern from the last project. That slice goes first because it feels tractable. Confidence is mistaken for priority.

The framework inverts this. Risk order wins for infrastructure and foundation slices. The first slice should prove the riskiest assumption — the thing that, if it turns out to be wrong, makes other work irrelevant or forces a rebuild. Comfortable work can wait. Uncertain work cannot.

What counts as risk

High
Technical unknowns at integration seams

API boundaries, auth, external services, data model changes — places where assumptions made early are painful to undo later. A screen that looks simple may require a data pipeline that doesn't exist yet.

High
External dependencies

Third-party APIs with unknown rate limits, data contracts with other teams, infrastructure that hasn't been stood up yet. These can't be parallel-worked around — they block everything downstream.

Medium
Work that validates other work

Slices whose outcome determines whether subsequent slices are even the right thing to build. Getting the data model wrong in Phase 1 means retrofitting every slice that reads it.

Lower
UI and presentation work

Once the data layer and core logic are proven, UI work is comparatively low risk. It can be wrong and corrected without structural consequences.

The tracer bullet

The first slice of Phase 1 is the tracer bullet — the thinnest complete path through the full system. It touches every integration layer: data, business logic, API, UI. It delivers something demoable. And it goes first not because it's the easiest, but because it proves that the core loop actually works end to end.

If the tracer bullet reveals a problem — a data model that won't support the use case, an API that doesn't behave as expected, an infrastructure gap — it reveals it before you've built anything on top of those assumptions. That's the point. The tracer bullet is an early warning system, not a warmup exercise.

⚠

When risk order and process order conflict

prd-to-plan resolves this conflict explicitly. Risk order wins for infrastructure and foundation slices. Process order wins for feature slices once infrastructure is in place. The skill names the conflict when it occurs rather than silently resolving it — so the sequencing decision is visible and intentional.

Step-by-step experience
How planning actually runs from start to GitHub issues
1

Read the backlog and deferred decisions log

prd-to-plan reads docs/backlog.md, the to-be process map, docs/design/deferred-decisions.md, and docs/tech-context.md. If the to-be process map doesn't exist, it stops and asks — that document is the sequencing foundation. If this is not a greenfield project, it also explores the codebase to understand existing integration seams.

2

Map every Ready slice to a process step

Each slice gets a process anchor: which step in the to-be map it implements, and its position in the flow (main path, branch, or exception). Uncovered steps — process steps with no slice assigned — are surfaced explicitly before sequencing continues. These either need a new slice, are intentionally deferred, or are handled in background logic.

3

Identify integration seams and flag high-risk ones

Every layer the slices cut through is listed: data model, business logic, API endpoints, UI, external integrations, infrastructure. High-risk seams — places where early assumptions are painful to undo — are flagged explicitly.

4

Optionally run grill-me

If the plan has significant assumptions, or if this is the first time building through a particular integration seam, grill-me runs here. It stress-tests the approach before the plan is committed. Not required — but valuable when the stakes of a wrong sequence are high.

Optional — but strongly recommended when Phase 1 has unknowns
5

Sequence phases, design tracer-bullet slices

prd-to-plan produces a multi-phase plan. Each phase shows its slices with process anchors, what process steps it completes, what it proves or de-risks, what blocks it, and its definition of done. The first slice in Phase 1 is the tracer bullet. Open questions that must be resolved before build starts are listed.

6

Present, discuss, and get explicit approval

The full plan is shown. Questions asked: Does the phase sequencing respect both risk and process order? Are there slices to merge, split, or reorder? Are uncovered steps accounted for? Are open questions blockers or stated assumptions? The plan iterates until explicitly approved. Process anchors are written to the backlog before proceeding.

7

Run to-issues to create GitHub issues in dependency order

to-issues converts the approved plan into GitHub issues — blockers first, so dependent issue numbers can be referenced in subsequent issues. Labels are applied: Phase 1/2/etc., Ready, Infrastructure, Blocked. Build starts from these issues, not from the plan document.

🔍
What grill-me does here
The skeptical PM role before you commit

In a real product team, before engineering commits to a plan, a PM or product lead asks hard questions. Not to be difficult — to catch the things that look obvious until someone interrogates them. grill-me is that role. It's the skeptical PM who hasn't been steeped in the context you have and is seeing the plan somewhat fresh.

The questions aren't procedural. They go after the load-bearing assumptions — the things you've been taking for granted because you've been close to the problem. "Is this solving the right problem?" "What are we assuming about how users will behave here?" "If the API doesn't support this use case, what's Plan B?" "What could cause Phase 1 to fail in a way that invalidates Phase 2?"

Questions it asks in the Plan context

  • Is this solving the right problem — or is the PRD scope drifted from the original need?
  • What assumptions are baked into Phase 1 sequencing?
  • What could fail in Phase 1 that would require rearchitecting Phase 2?
  • Which integration seam is least understood, and is it Phase 1 or Phase 2?
  • What deferred decisions, if reversed, would invalidate the plan structure?
  • Are there slices that look thin but imply substantial backend work not yet captured?

When to skip it

  • You've built this exact stack before and there are no new integration seams
  • The plan has zero open questions and no external dependencies
  • This is a small, contained feature with no architectural implications

When it's essential

  • Phase 1 touches an external API or service for the first time
  • The deferred decisions log has more than a few open items
  • The plan feels right but you can't fully explain why
  • Multiple plausible Phase 1 sequences exist and you're not sure which is right

The gate
What must be true before Build starts
🔑

GitHub issues created for Phase 1 Ready slices

Build cannot start without issues to work from. The gate is not "the plan is approved" — approved plans can sit in documents indefinitely. The gate is the moment the plan becomes a tracked set of issues in GitHub: concrete, assigned to a phase, labeled, and self-contained enough for Build to start.

Phase 1 slices have GitHub issues created in dependency order
✓
Each issue contains a design reference to the relevant screen and element
✓
Each issue contains the process anchor — which to-be step it implements
✓
Each issue has verifiable done criteria (not descriptions of effort)
✓
Process anchors are written to the backlog on every Ready slice

Handoff to Build
What Build reads and where it comes from
What to build

GitHub issues

The Phase 1 issues are the build queue. solo-build selects the next slice from the open Phase 1 issues, reads the issue, and starts. No other document is required to begin a slice — the issue is self-contained.

gh issue list --label "Phase 1"
What it looks like

Design screens

Each issue references a specific screen and element in the design sprint artifact. Build reads the design reference to understand the visual target. The HTML artifact is the source of truth — not a translation of it.

docs/design/sprint-v1.html
Status tracking

Backlog

The backlog tracks slice status across phases. As Phase 1 issues close, slice status updates. The backlog shows overall progress and what's in queue for Phase 2. Build doesn't read it for instructions — it reads it for context.

docs/backlog.md
✅

The build start prompt

When Build starts, the solo runs /solo-build. The skill reads the Phase 1 issue queue, selects the next issue (usually the tracer bullet first), reads the design reference and process anchor from the issue, and begins implementation. The issues created in this phase are the exact input that skill expects.

What to watch for
Common failure modes in this phase
Pattern Why it happens The consequence Correction
Comfort-first sequencing The familiar slice feels tractable; the uncertain slice feels big. Starting with what you know is psychologically easier. Phase 1 completes successfully but Phase 2 hits a fundamental problem that retroactively invalidates Phase 1 work. Name the riskiest assumption explicitly. If the first slice doesn't test it, ask why not. prd-to-plan is designed to flag this — don't override it without a stated reason.
Issues without design references The slice seems obvious — everyone knows what it should look like. The design reference step feels redundant. Build interprets the issue differently than Design Sprint intended. Scope creep, revisit cycles, and inconsistent visual language across slices. Every issue must reference a specific screen and element from the design sprint artifact. "Obvious" slices are where design drift is most common — precisely because no one double-checks them.
Skipping grill-me on assumption-heavy plans Planning took effort to produce. Subjecting it to scrutiny feels like backtracking. The plan looks good on paper. Assumptions that look reasonable in the plan turn out to be wrong in the build. Discovered late, after other work has built on top of them. Count the assumptions in the plan. If the deferred decisions log has more than a handful of open items, or if Phase 1 touches an integration seam you haven't proven, run grill-me first.
Plan approved but process anchors not written to backlog The plan was discussed and approved in conversation. Writing it back to the backlog feels like admin work. solo-build reads process anchors from the backlog before starting a slice. If they're missing, the connection between the issue and the agreed process is lost mid-build. prd-to-plan explicitly requires this. Process anchors must be in the backlog before Build starts — not just in the plan document.
Infrastructure slices sequenced after feature slices Infrastructure work is invisible in the UI. It's easy to delay in favor of screens that demonstrate visible progress. Feature slices start building on unproven infrastructure. When infrastructure work surfaces problems, feature work built on top has to be retrofitted or rewritten. Infrastructure slices always sequence before feature slices, regardless of process order. This is a hard rule in prd-to-plan — read docs/tech-context.md infrastructure slices first and place them at the front of the queue.
Slices approved without an Architecture type The field is new. It's easy to treat it as optional — especially on slices that feel obviously self-contained. An unclassified slice cannot reach Ready. If the plan is approved and slices land in the backlog without a type, build will stall the first time one of them tries to open a branch. Architecture type is required for every slice before the plan is approved. The quality scan in planning blocks approval if any slice is missing it. Two values only: Leaf node (nothing else depends on it) or Core architecture (touches data models, shared logic, or infrastructure). If you're unsure, err toward Core architecture — the PE review is low-cost and catches structural problems before they compound.
