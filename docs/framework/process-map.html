Framework Skills
Active phase execution
start
  • Reads opening message
  • Routes to Brainstorm (exploratory) or Discover (clear idea)
  • One question if ambiguous
  • Deployed project detected → new cycle protocol, new phase in existing backlog
brainstorming
  • Conversational exploration — not a checklist
  • Process sketch moment: captures preliminary as-is / to-be
  • Visual sketch mid-session as a thinking tool
  • Routes to Discover, Research, or more brainstorming
discover
  • Narrative conversation — beginning to end use case
  • Produces as-is map (how it works today, including manual processes)
  • Produces and agrees to-be map — the process contract
  • Produces discovery brief
tech-context
  • Establishes stack, constraints, infrastructure
  • Documents deploy method, branching model, CI/CD
  • Identifies infrastructure slices — prerequisites for all feature slices
design-sprint
  • On-ramp A: scratch → hero screen → warmer/colder → all screens
  • On-ramp B: Figma import via MCP
  • On-ramp C: reference-based aesthetic constraint
  • Walk-through conversation surfaces build/defer decisions
  • Produces deferred decisions log
data-scaffold
design-review
  • data-scaffold: generates realistic mock data · flags list entities as data behavior candidates · updates mock after data questions resolve
  • design-review: PM + Tech Lead + Data lenses applied every round
  • Data behavior pass — volume, list behavior, nulls, empty/error/loading states asked for every screen with external data → produces data questions log
  • Unresolved data question blocks dependent UI slices from Ready
  • Coverage check: every to-be step mapped to a slice or decided
  • Slice Ready requires: design ref, done criteria, process anchor, no open spikes, no blocking data questions
prd-to-plan
to-issues
  • Maps every Ready slice to its to-be process step
  • Sequences by risk order AND process order — conflicts named explicitly
  • Infrastructure slices always first
  • Defines deliverables: groups slices by shared outcome, assigns type (Screen/Logic), writes three-field record (technical spec · solo description · acceptance criteria)
  • Integration deliverables — Screen deliverable with external data gets a companion Logic deliverable (Data Integration); sequences after Screen is Accepted; requires data questions log resolved before scoping
  • Deliverables presented alongside plan — both approved together before build starts
  • to-issues: creates GitHub issues in dependency order
solo-build
autopilot
  • Two modes — solo chooses: solo-build (solo at every gate) · autopilot (simulator at every gate, solo reviews at phase test)
  • Creates feature branch on slice start
  • Requires four anchors: design · data · done · process
  • Journey order — first screen before second
  • Mid-build discoveries surfaced immediately
  • Commits on code-complete → auto-invokes code-review-and-quality
  • When all slices in a deliverable are Done → auto-triggers deliverable acceptance
  • Autopilot only: pre-flight check surfaces judgment gaps before build begins · deliverable boundary writes enforced automatically · Refinement cycles after phase test for implementation delta
code-review-and-quality
solo-qa
qa-triage
process-change
  • 7-check gate: pattern, data sourcing, expandability, cleanliness, scope, docs, stack
  • Pass → logs confirmation → auto-invokes solo-qa
  • solo-qa Part 1: active testing — renders screen, cross-checks values against mock data
  • solo-qa Part 2: structured solo browser sign-off
  • Unexpected findings → qa-triage: bug / missing req / regression / map-level gap
  • Map-level gap (no to-be step for this behavior) → qa-triage auto-invokes process-change
  • When all slices in a deliverable are Done → auto-runs deliverable acceptance: verifies all acceptance criteria against the full running experience, prompts solo sign-off on the build contract
  • Integration deliverable acceptance — UX impact check: does live data produce anything the design didn't account for? If yes → flag for design review, affected UI slices reopen
phase-test
  • Stage 1: Environment readiness (pre-flight gate)
  • Stage 2: Use case creator — test plan from discovery brief + design
  • Stage 3: Data specialist — mock→real swap, API read/write verified
  • Stage 4+5: Tester + Regression specialist
  • Stage 6: Acceptance reviewer — PM lens vs. discovery brief; process-level mismatch → process-change before gate
  • Stage 7: Gate — OPEN or HOLD with specific items
deploy
  • Reads tech-context — method and deployment path determined by stack
  • CI/CD Setup: one-time — generates GitHub Actions workflow, walks solo through platform token setup in plain language
  • Confirms gate OPEN + all branches merged
  • Method A: CI/CD on merge · B: MCP tool · C: Manual CLI · D: Define first
  • Verifies environment actually live — not just pipeline green

Reading this map: Time flows left to right. Each column is a phase. Each row is a lane — who's doing what. The Builder lane shows what the solo contributes at each phase. The Framework Skills lane shows which skills are active and what they execute. The Always-On lane shows the four skills that activate when guided or piloted mode is chosen — silent in bare mode (the default). The Outputs lane shows what each phase produces and where gates exist.

Gates are hard stops — the framework will not advance past them without the required condition being met. Auto-triggered phases (QA Chain) are invoked by the framework on completion of the prior phase — the solo does not need to invoke them.
