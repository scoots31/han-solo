Solo Builder Framework

Backlog Status Reference

Every status for every object in the backlog — what it means, who sets it, and how it flows. Three objects, three status chains, one lifecycle.

← All docs Skills Reference Build Guide QA Guide Phase Test Guide Status Reference
The backlog tracks three types of objects: Slices (individual units of work), Deliverables (groups of slices that produce one complete output), and Phases (groups of deliverables that prove one meaningful capability). Each has its own status chain. A status on one object does not imply a status on another — they are verified and advanced independently.
Slice
8 statuses
The atomic unit of build work. Each slice is one specific element or behavior — fully defined with four anchors before build starts. Slices are verified individually by solo-qa before contributing to a deliverable.
In Review → Ready → In Build → In QA → In Test → Done · branches: Blocked · Deferred
In Review
Set by: design-review
Identified but not fully defined yet. Needs more work before it can be built — a missing anchor, an open question, or a dependency that hasn't resolved.
Blocked
Set by: solo-build
Waiting on a named dependency. Always named explicitly — not "backend" but a specific slice ID, deliverable, or external dependency.
Deferred
Set by: solo-build / curator
Explicitly set aside with a trigger condition noted. Not abandoned — it has a named condition that would bring it back into scope.
Ready
Set by: design-review
Fully defined. All four anchors set, done criteria written, self-verification checklist defined, dependencies resolved. Build can start immediately.
In Build
Set by: solo-build
Work is active on this slice. The builder is implementing it now. Only one slice should be In Build at a time per active thread.
In QA
Set by: solo-build at code-complete
Code complete. solo-qa is running — AI verification against done criteria and design fidelity, then solo sign-off in the browser. Not done until both pass.
In Test
Set by: solo-qa after solo sign-off
QA passed. The slice is built correctly and the solo has confirmed it in the browser. Waiting on phase-test to verify it as part of the complete phase.
Done
Set by: phase-test at gate-open
Built, QA verified, and phase-tested end-to-end. This is the terminal state. Done is permanent unless a later slice reveals a regression.
Deliverable
4 statuses
A group of slices that together produce one complete output — a screen, a piece of logic, or a data integration. Deliverables are accepted by the solo after all their slices are In Test and the full flow is verified together. A deliverable is not accepted by the sum of its slice approvals — it requires its own verification pass.
Defined → Active → Pending Acceptance → Accepted
Defined
Set by: prd-to-plan
Record created with all fields complete. Plan approved. Build has not started yet — no slices in this deliverable are In Build.
Active
Set by: solo-build
At least one slice in this deliverable is In Build. Work is underway. Other slices may still be Ready or In QA.
Pending Acceptance
Set by: solo-qa
All slices are In Test. solo-qa has verified the complete flow and is presenting the deliverable to the solo for acceptance sign-off.
Accepted
Set by: solo sign-off (via solo-qa)
Solo signed off on the complete deliverable experience — not just individual slices. This is the terminal state for deliverables.
Type What it produces How it's verified
Screen Visible UI built against mock data Visually in the browser — does it look and behave correctly?
Logic Invisible background behavior Evidence: test results, data state, simulated flows through affected screens
Integration Real data connected to a Screen deliverable Both visual (does it look right with live data?) and evidence (are fields coming from the real source?). Always sequences after its Screen companion is Accepted.
Phase
4 statuses
A group of deliverables that together prove one meaningful capability. A phase is not a time box, a sprint, or a feature list. Every phase answers exactly one question — if that question can't be stated in one sentence, the phase boundary is wrong.
Planning → Ready for Build → In Progress → Completed
Planning
Set by: prd-to-plan
Phase is being defined. Deliverables and slices are being sequenced. This phase has not been approved yet — it is still taking shape.
Ready for Build
Set by: prd-to-plan on approval
Plan approved by the solo. All deliverables are Defined. All slices are Ready. Build can start on the first deliverable.
In Progress
Set by: solo-build
At least one deliverable is Active. Work is underway in this phase. Other deliverables may be Defined and waiting.
Completed
Set by: phase-test at gate-open
All deliverables Accepted. Phase test gate open. All slices marked Done. The capability this phase was built to prove has been confirmed end-to-end.
Who sets what — and when
Each skill owns specific status transitions. No skill advances a status it doesn't own.
Skill Status transitions it owns
design-review Births slices at In Review · advances to Ready when all anchors are set
prd-to-plan Sets all deliverables to Defined · sets all phases to Planning → Ready for Build on approval
solo-build Slice: Ready → In Build → In QA · Deliverable: Defined → Active · Phase: Ready for Build → In Progress · also sets Blocked and Deferred
solo-qa Slice: In QA → In Test · Deliverable: Active → Pending Acceptance → Accepted
phase-test Slice: In Test → Done (all slices in phase) · Phase: In Progress → Completed
