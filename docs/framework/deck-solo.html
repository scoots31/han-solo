Solo Builder Framework

Team Discipline.
Without the Team.

A complete lifecycle framework for building software solo with AI — from first idea through deployed and tested.

The Problem

Solo builders using AI drift toward waterfall.

No external forcing function — no standup, no sprint review, no teammate waiting. AI defaults to building what you describe, not challenging whether you're ready to build it.

What disappears
  • No shared design artifact before code
  • Scope and data gaps surface in code, not design
  • Quality gates disappear
  • "Phase complete" replaces "enough to learn from"
Getting Started

Three ways in. The framework routes you.

Entry 1
You know what you want

You have a clear picture of the product, you've thought through the problem, and you're ready to design it. Start with Discover to validate assumptions before building.

Signal: "I want to build X for Y users"
Entry 2
You're working through an idea

You have a direction but the shape isn't clear yet. Brainstorm explores what you're actually trying to solve and whether this is the right thing to build.

Signal: "I've been thinking about something that..."
Entry 3
You have an existing project

A project is already running but isn't inside the framework. Onboard reads the project directory, matches content to framework artifacts, and places it at the right phase.

Signal: "I already have code — let's bring it in"

How Much You Want the Framework to Carry

Three session modes.

Bare
Default. Nothing loads until you ask. Skills activate only when you invoke a phase explicitly. Token efficient — nothing extra running in the background.
Zero overhead
Piloted
You're flying it. Always-on skills load once at session start. You choose each phase and invoke it manually — the framework follows your lead.
You lead, framework supports
Guided
Framework leads. Always-on plus the full phase chain. You're consulted at every decision point — gates, direction, scope. Framework does the routing.
Framework leads, you decide
Enhanced mode — Claude Code only. Multi-agent orchestration for supported skills. Parallel workstreams, automated handoffs between agents.
Active in Guided + Piloted Mode

Four skills that never turn off.

Fire once at session start. Silent in bare mode. They hold the project's memory, process accountability, and health checks — running in the background so nothing slips through the cracks.

  • process-mapper
    As-is and to-be process maps, coverage checks throughout the lifecycle
  • product-continuity
    Decisions, assumptions, risks, and session memory — the project's institutional record
  • framework-health
    Background monitor — flags one issue at a time, doesn't pile on
  • retrospective
    Flags in the moment, processes at phase end, feeds learning back into the playbook
Before Code Starts

A shared HTML artifact — all screens, interactive, good enough to decide from.

01 — Data Questions
Every field gets an answer

Where does this data come from? Is it user input, an API, a calculation, or static? Every field in the design has a source before a line of code is written.

02 — Process Gaps
Missing steps surface early

When you see the full flow in a screen, missing steps become obvious. Gaps that would have been found in code get caught in the artifact — when they're still cheap to fix.

03 — Scope Decisions
Every element is explicit

Every button, filter, field, and state becomes a build-or-defer decision. Nothing sneaks into the build that wasn't deliberately included in the artifact.

04 — Implied Infrastructure
Surfaces hidden complexity

A simple toggle might hide a background job, a caching layer, or a new data relationship. The artifact makes that visible before it becomes a problem in the code.

Three on-ramps: scratch (warmer/colder loop) · Figma import · reference-based aesthetic

No Slice Starts Without Four Anchors

Four anchors required before any slice enters build.

Design Anchor

Exact screen and element from the design artifact. Not a description — a specific reference the build can be held to.

Missing: building the wrong thing
Data Anchor

Every field in the slice mapped to its source. API, user input, calculation, or stored value — defined before code starts.

Missing: hardcoded data in code
Done Anchor

Two to three verifiable done criteria. Not "it works" — specific, observable conditions checkable in the browser.

Missing: no way to know when done
Process Anchor

The to-be map step this slice implements. Every unit of work maps back to the agreed process contract.

Missing: framework stops and asks

The Backbone

The to-be map is the contract every phase is held to.

Not just a discovery artifact. It's the thread that runs from the first process conversation through the final Phase Test — referenced at every gate in between.

  • 1
    Discover — as-is and to-be maps produced and agreed with the solo
  • 2
    Design Sprint — every screen cross-referenced against the to-be map
  • 3
    Design Review — coverage check every round, gaps flagged before build
  • 4
    Build — process anchor required for every slice, no exceptions
  • 5
    Phase Test — use cases derived directly from the to-be map steps
Phase 5 — You Invoke It Deliberately

Seven specialist lenses before anything ships.

  • 1
    Environment Readiness
    Validates build environment, dependencies, and configuration before any testing begins
  • 2
    Use Case Creator
    Derives test cases from the to-be process map — coverage is process-driven, not assumption-driven
  • 3
    Data Specialist
    Verifies data integrity, edge cases, and boundary conditions across all data inputs and outputs
  • 4–5
    Tester + Regression Specialist
    Executes use cases and regression suite — surfaces any failures or regressions against prior-phase behavior
  • 6
    Acceptance Reviewer
    Reviews all findings against acceptance criteria — determines whether each condition is met
  • 7
    Gate Decision
    OPEN (ready to deploy) or HOLD (blocking issues documented) — explicit, not implied
Factual Differentiation

What makes this different.

  • 01
    Not a prompt collection
    28+ interconnected skills that chain automatically — each activates at the right phase and hands off to the next without manual routing.
  • 02
    Process contract
    The to-be map is agreed before design starts and held through Phase Test — every gate references it.
  • 03
    Design before code
    HTML artifact reviewed and approved before any slice enters build — scope, data, and process questions answered first.
  • 04
    Quality with evidence
    Solo QA cross-references rendered values against mock data — "it looks right" is not the bar.
  • 05
    Self-improving
    Retrospective flags real friction in real usage and feeds improvements back into the playbook — not a theoretical framework.
