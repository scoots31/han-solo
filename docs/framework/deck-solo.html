Solo Builder Framework

Team discipline.
Without the team.

A complete lifecycle framework for building software solo with AI — from first idea through deployed and tested.

The Problem

Solo builders using AI drift toward waterfall

  • ✕
    No external forcing function — no standup, no sprint review, no teammate waiting. Each phase gets completed before the next starts.
  • ✕
    AI defaults to building what you describe — not challenging whether you're ready to build it. The PM challenge disappears.
  • ✕
    No shared artifact before code starts — scope, data gaps, and flow issues surface in code, not in design. Expensive to fix.
  • ✕
    Quality gates disappear — without a team to push back, "it works" becomes the bar. What ships is inconsistent.

The agile reframe: replace "phase complete" with "enough to learn from." The framework provides the forcing functions a team would provide.

Where You Start

Three entry points — the framework routes for you

Entry 1
You know what you want to build
The idea is clear. Who it's for, what it does, what value it delivers. Skip brainstorming — go straight to Discover.
Signals: specific solution in mind, clear user, defined outcome
Entry 2
You're working through an idea
The problem is real but the solution isn't clear yet. Brainstorming gives the idea something concrete to react to — including a rough visual sketch mid-session.
Signals: exploratory tone, "what if", uncertain scope or user
Entry 3
You have an existing project
Code already exists — you want SBF discipline applied to it. Onboard reads the project, maps content to gate artifacts, surfaces gaps, and places you at the correct starting phase.
Signals: existing repo, mid-flight work, "bring this into the framework"

The start skill reads your opening message and routes automatically — one question if it's ambiguous. All three entry points converge into the same lifecycle.

Session Modes

Three modes. You choose how much the framework carries.

bare default
Nothing loads until you ask for it. You invoke the skill you need, it runs, the framework stays out of the way.
Right for: focused work where you know exactly what you want.
piloted
You're flying it. The four always-on skills load at session start — memory, continuity, process accountability, health. You set the course: you choose which phase to run next.
Right for: builders who want instruments live while staying in control.
guided
The framework leads, you decide. Always-on skills active plus the full phase chain — phases flow forward automatically. You're in every conversation and every sign-off, but not managing the navigation.
Right for: new projects or when you want to stay in the work, not the framework.
Enhanced mode Claude Code only — a separate capability layer, not a session mode. Unlocks multi-agent orchestration for supported skills. Can combine with any of the three modes above.
Always-On

Four skills that activate on piloted or guided mode — silent in bare

Bare mode is the default — framework is silent until you invoke a skill. Say "guided mode" or "piloted mode" to activate these four. One load per session. Token cost: guided (highest) · piloted (moderate) · bare (lowest).

Nothing gets captured from a Nivya conversation without explicit consent. For focused sit-downs, Nivya offers to huddle as a dedicated subagent.
Phase 2

The design artifact is the visual specification

Before a line of code is written, the Design Sprint produces an HTML artifact — all screens, interactive, good enough to make real decisions from.

This artifact isn't a finished product — it's a thinking tool that surfaces four categories of work that couldn't be seen before.

  • 📊
    Data questions — every field on screen gets "where does this come from?" answered before build
  • 🔄
    Process gaps — missing steps, decision points, and handoffs surface when you see the flow
  • ✂️
    Scope decisions — every element on screen becomes an explicit build/defer decision
  • ⚙️
    Implied infrastructure — a simple toggle may hide a data pipeline. The design surfaces it before code commits to it.

Three on-ramps: start from scratch (warmer/colder loop) · import from Figma · reference-based aesthetic

Phase 3 → Build

Not a user story. Not a feature. A contract.

A user story describes what a user wants. A feature describes what the product does. Both are written for stakeholders — they live outside the build. A deliverable is the agreed body of work between the solo and the framework. Defined before build starts. Accepted when it's complete. The framework reads and understands every field — it holds the full picture and surfaces the right thing at the right moment.

Each deliverable is made up of slices — individual units of work built, reviewed, and marked Done one at a time. When all the slices in a deliverable are complete, the deliverable moves to acceptance.

Screen deliverable
Output is visible. Reviewed in the browser against the design and acceptance criteria.
Logic deliverable
Output is invisible. Reviewed against evidence — test results, data state, simulated flows through affected screens.

Phase 4 — Build

No slice starts without four anchors

Design Anchor
The exact screen and element from the design sprint. No ambiguity about what's being built.
sprint-v1.html → Player Search input + results list
Data Anchor
The specific fields this slice reads, and where they live. Confirmed before code touches them. Mock data is generated by data-scaffold before build begins — realistic values, one swap point to replace with real API data when ready.
data/mock/players.json → id, name, position, team
Done Anchor
2–3 verifiable criteria. Not "it works" — specific, checkable statements.
Search returns results · empty state shown · no console errors
Process Anchor
Which step in the agreed to-be process map this slice implements. Nothing gets built that isn't traced to the agreed process.
to-be-lookup.md → Step 1: user searches · main path

If a slice has no process anchor → the framework stops and asks: "Is this building something not in the agreed process?" That question matters.

The Backbone

The process thread runs from first conversation to final test

The to-be process map isn't filed and forgotten. It's the contract every downstream phase is held to — automatically.

1
Discover
As-is and to-be process maps produced and agreed. The to-be map is the contract.
2
Design Sprint
Screens cross-referenced against to-be map. Gaps become explicit decisions.
3
Design Review
Coverage check every round. Every slice gets a process anchor before reaching Ready.
4
Build
Process anchor required on every slice. Stops if something isn't in the agreed process.
5
Phase Test
Use cases derived from the to-be map. Did we build what the process requires?
Phase 5 — Explicit

Phase Test: seven specialist lenses before anything ships

Stage 1
Environment Readiness — pre-flight gate. Nothing runs until mock is inactive, all slices deployed, app loads clean.
Stage 2
Use Case Creator — derives the test plan from the discovery brief and to-be process map. Not invented — derived.
Stage 3
Data Specialist — mock→real swap confirmed. API read and write verified. Data mapping validated.
Stage 4+5
Tester + Regression Specialist — every scenario with named evidence. All Done slices re-walked. Integration points checked.
Stage 6
Acceptance Reviewer — PM lens. Does what was built match what was discovered? Process intent, not just feature behavior.
Stage 7
Gate Decision — OPEN (deploy) or HOLD (specific items to resolve). HOLD = re-test affected scenarios only, no full re-run.

You invoke /phase-test explicitly when you decide the phase is built. Deliberate — not automatic.

Factual Differentiation

What makes this different

  • Not a prompt collection
    28 interconnected skills that chain automatically — plus Nivya, the on-demand companion. Skills invoke each other. The QA chain runs without you triggering it. The always-on skills run throughout.
  • Process contract
    The to-be process map is agreed before design begins and held as the accountability standard through Phase Test. Most tools have no concept of this.
  • Design before code
    The HTML artifact is produced before build starts — not after. Scope, data, and flow issues surface in design, not in code. The artifact becomes the starting point for the frontend.
  • Quality with evidence
    solo-qa cross-references rendered values against mock data files. Not "I reviewed the code" — actual verification with named evidence. Done means Done.
  • Self-improving
    The retrospective skill captures what works and what doesn't. Framework-level improvements queue to the playbook. Every project makes the framework better.
