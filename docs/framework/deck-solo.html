Solo Builder Framework

Team Discipline. Without the Team.

A complete lifecycle framework for building software solo with AI — from first idea through deployed and tested.

The Problem

Solo builders using AI drift toward waterfall. No external forcing function — no standup, no sprint review, no teammate waiting. AI defaults to building what you describe, not challenging whether you're ready to build it. No shared artifact before code starts means scope, data gaps, and flow issues surface in code, not in design. Quality gates disappear.

The agile reframe: replace "phase complete" with "enough to learn from." The framework provides the forcing functions a team would provide.

Three Entry Points

The framework routes for you based on your opening message.

Entry 1

You know what you want to build

The idea is clear. Who it's for, what it does, what value it delivers. Skip brainstorming — go straight to Discover.

Signals: specific solution in mind, clear user, defined outcome

Entry 2

You're working through an idea

The problem is real but the solution isn't clear yet. Brainstorming gives the idea something concrete to react to — including a rough visual sketch mid-session.

Signals: exploratory tone, "what if", uncertain scope or user

Entry 3

You have an existing project

Code already exists — you want framework discipline applied to it. Onboard reads the project, maps content to gate artifacts, surfaces gaps, and places you at the correct starting phase.

Signals: existing repo, mid-flight work, "bring this into the framework"

Three Session Modes

You choose how much the framework carries.

Default

Bare

Nothing loads until you ask for it. You invoke the skill you need, it runs, the framework stays out of the way.

Right for: focused work where you know exactly what you want.

Active

Piloted

You're flying it. The four always-on skills load at session start — memory, continuity, process accountability, health. You set the course: you choose which phase to run next.

Right for: builders who want instruments live while staying in control.

Guided

Guided

The framework leads, you decide. Always-on skills active plus the full phase chain — phases flow forward automatically. You're in every conversation and every sign-off, but not managing the navigation.

Right for: new projects or when you want to stay in the work, not the framework.

Enhanced mode — Claude Code only. A separate capability layer, not a session mode. Unlocks multi-agent orchestration for supported skills. Can combine with any of the three modes above.

Four Always-On Skills

These activate on piloted or guided mode — silent in bare. Bare mode is the default. Say "guided mode" or "piloted mode" to activate them. One load per session.

Process Accountability

process-mapper

Documents how the process works today (as-is) and what it should look like when you're done (to-be). Holds the to-be map as the contract — every phase is accountable to it.

Active across: Discover → Design Sprint → Design Review → Build → Phase Test

Institutional Memory

product-continuity

Captures decisions, assumptions, risks, questions, changes, and glossary across sessions. The first thing read when a session opens — you never lose context between sessions.

Active across: Every phase, every session

Framework Health

framework-health

Background monitor — checks signals, not full documents. Surfaces one issue at a time with a specific recovery path. Silent when everything is healthy. Never blocks.

Active across: Mode activation, between phases, session end

Continuous Learning

retrospective

Captures observations in the moment as lightweight flags. Processes them at phase end — finding patterns, naming root causes, proposing specific improvements to the playbook.

Active across: Every phase — flags in moment, processes at phase end

The Design Artifact

Before a line of code is written, the Design Sprint produces an HTML artifact — all screens, interactive, good enough to make real decisions from. This artifact isn't a finished product — it's a thinking tool that surfaces four categories of work that couldn't be seen before.

Data questions

Every field on screen gets "where does this come from?" answered before build.

Process gaps

Missing steps, decision points, and handoffs surface when you see the flow.

Scope decisions

Every element on screen becomes an explicit build/defer decision.

Implied infrastructure

A simple toggle may hide a data pipeline. The design surfaces it before code commits to it.

Three on-ramps: start from scratch (warmer/colder loop) · import from Figma · reference-based aesthetic

The Deliverable

Not a user story. Not a feature. A contract. A deliverable is the agreed body of work between the solo and the framework — defined before build starts, accepted when it's complete. Each deliverable is made up of slices — individual units of work built, reviewed, and marked Done one at a time.

FieldWho reads itPurpose
Technical spec Framework How to build it correctly — implementation-level, precise
Solo description Framework + Solo What the solo is getting — plain language, outcome-focused. Framework surfaces this at review time.
Acceptance criteria Framework + Solo Shared contract — agreed before build, self-verified by the framework, confirmed by the solo

Two views available at any time — "show progress" for the active deliverable · "show plan" for the full build

Build Discipline — Four Anchors

No slice starts without four anchors.

Design Anchor

The exact screen and element from the design sprint. No ambiguity about what's being built.

Data Anchor

The specific fields this slice reads, and where they live. Confirmed before code touches them. Mock data generated by data-scaffold before build begins — one swap point to replace with real API data when ready.

Done Anchor

2–3 verifiable criteria. Not "it works" — specific, checkable statements.

Process Anchor

Which step in the agreed to-be process map this slice implements. Nothing gets built that isn't traced to the agreed process.

If a slice has no process anchor → the framework stops and asks: "Is this building something not in the agreed process?" That question matters.

The Process Thread

The to-be process map isn't filed and forgotten. It's the contract every downstream phase is held to — automatically.

PhaseHow the process thread runs
Discover As-is and to-be process maps produced and agreed. The to-be map is the contract.
Design Sprint Screens cross-referenced against to-be map. Gaps become explicit decisions.
Design Review Coverage check every round. Every slice gets a process anchor before reaching Ready.
Build Process anchor required on every slice. Stops if something isn't in the agreed process.
Phase Test Use cases derived from the to-be map. Did we build what the process requires?

Phase Test — Seven Specialist Lenses

You invoke /phase-test explicitly when you decide the phase is built. Deliberate — not automatic.

StageWhat it does
Stage 1 — Environment Readiness Pre-flight gate. Nothing runs until mock is inactive, all slices deployed, app loads clean.
Stage 2 — Use Case Creator Derives the test plan from the discovery brief and to-be process map. Not invented — derived.
Stage 3 — Data Specialist Mock→real swap confirmed. API read and write verified. Data mapping validated.
Stage 4+5 — Tester + Regression Specialist Every scenario with named evidence. All Done slices re-walked. Integration points checked.
Stage 6 — Acceptance Reviewer PM lens. Does what was built match what was discovered? Process intent, not just feature behavior.
Stage 7 — Gate Decision OPEN (deploy) or HOLD (specific items to resolve). HOLD = re-test affected scenarios only, no full re-run.

What Makes This Different

WhatWhy it matters
Not a prompt collection 28 interconnected skills that chain automatically — plus Nivya, the on-demand companion. Skills invoke each other. The QA chain runs without you triggering it. The always-on skills run throughout.
Process contract The to-be process map is agreed before design begins and held as the accountability standard through Phase Test. Most tools have no concept of this.
Design before code The HTML artifact is produced before build starts — not after. Scope, data, and flow issues surface in design, not in code. The artifact becomes the starting point for the frontend.
Quality with evidence solo-qa cross-references rendered values against mock data files. Not "I reviewed the code" — actual verification with named evidence. Done means Done.
Self-improving The retrospective skill captures what works and what doesn't. Framework-level improvements queue to the playbook. Every project makes the framework better.

How to Start

Open a new project. Let the framework route you — Brainstorm or Discover. Stay with it through Design Sprint, Build, and Phase Test. Everything else follows from there.
