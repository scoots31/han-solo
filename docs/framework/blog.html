Release Notes

What changed in the framework, what was added, and why — in plain language.

May 7

Session Re-Entry Is Now Lighter — Full Context Loads Only When You Need It

Improved
Faster re-entry — handoff reads only the two active sections on first load

When you resume a named project, the framework now reads only two sections from the handoff on initial load: "Open right now" and "Next session picks up at." That's enough to orient and get started. The rest of the handoff — earlier decisions, stack context, phase history — loads on demand, the moment the session actually needs it. If you ask about an earlier architectural decision or a stack question, that's when the full handoff loads, not before. If those two sections are absent or empty, the framework falls back to a full read as before. The result is less upfront token cost and faster orientation on every re-entry.

May 7

Live Preview Verification — Browser Inspection Baked Into Design Review and Build

New
Design review — Step 1.5: preview pass before specialist review runs

Before the three specialist lenses run their review, the framework now opens each design sprint HTML file in a real browser, takes desktop and mobile screenshots, and inspects key elements against the design identity tokens you set at sprint time. A raw hex value where a token name is expected is flagged as a design gap. Console errors and responsive failures are surfaced too. All of this feeds directly into the specialist review as design gap findings — so the review starts with a fuller picture. In Claude Code this fires automatically with no activation needed. Non-visual projects and rounds with no HTML design artifacts skip silently.

New
Solo-build — preview check inside self-verification, before commit

As part of the existing self-verification step — before the builder declares code-complete — the framework now starts the app server, navigates to the slice's review URL, takes a screenshot, checks console logs, checks for failed network requests, clicks through the main interactive elements, and tests the mobile viewport. Console errors or failed network requests directly caused by the slice are ✗ items in builder confirmation and must be fixed before commit. This runs within the existing self-verification flow, not as a separate step. Non-visual slices (review URL is None) skip silently. In Claude Code only — Cursor falls back to the existing self-verification behavior without the preview check.

New
Auto-created launch config — one-time setup, reused every session

Both preview steps check for a .claude/launch.json file before starting. If it doesn't exist, the framework creates it automatically: design review writes a static Python HTTP server config, and solo-build reads the run command and port from docs/tech-context.md. Once created, the file is reused on every subsequent session — no manual setup required.

May 5

Every Design Sprint Now Starts With a Design Identity

New
Design Identity step — library-matched direction before the first screen

Before any HTML is produced, the design sprint now opens with a Design Identity step. You describe what you're building in a few words, and the framework searches a local library of 500+ curated design styles — each one characterized by a north star phrase, named color tokens, typography, category, and tags. Three matching references are presented. You pick one, name a product to match, describe an aesthetic, or let the framework choose. The result is a docs/design/design-identity.md file: a north star phrase, a named color token table, a typography table, and 2–3 design principles derived from that identity. Every screen produced in the sprint is checked against it. Every design review round checks for north star drift — if a screen no longer feels like the established identity, it's flagged before slices are defined.

New
Design library — 1,322 curated styles with north star phrases

The library lives locally in the framework as a searchable JSON file. Each entry carries a north star phrase (an evocative sentence capturing the design identity — "Architectural blueprint on white marble"), named color tokens with usage guidance, typography roles, industry tags, and category. The library was seeded from Refero's API. Two maintenance tools ship alongside it: one re-pulls from the API to keep it current, the other visits any URL, extracts its design system via CSS parsing, and generates a north star phrase via Claude — so any product or reference site you admire can be added to the library in seconds.

May 5

Solo Companion Now Has a Board View

New
Board view — all active work in one kanban

The companion now has a Board tab alongside the existing project views. It shows all your active deliverables and slices arranged in columns by status — Ready, In Build, In Review, In Test. Done work doesn't appear. The default view shows everything across all active projects; a filter narrows it to a single project. A toggle switches between deliverable cards (one card per deliverable, column determined by its most-advanced slice) and slice cards (every active slice individually). Clicking any card opens the same detail overlay already used in the project view — no new UI to learn. Board ships in both the local companion and the cloud viewer.

May 5

Cloud Deployments Can Now Authenticate to Claude Without a Static API Key

New
Workload Identity Federation — cloud auth without a stored secret

When your project deploys to a cloud platform (Railway, Render, AWS, GCP, Azure, Fly.io, Cloudflare), there's now a better way to authenticate to the Claude API. Instead of storing a static ANTHROPIC_API_KEY in an env var or secret manager, the workload uses its existing cloud identity — an IAM role, service account, managed identity, or GitHub Actions OIDC token — to get a short-lived access token minted on demand. No static secret to store, rotate, or leak. Tech-context now records this as the preferred Claude API auth pattern for all cloud deployments. Local Mac projects are unaffected and continue using an API key in an env var. All other secrets (database passwords, third-party API keys) continue using env vars locally and platform secret stores in production. Workload Identity is configured at platform.claude.com → Settings → Workload identity, and the deploy step handles setup at deploy time.

May 5

Every Project Now Gets a Security Classification — and It Follows Every Slice

New
Security classification at tech-context — six classes beyond the baseline

Every project now receives a security classification during the tech-context phase — before design, before any code. The Standard baseline applies to all projects by default. Six additional classes layer on top: Personal (PII), Financial (transaction records), Authentication (managing others' credentials), Multi-tenant (shared infrastructure with isolated data), Confidential (proprietary business data), and Regulated (HIPAA, PCI-DSS, GDPR, COPPA). Multiple classes can apply to a single project. Regulated projects receive an acknowledgment gate at tech-context and a hard stop at deploy — production requires external compliance review documented in the handoff before any deployment proceeds.

New
Classification follows every slice — security checks expand from 5 to 20

The classification isn't just a label on the project — it travels through the entire build process. At design sprint, the process-mapper checks for security process gaps: missing deletion paths, tenant-scoping issues, and data collection without disclosure. At build, a pre-flight check confirms the quality contract addresses class-specific requirements before any branch opens — if it doesn't, the slice goes back to design review. At code review, Check 9 now reads docs/tech-context.md and expands from 5 baseline checks (9a–9e) to up to 20 total (9a–9t) depending on the project's classification. The baseline always runs. Class-specific checks layer on top only for projects that carry those classes.

May 4

Every Slice Now Has an Architecture Type — and Core Ones Pause for Review

New
Architecture type field — Leaf node or Core architecture, set at planning time

Every slice now carries an Architecture type: either Leaf node (self-contained — nothing else depends on it, tech debt doesn't spread) or Core architecture (touches the trunk — data models, shared logic, infrastructure, or anything other slices depend on). The distinction was always real; it's now explicit and enforced. Classification happens at planning time, when the cost of getting it wrong is lowest. An unclassified slice cannot reach Ready — it's a gate, not a suggestion.

New
Automatic PE review before any core architecture slice enters build

When a slice marked Core architecture moves to In Build, a principal engineer review fires automatically before the branch opens — no manual trigger, no option to skip. The reviewer reads the slice, assesses whether the approach is right, and the outcome lands in the slice's Notes field before any code is written. This isn't friction for its own sake: core architecture changes ripple. Finding a structural problem at review is cheap. Finding it three slices later is not.

Improved
Solo Companion — core architecture visibility across your project

The companion now surfaces architecture type in its project views. Slice rows carry a CORE badge with orange highlight when the type is Core architecture. A callout appears at the top of the project view whenever one or more core architecture slices are currently In Build. A filter lets you pull a core architecture only view across the full project — useful when you want to see everything touching the trunk at once.

May 3

Four Skills Now Run Smarter — In Parallel Where It Matters

Improved
Design review — multi-agent orchestrator fires automatically in Claude Code

Previously you had to manually activate the multi-agent orchestrator. Now in Claude Code it fires automatically — the three specialist lenses (PM, Tech Lead, Data) run as separate agents and surface findings as a merged result. If you see four specialist findings in a design review pass instead of one sequential list, that's expected. Cursor users see no change — sequential review is the fallback.

Improved
Phase test — tester and regression checks now run in parallel

Stage 4 (tester) and Stage 5 (regression specialist) used to run back to back. In Claude Code they now run simultaneously — the tester walks every scenario while the regression check re-walks all In Test slices at the same time. The gate logic is unchanged; it just waits for both to finish before deciding. Cursor gets the same sequential behavior as before.

Improved
Design sprint — secondary screens fan out in parallel after hero approval

Once the hero screen is approved on structure and skin, all secondary screens now produce simultaneously in Claude Code rather than one at a time. The rule that gates them behind hero approval is unchanged — nothing fans out until both dimensions are warm. Result: the full screen set arrives faster, with a consistent design system applied across all of them. Cursor produces screens sequentially as before.

Improved
Code review — now runs as an isolated sub-agent with no build context

Code review used to run in the same context as the build pass — meaning the reviewer had access to all the reasoning and decisions made during implementation. Now it runs as a separate agent that sees only the code, the slice spec, and the quality contract. No build history, no implementation notes. This makes it meaningfully more likely to catch things the build pass rationalized away. If code review surfaces something the build pass didn't flag — that's the point.

Doc Updates Are Automatic Now. And Framework State Persists Across Machines.

New
Comms cascade sub-agent — automated doc updates on every version ship

When a version ships, the framework curator spawns a dedicated sub-agent whose only job is updating the documentation. It reads every comms doc, writes what's relevant, commits, pushes to GitHub, and deploys to Cloudflare — in one pass. The curator provides a structured brief (version, summary, skills affected, watchfor items, guide sections). No more manual doc pass after a build.

New
Cross-device memory — framework state persists across machines

The framework curator now has a private GitHub repo storing current framework state, pending work, and a curator-context summary layer. Every session starts with a fetch from that repo so context is accurate regardless of which machine is being used. A daily 8am health check sends a macOS notification with GREEN/YELLOW/RED staleness scores for key framework indicators.

May 2

Solo Companion Can Now Search Your Memory — and Write to It

New
Cross-source search — project records and MemPalace in one query

A single search box queries both your project records (slices, decisions, changes, open questions) and MemPalace simultaneously. Two-pass engine: natural language and keyword-only, run concurrently, merged by best score per source. Memory results show markdown-rendered content with source filename. Filter chips let you narrow to project records or memory only.

New
Capture — write directly to memory from the companion

A new Capture page lets you write into memory without opening an AI session. Type a title, tag a project, write freely — the note saves as a dated markdown file and mines into MemPalace in the background. Searchable within seconds. The use case: you're mid-build, something clicks — now you can save it immediately and the brainstorm skill can find it later.

May 2

The Framework Now Knows If It's Working

New
Automatic metrics collection — every build produces docs/metrics.json

No configuration required. The file initializes at first slice and accumulates as the build runs: rework cycles per slice, phase test outcome, refinement cycles if autopilot was used. Rework rate tells you if your gates are catching things before they ship. Refinement cycles tell you if discovery was thorough enough. Phase test outcome tells you if the product actually landed. Solo Companion reads it — cross-project aggregation coming.

New
Rework cycle detection — framework flags when a slice re-enters build after QA

When a slice re-enters build after failing QA, the rework counter increments automatically. Increment-only — a slice reworked twice was reworked twice, even if it passes clean on the third attempt. Rising rework rate across a build is a signal something upstream is broken.

May 2

Automated Testing, CI/CD, and Seven Deployment Paths

New
Automated test generation

When a slice reaches code-complete and the project has GitHub Actions configured, the framework generates a pytest test file from the slice's done criteria. The full suite runs before every commit — a red suite blocks the commit until the regression is fixed. Tests accumulate across slices and become a regression floor that protects earlier work throughout the build.

New
CI/CD Setup — one-time, plain-language walkthrough

At first deploy, if GitHub Actions is declared in tech-context but not yet configured, the deploy skill generates the complete GitHub Actions workflow file and walks the solo through the only manual step: pasting a token or key into GitHub. The solo never writes or reviews YAML. Every push to the base branch runs tests automatically. Every merge to main deploys.

New
Seven named deployment paths

Tech-context now maps to one of seven fully-supported paths: Local Mac, Cloudflare Pages/Workers, Railway, Render, Fly.io, AWS App Runner, or No pipeline. The solo is never asked to choose from a list — the framework captures intent through plain-language questions and maps internally. Observability recommendation auto-maps to the selected path (CloudWatch for AWS App Runner; Datadog for Railway/Render/Fly.io; none for local).

Improved
Autopilot now includes test generation

The autonomous build loop generates and runs the test suite at each slice, the same as solo-build. Regressions are caught and logged before the simulator gate check. No behavior change for projects without CI/CD configured.

May 1

The Retro Now Listens. And Talks to the Framework.

New
Session-present listening — retrospective captures in the moment, not at phase end

The retrospective used to wait. It fired at phase end, read the notes, and wrote a report. By then, the moment was gone and the details were thin. Now it listens throughout every guided and piloted session. Three trigger types: framework-detected (stuck protocol fired, code review failed twice, builder self-corrected), solo-expressed (any signal that the output wasn't what was expected — "why did that happen", "that took way longer than it should have"), and explicit-flag (solo says "note this"). Mid-session acknowledgment is one line and the work continues. Nothing is lost to end-of-session recollection.

New
Cross-machine retro log — observations from all sessions reach the framework owner

Session observations now flow to a shared log in the engineering-playbook repo. At session end, the Stop hook pushes retro notes to shared/retro-log.md alongside session signals. The framework curator reads this log at session start in owner mode — the only consumer. Framework skills never read it. Same-day skip logic prevents surfacing the same observations twice in one day. This is how real usage patterns across all machines and all projects become visible to the framework — not through manual reporting, but through the session itself.

May 1

Session Boundaries That Mean Something

New
Mid-deliverable checkpoint — automatic handoff update at slice 4

When a deliverable has more than four slices and the fourth is complete, the framework updates the handoff automatically and continues building. No pause, no solo approval. This is a context hygiene measure: if the session is interrupted or context compresses, the handoff has a current record of where things stand. It fires once per deliverable. The solo doesn't have to do anything — it happens in the background.

Improved
Deliverable completion — framework recommends closing the session

When a deliverable is fully accepted, the framework names the boundary: the work is coherent, the handoff has something meaningful to record, and the next deliverable starts fresh without inheriting this session's context. The solo can close out or continue — it's an explicit choice. The point isn't to force stops, it's to surface the right moment when one exists. A deliverable boundary is a real seam in the work. The framework now says so.

May 1

Structure Before Skin — Design Sprint Round One Gets a Rule

New
Wireframe-first round one — no color, no typography until structure is confirmed

Round one of the design sprint now produces structure only: white background, dark text, gray borders, system fonts. No color decisions. No typography decisions. No skin of any kind. The constraint is absolute. The solo confirms the layout is right before any aesthetic direction begins. This prevents the common failure mode where a solo is reacting to the visual treatment before they've agreed on the underlying structure — and misses a layout problem that would have been obvious in a wireframe.

New
Element naming pass — builder names every interactive element before feedback begins

After producing the round one HTML, the builder walks every interactive element in the artifact in order: names the component type (dropdown, filter, search, toggle) and the expected behavior. The solo corrects anything misidentified before layout feedback starts. This is a one-time identification pass that prevents the builder from silently categorizing a filter as decorative or a search box as non-functional. The solo knows exactly what they're looking at before they say anything about it.

May 1

Autopilot — Build Without a Solo in the Loop

New
Autopilot — autonomous build mode

A new alternative for the Build phase. The solo completes discovery (and optionally the design sprint), then hands off. The framework builds the full product autonomously — the solo-simulator plays every gate, every slice is anchored and committed, and the solo sees nothing until phase test. Phase test is the first human touch point. The solo reviews the finished product, not individual slices.

New
Pre-flight check — mandatory gate before autonomous build begins

Before a single slice is built, the simulator reads all discovery docs and surfaces judgment gaps — places where the spec doesn't decide and the simulator would have to guess. The solo answers each gap. The brief is locked only when no unresolved gaps remain. A discovery quality check also runs: if the to-be map is vague or missing edge cases, this is flagged before gap questions. The cost of removing solo gates from the build is paid upfront here, not spread across the product as post-hoc fixes.

New
Refinement cycles — delta iteration after phase test

After phase test, if the solo has changes, Refinement builds the delta — not the full product again. Each issue is classified first: implementation issues (behavior off, display wrong, data broken) go into Refinement. Structural issues (wrong screen, wrong flow at the process level) require a guided session. The exit ramp to guided mode is surfaced at every Refinement cycle completion. Refinement is autopilot-only — it doesn't exist in guided mode.

New
Two entry points — post-discover or post-design-review

Autopilot can start after discovery alone (faster to begin, more simulator judgment calls, higher Refinement likelihood) or after a full design sprint and design review (richer anchor, fewer judgment calls, tighter first-pass output). Entry Point 2 is recommended for products with significant UI or interaction complexity. The solo chooses based on how much design clarity they want to front-load.

May 1

Figma to the Letter, Rollback by Protocol

New
Figma fidelity — interactive element classification required before build

Every interactive element in a Figma frame scope must now be classified before the slice reaches Ready: Functional (logic wired in this slice), Deferred (logic in a future slice — a companion slice is created in the backlog now), or Out of scope (appears in Figma but is not part of this product). There is no fourth option. The classification happens at design review, written into the slice's Notes field, before the builder sees the slice. A builder who encounters a filter, search box, or pagination control at build time cannot make that product decision silently — the decision was already made.

New
Figma fidelity — node properties extracted before coding, not approximated

When the design source is a Figma file, the builder extracts exact node properties — spacing, color, typography — from Figma Dev Mode or MCP before writing any code. Visual estimates are not acceptable. This matters because the gap between "looks close" and "matches the spec" is where design intent erodes. A layout that approximates the spacing is not the same as one that implements it. The Figma file is the specification; reading it visually is not the same as reading it.

New
Code review Check 10 — design fidelity (Figma-sourced slices only)

A new Check 10 runs on every Figma-sourced slice after code-complete. Four sub-checks: the interactive element inventory must be present in the Notes field (missing inventory fails immediately — the slice returns to design review, not to build); every element listed in the inventory must exist in the implementation; Functional elements must have wired logic; and Deferred elements must be explicit non-interactive placeholders. Slices with non-Figma design sources skip Check 10 entirely.

New
Rollback protocol — formal mechanism for undoing completed work

When a slice that has already passed QA needs to be rebuilt, the framework now has a protocol for it. The builder first assesses whether the scope is a targeted fix (the approach is sound, a specific behavior is wrong) or a full rebuild (the approach itself is structurally wrong). The builder states which scope applies and why. The solo confirms before anything changes. On confirmation: a rollback log entry is written, the slice returns to In Build, and the deliverable reverts from Accepted to Pending Acceptance. Nothing happens before confirmation — no status changes, no code discarded.

Improved
Plain language — skill names removed from solo-facing output

The framework's Output Contract says skill names never appear in what the solo reads. A handful of quoted output examples and gate decision templates were violating this — all twelve occurrences corrected. The framework's internal routing stays internal — the solo sees actions and outcomes, not the machinery producing them.

May 1

Built to Ship, Not Just to Pass

New
Quality contract — four required categories on every slice

Every slice now requires a quality contract written during design review, before build starts. The contract is structured into four required categories: Failure states (what happens when an external call fails or times out), Edge cases (empty data, max values, concurrent actions), Input validation (what is rejected, when, with what feedback), and Security (user input sanitized, data scoped, auth server-side). Each category must be addressed or explicitly marked N/A with a reason — a blank category is not valid. A slice without a quality contract cannot reach Ready.

New
Code review Check 8 — adversarial quality contract verification

Code review now reads the quality contract before reading the code, then checks every contract line against the implementation with deliberate adversarial specificity. A try/catch that swallows the error silently does not satisfy "user sees an error message." A validation check on the wrong event does not satisfy "rejects empty string at submission." The contract is the spec. The code either satisfies it or it doesn't — there is no partial credit and no nearest-thing rationalization.

New
Code review Check 9 — independent security baseline

A fixed five-item security gate now runs on every slice, independent of what the quality contract says. The five sub-checks: input sanitization (no raw rendering of user-supplied content), data scoping (data returned scoped to the authenticated user), server-side auth (authorization logic lives on the server), injection prevention (user input not string-concatenated into queries or commands), and no secrets in client-visible code. Check 9 is independent by design — it catches classes of vulnerability a per-slice contract writer might not surface.

Improved
Self-verification now walks observable contract items

Before declaring code-complete, the builder is now required to walk every observable quality contract item in the running app — not just the happy path. This means: simulating a failure and confirming the error state renders, submitting empty or invalid input and confirming validation fires at the right point, and for any security contract line, confirming user-supplied content is not rendered raw.

April 30

Build Discipline, Session Hygiene, and Observability

Improved
Pre-build read gate — verbatim quote required

The pre-build read gate in solo-build has been tightened: the builder must now produce a verbatim quote from the design file — the exact string, word for word — before writing any code. A paraphrase no longer satisfies the gate. This closes the loophole where a builder could confirm the read from memory or a prior session rather than actually opening the file.

New
Session signal system — passive telemetry across all machines

Four skills now append one-line signals to a temporary file when notable failure events occur during a build session: stuck protocol fired, slice rebuilt, priority deviation (solo-build); QA caught a missed self-verification (solo-qa); code review failed (code-review-and-quality); phase test on hold (phase-test). A Stop hook wired via onboard reads the file at session end, appends structured lines to a shared session log in the engineering-playbook repo, pushes, and clears the file. If push fails, signals are held in a pending file and included in the next successful push.

New
Handoff staleness hook — automatic warning when backlog outpaces handoff

Onboard now wires a Stop hook into every project: at the end of each session, if the backlog has been modified more recently than the handoff document, a warning surfaces. The handoff is what the framework reads at session start to orient — if the backlog changed and the handoff didn't, the next session opens on stale context. The hook is silent when handoff is current.

New
Observability tool decision locked at tech-context

Tech context now explicitly asks whether runtime observability is required and which tool is in place — Datadog, CloudWatch, or none. The choice is mutually exclusive and locked at tech-context time. If a tool is declared, an observability setup infrastructure slice is required in the backlog before any feature slices start. This prevents the common pattern of shipping features without instrumentation and wiring it in after the fact.

April 30

Continuity, Collaboration, and the Cycle After Launch

New
Onboard — companion compatibility pass

Onboarding an existing project now ends with a four-check compatibility pass that verifies the project will sync cleanly with the Solo Companion before exiting. The checks: project registered in the framework registry, backlog has the three required section headers, handoff contains an Open right now section, and every slice record has a Review URL field. A project that silently fails any of these shows up in the companion with empty dashboards and missing cards. The pass fills or flags each gap — so the first companion sync after onboarding just works.

New
Shared ideas backlog — framework collaboration with owner sign-off

The framework curator now supports a shared ideas backlog at shared/ideas.md. Any contributor can brainstorm with the curator and log proposed ideas — the curator handles the write, commit, and push. Ideas sit as Proposed until the framework owner reviews them. At the start of every owner session, the curator surfaces pending items for sign-off. Approved ideas can then move through the normal curator workflow to become real framework changes. Nothing proceeds to build without owner approval.

New
Deployed project — new cycle routing in start

When a project is opened and all its phases are at Deployed status, the framework no longer tries to continue the old work. It detects the deployed state, orients on what was shipped, asks what new work is being added, and opens a new phase in the existing backlog. Completed phases stay exactly as they are — read-only history. The new phase inherits the full context of the project without requiring any of it to be recreated.

April 27

Canonical Records — The Backlog Gets a Specification

New
Records specification — canonical format for all three record types

The backlog now has a single authoritative specification for what every record must contain. Every skill that reads or writes backlog records reads this document first. It defines the full field set for Slice (14 fields), Deliverable (13 fields), and Phase (14 fields) — including status structures, field rules, the skill ownership table, re-phasing protocol, and the decisions and change log format. No skill invents fields. No skill omits required fields. The spec is the contract.

New
Two descriptions required on every record — plain language and technical

Every slice, deliverable, and phase now carries two independent descriptions. The plain language description is written for the solo — what they will see, have, or be able to do when the work is complete. The technical description is written for the builder — implementation approach, constraints, edge cases. Neither copies from the other. Neither summarizes the level below it.

New
Three-level verification — self-verification checklist, acceptance criteria, builder confirmation

Every record type now carries the same three verification fields. The self-verification checklist is what the builder runs before presenting to the solo. The acceptance criteria is the shared contract — the same list the builder checks and the solo reviews, independently. The builder confirmation is populated at presentation time with ✓/✗ results and observations, so the solo arrives at review with the builder's work already shown.

New
In Test — new slice status between solo QA sign-off and Done

Slices now move to In Test after solo-qa sign-off, not directly to Done. Done is reserved for phase-test completion — when the full phase has been verified end-to-end with real data. This resolves a genuine ambiguity: a slice that passed solo QA against mock data and a slice that has been confirmed working in the complete, integrated phase are meaningfully different things. The full chain: In Build → In QA → In Test → Done.

New
Integration — third deliverable type

Deliverables now have three types: Screen, Logic, and Integration. Integration connects real data to a Screen deliverable — it may contain both UI slices and data integration slices, always sequences after its Screen companion is Accepted, and can spawn new slices when real data reveals gaps. This makes the mock-to-real transition explicit in the plan rather than buried in individual slices.

New
Re-phasing protocol — formal process for moving work between phases

When a slice or deliverable needs to move to a different phase mid-build, a 7-step protocol now executes in full: solo confirms the move → decision logged → moved item updated → source phase "Explicitly out of scope" updated → destination phase deliverables list updated → source phase re-assessed → destination phase re-assessed. All seven steps happen in the same action. The decision log entry and the record updates are never separated.

New
Backlog Status Reference — standalone reference page

New page in the comms docs showing all three status grids side by side — Slice (8 statuses), Deliverable (4 statuses), Phase (4 statuses) — with flow chains, plain-language meanings, who sets each status, and the skill ownership table. Linked from the index under Framework Overview. When the backlog shows an unfamiliar status, this is the one-click answer.

Improved
Build, QA, and phase test guides aligned with the new record format

The build guide, QA guide, phase test guide, and skills reference all updated to reflect the new status chain and record format. Builder self-verification step added to the code-complete protocol. In Test introduced throughout the QA flow. Done definition table updated to show phase-test as the final step. Six skill cards in the skills reference updated for the new status names and deliverable fields.

April 25

Delivery Rules — Commands and Review

New
AI runs terminal commands — never the solo

The framework never pastes a command for the solo to copy and run. If a command needs to execute, the AI runs it. If the action is significant or destructive, the AI asks permission first — then runs it. Applied as a load-bearing rule in the Output Contract, framework principles, solo-build skill, and both installed Cursor rules.

New
Review is always visual

When a build produces UI or HTML output and review is needed, the AI serves it as a viewable page before asking for feedback. The solo looks at the work, then responds. Asking for sign-off on a description of what was built is not review.

New
Release notes blog launched

This page. Framework improvements are now tracked in plain language — what changed, what was added, and why — rather than living only in git commit messages.

New
Deliverable — new framework concept

The framework now has a named layer between the plan and individual slices. A deliverable is the body of work a set of slices collectively produce. Two types: screen (reviewed visually in the browser) and logic (reviewed against evidence — test results, data state, or a simulated flow through affected screens). Three fields per deliverable: technical spec for the AI, solo description for the human, and acceptance criteria as the shared contract both parties build and review against.

New
Two review levels

Slice approval (did this unit of work complete correctly) and deliverable acceptance (is the full body of work done and does it match the agreed contract) are now distinct moments. The AI self-verifies against acceptance criteria before presenting a review to the solo. QA runs a deliverable-level check once all slices in a deliverable reach Done.

New
"show progress" and "show plan"

Two plain-language phrases available at any time during build. "show progress" returns the active deliverable — all slices, current status, acceptance criteria. "show plan" returns the full build — all deliverables, delivered / in progress / backlog.

Improved
Session modes renamed: bare / piloted / guided

The three session modes were renamed from bare / assisted / auto-pilot. "Auto-pilot" created the wrong mental model — people assumed the AI worked independently while they stepped away. The new names reflect what each mode actually means: bare (you invoke, nothing pre-loads), piloted (you're at the controls, always-on instruments are live), guided (the framework leads the phase sequencing, you're in every decision). Cascaded across 18 files.

New
Data integration planning — data questions log + integration deliverables

The mock→real data swap is no longer treated as a checklist item at phase test. It's now explicitly planned from design review onward. Design review gets a named data behavior pass — required for every screen with external data — that asks scale and behavior questions upfront. Every unanswered question goes into a data questions log that gates dependent UI slices from reaching Ready. In planning, every Screen deliverable with external data gets a companion Logic deliverable for data integration — always a separate deliverable, always sequencing after the Screen is accepted.

New
process-change — to-be map change management

New skill providing a consistent protocol for changes to the agreed to-be process map, regardless of cause or where in the build they surface. Stakeholder input, demo gaps, technical constraints, and solo realizations all receive the same seven-step response: capture trigger → document change → three-tier impact scan → solo decides → execute → re-prioritize → solo confirms before resuming.

Improved
Phase test timing calibration — readiness table + framework-health nudge

The "When to Invoke" judgment call now has structure. A three-state readiness table (Ready / Not Yet / Overdue) gives new solos the calibration guidance the gate criteria alone couldn't provide. The Overdue state is new — it names the scope-creep failure mode where a solo keeps adding slices rather than pulling the gate trigger on a finished phase.

Improved
Long-gap recovery — framework-health detection + product-continuity re-entry protocol

When a session resumes after 21 or more days, the framework now detects the gap and responds to it. framework-health parses the handoff date at mode activation and if the gap is 21+ days, surfaces a re-entry prompt before any work begins. product-continuity handles the re-entry as three sequential checks: codebase state, environment, and context validity.
