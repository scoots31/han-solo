What This Phase Is

Phase Test sits between QA/Acceptance and Deploy. It is the last gate before anything goes live. Every individual slice passed its own QA — Phase Test asks whether all those slices, together, deliver what you set out to build.

The core question: "Does this phase — as a complete, integrated experience — solve what we set out to solve, with real data, end to end?"

Slice QA verified each slice against its done criteria in isolation. Phase Test verifies the slices working together as a product — real data, full user journeys, integration points between screens, and the original intent from the discovery brief. These are different questions. Both are required.

This phase does not auto-trigger. The solo invokes it explicitly when they have made a deliberate decision: this phase is built. Not when the last slice hits Done automatically — when you look at the backlog and say "we're done with this phase." That decision belongs to you.

Before invoking, you should be able to confirm all three: all slices are Done in the backlog (not In QA, not In Progress — Done), the product is deployed to a test environment with real APIs and no mock data active, and a real user could walk the core journey — not "almost works," not "works locally." If any of these aren't true, it's too early.

The 7 Stages

Run in sequence — except Stages 4 and 5, which run simultaneously. Each stage must complete before the next begins. Phase Test is also deliverable-structured: it walks each deliverable in the phase, confirms deliverable acceptance was completed during QA, then verifies all deliverables hold together in integration. A deliverable that passed QA individually may still surface integration issues when running alongside others — Phase Test is where that gets caught.

Stage 1 · Environment Readiness

Pre-flight gate — nothing runs until this passes

Confirms the test environment is actually ready to test against. Testing against a misconfigured environment produces false results — passes that will fail in production, failures that are config problems, not bugs.

Checklist: mock layer inactive / real APIs active · all Done slices deployed · auth configured for test environment · app loads clean with no console errors · no broken states before interaction.

If anything fails: stop. List exactly what's missing. Do not proceed to Stage 2 until the environment is confirmed ready.

Stage 2 · Use Case Creator

Builds the test plan every other specialist works from

Reads the discovery brief and design sprint artifact, then derives a structured test plan. This is not invented — it's extracted from what was already agreed on. Every use case, screen transition, edge case from mock data, and done criteria becomes a numbered scenario.

Output is a numbered test plan. Every specialist from Stage 3 onward works from this shared reference. Tester, Regression Specialist, and Acceptance Reviewer all anchor to it.

Stage 3 · Data Specialist

Confirms real data is flowing before anyone walks a scenario

Verifies the mock-to-real transition actually worked — not just that config points at the right place, but that data is reading and writing correctly through the full stack. Validates every field in data-mapping.md against the live API response.

Checks: read operations return expected shape · field values render in UI (not hardcoded) · write operations persist and reflect · failed writes handled gracefully · empty/null responses handled.

Any data issue that would make test scenarios unreliable routes to qa-triage before the tester runs on affected scenarios. Blocked scenarios are flagged; unaffected scenarios proceed.

Stages 4 + 5 · Tester and Regression Specialist (run simultaneously)

Stage 4: walks every scenario with named evidence · Stage 5: re-walks Done slices and checks integration points

Tester (Stage 4): Executes each scenario from the test plan. Active testing — not reading code, not inspecting config. Walking the product the way a user would, recording pass or fail with specific named evidence every time. Sets up precondition, executes steps, compares against expected outcome. Failures route to qa-triage and the run continues.

Regression Specialist (Stage 5): Specifically focuses on integration points — the places where slices connect. A slice can pass QA in isolation and fail when another slice changes shared state, shared components, or shared navigation. Checks: navigate to each Done slice in test env · verify done criteria still hold with evidence · state from prior screens reaches this slice · shared components remain consistent · navigation intact after later slices added.

Stage 6 · Acceptance Reviewer

PM lens — runs after Stages 4 and 5 complete

Asks a different question than every other stage: not "does it work" but "does it solve the problem?" Goes back to the discovery brief and asks whether the product that was built matches the intent that was defined. The tester asks "does it work?" — the acceptance reviewer asks "is it the right thing?"

The acceptance question for each use case: "A user who needs to [use case] can now [accomplish goal] — and the experience feels like it was designed for them, not assembled from parts." Pass or fail. With specific evidence if it fails.

Stage 7 · Gate Decision

Synthesizes all specialist outputs into a single gate decision

The orchestrator collects environment, data, tester, regression, and acceptance results and produces one of two outcomes: OPEN or HOLD. No partial passes. The gate is binary.

The gate decision output states: environment status · data status · testing results (X/Y scenarios passed, open items in qa-triage) · regression results · acceptance results. Followed by OPEN or HOLD with specific named items if HOLD. On HOLD: once items are resolved and re-tested, only affected scenarios re-run — no full re-run required.

OPEN vs HOLD

Gate Decision

OPEN

All stages passed. The phase is confirmed to work as an integrated experience, with real data, matching the intent from the discovery brief.

  • Proceed to deploy
  • Phase completion record appended to docs/backlog.md
  • Record states: slices completed, test confirmation, what was delivered, what comes next
  • Confirm all slice branches merged before marking complete
Gate Decision

HOLD

Specific items were found that must be resolved before deploy. Not a soft gate — this is a hard stop. Deploy does not run while HOLD is active.

  • Specific items listed — not "some failures" but named issues
  • Fix the affected items only
  • Re-test only affected scenarios — no full re-run
  • Gate opens when all hold items are cleared
HOLD does not mean start over. It means: here is the specific list of what needs to be resolved. Fix those items, re-test the affected scenarios, and the gate opens. The full phase test does not re-run — only the affected areas.

When Failures Occur

A failure in any stage does not stop the phase test run. When a scenario fails, the tester records the failure with full evidence, routes it to qa-triage, and continues to the next scenario. The test run builds a complete picture — not a partial one that stopped at the first problem. A single visible failure often masks others. Stopping early produces an incomplete report.

ClassificationWhat it meansWhat happens
Bug Something that worked before is broken, or a slice doesn't behave as its done criteria described Root cause identified, fix scoped, added to HOLD list
Missing Requirement A use case from the discovery brief is not covered by any slice — a scope gap Needs to be defined and deferred or added before OPEN is possible
Regression A Done slice that passed its own QA is now broken because a later slice changed shared state, shared components, or navigation Root cause is the later change — that's what gets fixed

The Output Document

The full phase test record lives at docs/phase-test-[phase]-[date].md. Deploy reads this document directly and confirms the gate OPEN status before anything runs. If the report doesn't exist or the gate status is HOLD, deploy stops.

The Gate — Phase Test to Deploy

OPEN decision required before deploy. No exceptions. Deploy reads the phase test report and confirms OPEN status before running. There is no deploy-without-phase-test path.

The gate exists because slice QA alone isn't sufficient. Slice QA confirms each piece was built correctly in isolation. It doesn't confirm the slices work together, that real data flows correctly end to end, or that the product delivers on the original intent. Phase Test is the difference between "all the parts passed their tests" and "the thing we set out to build actually works."

HOLD is not a soft gate. The items listed in a HOLD decision are the explicit conditions that must be met before the gate opens. They must be resolved and re-tested. "Tested and ready to deploy" means: all phase slices Done in the backlog · phase test gate is OPEN (confirmed in the report) · no open qa-triage items from the phase test · acceptance reviewer confirmed use case intent.

Handoff to Deploy

When the gate opens, the orchestrator does three things before handing to deploy. First: every slice in the phase moves from In Test to Done in the backlog — this is the only moment Done is set at the slice level. Second: the phase record status updates from In Progress to Completed, and the phase-level builder confirmation is populated with the phase test results. Third: a phase completion record is appended to docs/backlog.md — a clear declaration you can point at, share with a stakeholder, or orient from at the start of the next session.

The phase completion record includes: what was delivered (plain English, no slice IDs), slices completed count, test confirmation with report reference, and what comes next in the following phase. Before the record is appended, version control must be clean: all In Test slice branches pushed and merged, no open feature branches for this phase, the base branch reflecting the full phase.

What to Watch For

Anti-patternWhat goes wrongDo this instead
Invoking phase test before all slices are Done Testing an incomplete build produces unreliable results. Passes that don't mean anything. Failures that are just "not built yet." The test plan scenarios reference work that doesn't exist. Confirm every slice in the phase is Done before invoking — meaning solo-qa has signed off on each one individually. In QA doesn't count. In Build definitely doesn't count.
Treating HOLD as optional or soft You skip the items listed in the HOLD, deploy anyway, and ship known problems to production. The gate exists specifically to prevent this. HOLD is not a judgment call. Resolve every item listed in the HOLD decision. Re-test the affected scenarios. Only then does the gate open — and only then does deploy run.
Full re-run of the test suite after a HOLD fix Wasteful and unnecessary. The test plan was already run. The scenarios that passed already have their evidence. Re-running everything adds time without adding information. After fixing HOLD items, re-test only the affected scenarios. Regression re-checks affected slices. Acceptance re-walks flagged use cases. Environment and data re-run only if the fix touched config or APIs.
Treating acceptance review as a repeat of testing You end up with two testers and no PM lens. The acceptance reviewer answers a different question than the tester — not "does it work" but "did we build the right thing." If the acceptance reviewer just re-runs technical checks, the PM question never gets asked. Acceptance review reads the discovery brief and asks whether the built product delivers what the use cases described. It doesn't re-execute test scenarios. It asks whether a person using this feels like it was designed for them.
Solo sign-off at the phase level without running phase test Slice QA alone isn't sufficient. Each slice confirmed its own done criteria in isolation. Slice QA doesn't confirm integration, real data flow, or use case intent. You can have 10 passing slices and a broken product. Phase test is required before deploy — not optional, not replaceable by accumulated slice QA passes. The gate enforces this. Deploy confirms OPEN before running.
