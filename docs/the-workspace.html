The Workspace

Where work lives

The project layer of Han Solo — a live view of every project's phases, slices, deliverables, and status. Connected to Ren's memory, the framework's skill library, and the T4 data store that ties them together.

What the workspace replaced

Solo Companion started as a local Mac app — a read-only viewer for framework memory that lived on one machine and couldn't be shared. As the system evolved, the useful parts were rebuilt as a live section inside Han Solo: cloud-hosted, accessible to both Scott and Ted, and connected directly to the same database Ren reads and writes.

The workspace isn't a separate product bolted on. It's the same system — the same Postgres database, the same T4 project data, the same skill records — surfaced as a UI instead of an API call.

Two sides of the sidebar

The workspace sidebar has two distinct sections. The top half is Ren-facing: conversation, memory, signals, and notecards. The bottom half is project-facing: the live state of every build in progress.

The framework in the database

The Solo Builder Framework's skills and phase content used to live exclusively as local files on disk — readable by Claude Code but invisible to Ren. They're now stored in Postgres alongside all of Ren's memory data. Ren can read them directly. Claude Code can query them via the /api/skills/{phase_slug} endpoint. Both sides of the system have access to the same source of truth.

Every skill has a phase_slug (the skill's identity, e.g. discover, design-sprint, solo-build), a layer that marks whether it's always-on or phase-active, and a content field with the full skill text. Updates to a skill are reflected immediately — no file sync, no local path dependency.

phase_slug text -- e.g. "discover", "solo-build", "ren" layer text -- "phase-active" | "always-on" content text -- full skill content updated_at timestamptz

The Solo Builder section of the workspace shows the full skill library browsable by category — phase skills, supporting skills, always-on context. When Ren and Scott discuss a project and Ren needs to orient Claude Code on what phase it's in, the relevant skill content can be passed directly from her context without Scott manually pasting files.

Project visibility

Every project in the workspace is either Private or Shared. Private projects are only visible to their owner. Shared projects are visible to all users with workspace access — currently Scott and Ted.

Private

Default for all new projects. Only the project owner sees it in their workspace. Ted won't see Scott's private projects, and vice versa.

Shared

Visible to all workspace users. Scott marks a project Shared from the Projects view when he wants Ted to have full visibility into its phases, slices, and deliverables.

Ren's access isn't filtered by visibility — she reads all project state via MCP tools and her portrait data. The visibility model controls what appears in the workspace UI, not what Ren knows.

T4 as the data layer

Everything the workspace shows comes from T4 — the project data layer in Postgres. T4 entries have a type (slice, deliverable, phase, decision, context, status) and a project scope. Claude Code writes to T4 after every build session. Ren reads from T4 when asked about project state. The workspace reads from T4 to render the Board, Dashboard, and project drill-downs.

There's no manual sync step. A slice that Claude Code marks done in the build session shows up in the workspace Board the next time it loads. A decision Ren writes to T4 during a chat session is visible in the project detail view immediately.

The connection: Ren holds the context. Claude Code does the work. T4 is the shared record. The workspace is how you see all three at once.
