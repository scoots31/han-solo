Session Architecture

What loads, when it loads, and how the Solo Hook works — the complete picture of how a Claude Code session starts and runs with the framework active.

1 — Session Start

Three things load automatically before you type your first message. Nothing else fires until you submit a message or explicitly invoke a skill.

1

Global CLAUDE.md

Always-on context for every project. Contains: partnership rules, coding behavior, framework routing (where skills live, how modes work), output contract pointer, and Ren invocation directive. Updated this session to route to ~/Developer/Framework Vers1 — engineering-playbook references fully retired.

2

Project CLAUDE.md

Project-specific context, if present in the working directory. For the han-solo repo this includes the component map, deployment rules, engineering standards reference, and the no-agents rule. Loads alongside the global file — both are in context.

3

Memory Index (MEMORY.md)

Claude Code's local persistence layer at ~/.claude/projects/{path}/memory/MEMORY.md. Not Letta. A local index of persistent memory files — user profile, project notes, prior decisions, feedback, framework preferences. Loads automatically each session so Claude recalls context across conversations without being told.

2 — The Solo Hook

The UserPromptSubmit hook fires before Claude sees every message — including the first one. It injects framework content into Claude's context so that compaction and worktree switches cannot permanently drop the framework. The hook always exits 0 and never blocks message delivery.

Key distinction: The hook fires on message submit, not on session open. The output contract is re-injected on every single message — not just the first. This is what makes the framework resilient to context compaction.

Hook Execution Order

  • 1
    Always-on injection every message Reads ~/.claude/hooks/framework-always-on.md and injects the full output contract: voice rules, what never appears in output, naming conventions, input rules, response sizing, transition patterns, and session close-out reminder.
  • 2
    Ren-active check if marker exists Checks for ~/.claude/ren-active. If present, evaluates the incoming message:
    → "this is the way" (exact match) — injects CLOSE-OUT TRIGGERED block: tells Claude to call get_skill('ren') and execute the full checklist before responding.
    → any other message — injects REN ACTIVE re-anchor (~80 chars): tells Claude Ren is active and to call get_session_brief() if context was lost to compaction.
  • 3
    Phase skill injection if phase file exists Checks {cwd}/docs/continuity/current-phase.md. If present, extracts the active phase, maps it to a skill slug, then fetches skill content: Han Solo API first (5s timeout), fallback to Framework Vers1/skills/{slug}/SKILL.md. Phase injection runs after Ren-active so close-out fires clean, not buried under skill content.
  • 4
    Debug log write every message Writes one line to ~/.claude/hook-debug.log, overwriting on each run. Format: [timestamp] always-on:yes ren:yes|no|close-out phase:slug|none tier:han-solo|file|none closeout:yes|no. Check this file when session behavior feels off.

Stop Hook

A second hook fires after every Claude response. The Cloudflare deploy check (cloudflare-deploy.sh) runs here — if any comms docs changed in the last git commit, it deploys to Cloudflare Pages automatically.

3 — Ren Invocation

Say "Ren" anywhere in a message — the hook fires first (always-on injects), then Claude reads the Ren skill from ~/Developer/Framework Vers1/skills/ren/SKILL.md and runs the startup sequence.

  • 1
    Read ren-local Last 20 entries from the local session log — prior session summaries, decisions, open threads.
  • 2
    Get session brief mcp__han-solo__get_session_brief — Ren's pending thoughts, active notes from her Letta context.
  • 3
    Read T4 daily-active handoff get_t4_entry("han-solo", "handoff", "daily-active") — bidirectional notes from the prior session close-out: Ren → Claude and Claude → Ren.
  • 4
    Write session marker Runs touch ~/.claude/ren-active. This marker file gates the re-anchor injection and the close-out trigger for the rest of the session.
  • 5
    Orient and engage One sentence of orientation, then present. No preamble, no announcing what loaded.
Skill file sync: ~/.claude/skills/ren/SKILL.md is a symlink to ~/Developer/Framework Vers1/skills/ren/SKILL.md. Both the "Ren" spoken invocation and the /ren slash command read the same file. Han Solo DB holds a separate copy for Ren's self-read — all three stay in sync via write_skill after any change.

4 — Session Close-Out

When you type "this is the way" — and ren-active exists — the hook injects the close-out trigger before Claude sees your message. Claude calls get_skill('ren') to load the full checklist and executes every step before responding.
